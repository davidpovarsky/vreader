## Round 1

5 findings (3 Medium, 2 Low). No Critical/High.

| File:line | Severity | Issue | Resolution |
|---|---|---|---|
| `project.yml:87` (Inject script: missing source plist branch) | Medium | Debug builds silently succeeded if `DebugBridge.plist` was missing — would re-introduce bug #121 without failing the build. | **Fixed**: changed the missing-source branch from `warning: ...; exit 0` to `error: ...; exit 1`. Debug builds now fail fast if the registration input is absent. |
| `project.yml:95` (Inject script: blind delete of CFBundleURLTypes) | Medium | The script deleted all CFBundleURLTypes before merging. Today this is equivalent to add-from-empty, but it would silently strip any future base URL schemes. | **Fixed**: replaced PlistBuddy `Delete + Merge` with a Python `plistlib` surgical merge that (a) preserves any pre-existing CFBundleURLTypes, (b) drops any prior `vreader.debug-bridge` entry by `CFBundleURLName` match (idempotency), (c) appends only the URL types from DebugBridge.plist. |
| `project.yml:98` (PlistBuddy Merge trusts source shape) | Low | `Merge` would inject any future top-level key from DebugBridge.plist into the built Info.plist. | **Fixed**: the new Python merge asserts `set(src.keys()) == {"CFBundleURLTypes"}` and exits with a clear error otherwise. Future-safe — extra keys cause the build to fail. |
| `vreader/SupportingFiles/DebugBridge.plist:4` (stale comment) | Low | Header still claimed per-configuration INFOPLIST_FILE wiring; the actual fix is `postCompileScripts` merge. Misleading. | **Fixed**: rewrote the header to describe the real build flow (excluded from Resources phase; merged into BUILT Info.plist via postCompileScripts; Debug-only; both verify scripts named). Includes editor guidance about the single-top-level-key invariant. |
| `scripts/verify-debug-has-debugbridge.sh:21` (find ordering) | Low | `find | head -1` returns arbitrary order, could pick a stale DerivedData app. | **Fixed**: added `stat -f "%m %N" | sort -rn | head -1` so the script picks the actual most-recently-modified Debug build. |

## Round 2 (verify only)

Same Codex thread, after the 5 fixes.

> Final verdict: no remaining findings.
>
> The revised fix now covers the earlier gaps correctly:
> - project.yml fails fast when DebugBridge.plist is missing in Debug, so bug #121 can't silently regress.
> - The new plistlib merge is the right shape: it preserves existing CFBundleURLTypes, is idempotent for the DebugBridge entry, and narrows injection scope to the intended key only.
> - The top-level-key assertion closes the "checked-in plist can inject arbitrary metadata" concern.
> - The DebugBridge.plist header now matches the actual build wiring and documents the invariant the script enforces.
> - verify-debug-has-debugbridge.sh now selects the newest DerivedData app deterministically.
>
> On the audit dimensions: root cause is fixed, the duplicate-resource leak remains blocked, the approach is aligned with the bug #111 DebugFixtures pattern, and the Release gate still protects against shipping DebugBridge surface.

## Verdict

**ship-as-is.** 5 findings, 1 round, all addressed. Bug #121's root
cause (orphaned `DebugBridge.plist` not merged into Info.plist) is
fixed: Debug builds now register `vreader-debug` URL scheme;
`simctl openurl vreader-debug://...` returns exit 0 instead of
`LSApplicationWorkspaceErrorDomain code=115`; Release builds remain
free of DebugBridge surface.

**Out-of-scope observation** (Codex called this out separately and I
agree it's a separate concern, not a regression of #121): after the
fix, `simctl openurl vreader-debug://snapshot?dest=state.json` reaches
the in-app handler (URL scheme is registered), but the resulting
`state.json` is not written to `Library/Caches/DebugBridge/`. The
handler creates the directory but doesn't complete the snapshot file
write. This is a handler-path bug, not a URL-registration bug; will
file separately if it persists after the verification harness can
exercise it cleanly.
