## 1. Problem

The reader settings panel (`ReaderSettingsPanel`) exposes a segmented **"Reading Mode" picker** with two options, `Native` and `Unified`. The picker is driven by `ReaderSettingsStore.readingMode: ReadingMode` (persisted to UserDefaults key `readerReadingMode`) and, per-book, by `PerBookSettingsOverride.readingMode: String?`.

This leaks an implementation detail. `Unified` is the name of a rendering engine, not a user intention. What users actually want is concrete outcomes — "apply my replacement rules", "convert Simplified ↔ Traditional Chinese". Today those outcomes are *coupled* to flipping a mode whose actual effect is to swap to the `UnifiedTextRenderer`, which (per bugs #156/#157/#158/#162) renders worse than the native readers for most formats: TXT was stripped of `.unifiedReflow` entirely because the unified renderer truncated content and dropped chrome; EPUB unified falls back to native for complex books; AZW3 unified shows only `UnifiedPlaceholderView` ("Unified Mode Coming in V2").

The result is a confusing, partially-broken control. The picker is already conditionally hidden for formats lacking `.unifiedReflow` (bug #158 gate), the Chinese-conversion picker is already disabled in native mode for non-TXT/MD formats (bug #120), and TXT/MD already convert Chinese *natively* without any mode switch (feature #28 / bug #178). The toggle has been hollowed out one bug at a time and now mostly survives as dead surface plus a small set of still-live `.unified` code paths.

Feature #54 finishes the job: hide the picker in normal use, route rendering by an internal `ReaderEngine` enum instead of a user-visible `ReadingMode`, wire replacement rules into the native MD reader so transforms work without a mode switch, migrate-and-remove `readerReadingMode`, and retire `UnifiedPlaceholderView` and the dead Unified paths.

**User-facing outcome:** the reader settings panel no longer shows a "Reading Mode" picker (and no longer shows the Unified-only Tap Zones section); replacement rules and Chinese conversion just work in the native MD reader; native readers route by format; nothing else about reading changes.

## 2. Surface area

### 2.0 Investigation findings that shape the scope

Read before trusting the row's "(1)/(3)(MD)/(4) can land independently" split. Confirmed by reading the code at the cited file:line locations:

1. **The picker is already heavily gated.** `ReaderSettingsPanel.readingModeSection` (lines 379-391) only renders when `Self.shouldShowReadingModeSection(for: formatCapabilities)` returns true (lines 396-403) — i.e. the active format has `.unifiedReflow`. Per `FormatCapabilities.capabilities(for:)`: **TXT and PDF never have `.unifiedReflow`** (the picker is already hidden for them); **MD, simple EPUB, AZW3 do**. So the picker is *currently visible* for MD, EPUB, AZW3 books.

2. **TXT/MD already convert Chinese natively, no mode switch.** `TXTReaderContainerView` applies `SimpTradTransform` via `TextMapper.apply` in its `.task(id: attrStringKey)` before building the attributed string (feature #28 WI-A, lines 277-354 — across the chapter, full-text, and chunked render paths). `MDFileLoader.load(...)` takes a `chineseConversion:` parameter and applies `SimpTradTransform` to the decoded source text before Markdown parse (bug #178 / GH #606 — `MDFileLoader.swift` lines 49-58). The `ReaderUnifiedCoordinator` Unified path also applies it. So Chinese conversion is **not** the gap.

3. **Replacement rules are wired ONLY into the Unified path.** `ReaderContainerView+Sheets.loadReplacementRules()` builds a `ReplacementTransform` from `ContentReplacementRule` SwiftData rows and assigns `unifiedCoordinator.activeTransforms`. It is called from `ReaderContainerView.body`'s `.task` *only when* `settingsStore.readingMode == .unified && resolvedBookFormat.capabilities.contains(.unifiedReflow)` (`ReaderContainerView.swift` lines 366-369). **Native MD does NOT apply replacement rules at all today.** This is the real gap step (3)'s MD half closes.

4. **Native EPUB loads XHTML directly from disk — there is no Swift-side HTML string.** `EPUBWebViewBridge.updateUIView` calls `webView.loadFileURL(contentURL, allowingReadAccessTo: baseDirectory)` where `contentURL` is the on-disk spine document. Unlike TXT/MD (which have a plain-`String` seam), native EPUB can only have replacement rules applied as **JS text-node DOM preprocessing inside the WKWebView** — exactly the row's "(JS text-node preprocessing, CFI-safe)" wording, and exactly the CFI-safety risk. **The #54 row defers the full EPUB engine swap to #42**, and step (2)'s `epubWKWebView`/`foliateWeb` engine cases describe a routing surface that #42 reshapes. Therefore: **EPUB replacement-rule wiring is DEFERRED** (see §4 Phase D).

5. **TXT replacement rules are NOT a cheap one-line seam — they are deferred (Gate-2 round-1 HIGH, confirmed genuine by the round-2 audit).** v1 claimed TXT was "the same `attrStringKey` seam feature #28 used" and added a TXT replacement-rule WI. **This was corrected in v2 and the round-2 audit confirmed the fix is real — it is preserved unchanged.** Feature #28's `SimpTradTransform` is safe in native TXT *only because it is documented 1:1 UTF-16 for BMP CJK characters* — `TXTReaderContainerView.swift:291` explicitly says "SimpTradTransform is 1:1 UTF-16 for BMP CJK chars; offsetMap discarded — reading positions and highlights in source-text coordinates remain valid." A `ReplacementTransform` is **not** 1:1: it can change text length. The native TXT stack persists and restores **global UTF-16 offsets** and **chapter boundaries** in source/global coordinates:
   - `TXTReaderViewModel.currentOffsetUTF16` — "Current scroll position as UTF-16 char offset (global/book-level, for persistence)" (`TXTReaderViewModel.swift:41-42`).
   - `TXTReaderViewModel` chapter handling uses `chapter.globalStartUTF16` to translate local↔global (e.g. `updateScrollPosition`, `makeLocator`, `openChapterBased`).
   - `TXTReaderContainerView.task(id: attrStringKey)` (lines 277-354) builds the *displayed* text by passing it through `TextMapper.apply` and feeds the transformed string to `TXTTextViewBridge` / `TXTChunkedReaderBridge` / `NativeTextPagedView`, while restore uses `initialRestoreOffset` (a source-coordinate offset).

   With a length-changing `ReplacementTransform` applied, the displayed text no longer has the same length as the source — but restore, highlight ranges, the scrubber, and chapter boundaries all still address *source* coordinates. There is **no source↔display offset-map plumbed through the TXT VM / bridge / persistence path**. A saved position or highlight would land at the wrong place **as soon as any length-changing rule is active — not only after a later rule edit**. The unified renderer keeps an `OffsetMap` precisely for this; the native TXT path per feature #28 deliberately discards it (safe only for 1:1 transforms). Closing this correctly requires a real end-to-end source↔display offset translation through `TXTReaderViewModel` (offset apply + chapter-boundary translation), `TXTTextViewBridge`/`TXTChunkedReaderBridge` (restore + highlight ranges), and the position-persistence path. That is itself feature-class work. **Therefore TXT replacement-rule wiring is DEFERRED to Phase D (D-3), to ship together with the offset-map work.** This also realigns the autonomous scope with the feature row, which says "replacement rules work in native **EPUB and MD**" — TXT was never in the row's stated scope.

6. **Conclusion — what is #42-independent and deliverable now:**
   - **Step 1 (hide picker when no transforms configured)** — independent. Today the picker shows for MD/EPUB/AZW3. The row wants it hidden in *normal use*; the cleanest non-#42 framing is: the picker disappears entirely because `ReaderEngine` routing removes the user-facing mode. **Deliverable now.**
   - **Step 2 (internal `ReaderEngine` routing)** — *partially* independent. The TXT/MD/PDF cases (`textNative`/`markdownNative`/`pdfKit`) are pure-Swift dispatch over the existing native hosts. The `epubWKWebView` vs `foliateWeb` distinction is the engine choice #42 owns. **Plan introduces `ReaderEngine` with all five cases now** (a foundational enum is cheap and lets #42 slot in), **but the routing only collapses the `readingMode` branch** — it does not pre-empt #42's `epubWKWebView`↔`foliateWeb` flag decision.
   - **Step 3 (wire replacement rules) — MD: deliverable now** (plain-text seam in `MDFileLoader`; the existing `chineseConversion` block is the exact insertion pattern). **TXT: DEFERRED to Phase D** (see finding 5 — needs the offset-map; not the cheap seam v1 assumed). **EPUB: DEFERRED to Phase D** (needs the JS-DOM-preprocessing path, #42-coupled).
   - **Step 4 (migrate + remove `readerReadingMode`)** — independent. Pure persistence change. **Deliverable now.**
   - **Step 5 (retire `UnifiedPlaceholderView` + dead Unified paths)** — independent *for the placeholder and the toggle UI*. The `UnifiedTextRenderer` / `ReaderUnifiedCoordinator` stack is *also* potentially used by feature #56 (bilingual) per `dev-docs/plans/20260518-feature-56-bilingual-reading-mode.md` — see §2d. **Plan retires the placeholder and the mode-driven dispatch; it does NOT delete `UnifiedTextRenderer`/`ReaderUnifiedCoordinator`** (out of scope, §2d).

### 2c. Backup restore — `readerReadingMode` in OLD backups (no file change, but a design point)

`BackupDataRestorer.restoreSettings(from:)` decodes a `BackupSettingsEnvelope` and writes each captured UserDefaults key back. An **old backup created before #54** will contain a `readerReadingMode` entry. After #54 ships:

- Restoring an old backup will write `readerReadingMode` back into UserDefaults — a now-orphan key. **This is harmless** (nothing reads it) but untidy.
- **Mitigation chosen:** `ReadingModeMigration.run` runs at *every* launch and is idempotent, so the orphan key written by an old-backup restore is removed on the next launch. `restoreSettings` itself needs **no change** — it is generic over the key list in the *backup*, not the current `BackupSettingsKeys.all`. A test (`ReadingModeMigrationTests`) pins that an externally-set `readerReadingMode` is cleared by a re-run. (Optionally, `restoreSettings` could skip unknown keys, but that is a behavior change to a `VERIFIED` subsystem and is **out of scope** — the migration re-run covers it.)
- **Note on timing:** the migration runs synchronously at *launch*; a backup restore happens *later in the same session* (user-triggered). So the orphan key written by an in-session restore persists until the *next* launch — harmless (nothing reads it), and pinned by the test. This is acceptable: the key is inert.

### 2f. `openPositionUnsupportedInUnifiedMode` — corrected producer/consumer audit (round-2 Finding 1 — MEDIUM)

## 3. Prior art / project precedent / rejected alternatives

**Project precedent — the toggle has been incrementally hollowed out.** Bug #156 (auto-page-turn gate), #157 (TXT auto-page-turn regression), #158 (TXT loses `.unifiedReflow`), #162 (tap-zones gate), #120 (Chinese-conversion native-mode disable), #178 (MD native Chinese conversion) each chipped a piece off the Native/Unified toggle. `FormatCapabilities.reflowableBase`'s doc comment explicitly documents `.unifiedReflow` being pulled from TXT. #54 is the planned terminal step that removes the now-vestigial toggle wholesale. The capability-gate pattern (`shouldShowReadingModeSection`, `chineseConversionDisableReason` as testable `static` helpers) is the established shape for these reader-settings gates.

**Project precedent — native text transforms (and its 1:1-only safety condition).** Feature #28 WI-A wired `SimpTradTransform` into native TXT via the `attrStringKey` + `.task(id:)` rebuild seam; bug #178 wired it into native MD via `MDFileLoader.load(chineseConversion:)`. **The precedent's load-bearing precondition is that `SimpTradTransform` is 1:1 UTF-16** — `TXTReaderContainerView.swift:291` and `MDFileLoader.swift`'s doc comment both state this explicitly, and it is *why* the `OffsetMap` is discarded. **Step 3 of #54 for MD is the identical pattern** extended to compose `ReplacementTransform` + `SimpTradTransform` — and MD is safe because MD's reader does not persist source-coordinate scroll offsets the way the TXT chapter stack does (MD restores via `MDFileLoader.restoreOffset` clamped to `renderedTextLengthUTF16` — the rendered/displayed length, which a transform-before-parse pipeline produces consistently). **For TXT the 1:1 precondition does NOT hold for `ReplacementTransform`**, and the TXT VM persists *global source-coordinate* offsets + chapter boundaries — so the precedent does *not* extend to TXT replacement rules without an offset-map. This is the v2 reason TXT replacement rules are deferred (§2.0 finding 5, §6 Risk 3) — confirmed genuine by the round-2 audit and preserved in v3.

**Project precedent — launch-time migration, and why `ReadingModeMigration` CANNOT copy `WebDAVProfileMigrator` (Gate-2 round-1 HIGH — confirmed genuine by round-2, preserved unchanged).** `WebDAVProfileMigrator` (feature #52) runs as `Task.detached(priority: .background)` fire-and-forget at the end of `VReaderApp.init()` (`VReaderApp.swift:298-305`). v1 cited it as the precedent for `ReadingModeMigration`. **v2 corrected this and the round-2 audit confirmed the correction is real.** `WebDAVProfileMigrator.migrateIfNeeded` is `async` and writes through `WebDAVServerProfileStore`, which is an **`actor`** — all its mutations are inside a serialized actor region, so a detached migration cannot race a concurrent profile edit. `ReadingModeMigration` has **no such serialization**: per-book overrides are plain JSON files at `<baseURL>/<sanitizedKey>.json`, read and written by `PerBookSettingsStore.{settings,save,delete}` with no actor and no lock. Verified concurrent, unsynchronized access sites:
- `ReaderSettingsPanel.swift:769` (`loadPerBookState` → `PerBookSettingsStore.settings`), `:782` (`savePerBookSnapshot` → `save`), `:787` (`deletePerBookOverride` → `delete`).
- `ReaderContainerView.swift:289` and `:343/:348` (reader-open reads `settings` / `resolve`).
- `BackupDataRestorer.swift:77` (`restorePerBookSettings` → `save`).
- `BackupDataCollector.swift:177` (`collectPerBookSettings` → `settings`).

A fire-and-forget directory rewrite could (a) clobber a concurrent `savePerBookSnapshot`, or (b) let a reader open and a `ReaderSettingsStore` initialize while the migration still has `readerReadingMode` set — leaving the session reading a stale value the store already snapshotted. **Therefore `ReadingModeMigration.run` is synchronous and runs in `VReaderApp.init()` BEFORE the store / debug-bridge / UI are constructed** — at app launch there are no readers open, no panel mounted, no backup running, so the migration has the per-book directory and `UserDefaults` to itself. It is a one-shot launch gate, not detached best-effort work. The `WebDAVProfileMigrator`'s *idempotent-namespace-enum* shape is still followed; only its *detached scheduling* is rejected.

**Project precedent — `ReaderEngine`-style routing.** `ReaderContainerView.nativeReaderView` already `switch`es on `book.format.lowercased()` to pick a host. `ReaderEngine.resolve(format:)` is a thin, testable rename of that switch's *selector* — turning a stringly-typed format switch into a typed-engine switch. `FormatCapabilities.capabilities(for:)` is the precedent for a pure per-format resolver.

**Rejected alternative A — keep `ReadingMode`, just hide the picker.** Hide the `readingModeSection` and stop there. *Rejected*: leaves `readerReadingMode` in UserDefaults and `PerBookSettingsOverride`, leaves the `readingMode == .unified` dispatch branch live (a user with a stale `.unified` per-book override would still hit `UnifiedTextRenderer`), and leaves dead code. The row explicitly asks for the key to be *removed with migration* and the placeholder *retired* — a half-measure fails the acceptance criteria.

**Rejected alternative B — delete the entire Unified renderer stack in this feature.** *Rejected*: `UnifiedTextRenderer` / `ReaderUnifiedCoordinator` may be consumed by feature #56 (bilingual). Deleting ~6 files of rendering code is a distinct, riskier change with its own verification surface.

**Rejected alternative C — wire native-EPUB replacement rules now via the legacy `EPUBWebViewBridge` JS.** *Rejected for the autonomous scope*: (a) the #54 row explicitly makes the full EPUB engine swap depend on #42; building a *second* legacy-bridge JS transform path that #42 throws away is wasted work; (b) text-node DOM mutation in a paginated WKWebView risks invalidating saved CFIs and highlight anchors. **Deferred to Phase D.**

**Rejected alternative D — make `ReaderEngine` a 3-case enum and add EPUB/AZW3 cases later.** *Rejected*: the row names all five cases; a 3-case enum forces a breaking enum change when #42 lands. A 5-case enum costs nothing now and gives #42 a ready slot.

**Rejected alternative E — wire TXT replacement rules now via the `attrStringKey` seam, accepting position drift.** v1 included TXT replacement rules as WI-6, documenting offset drift as "an accepted limitation" mirroring MD `chineseConversion`. *Rejected in v2 (Gate-2 round-1 HIGH); the round-2 audit confirmed the rejection is genuine — preserved in v3.* The "accepted limitation" framing is wrong for TXT: (a) the MD `chineseConversion` parallel does not hold — `SimpTradTransform` is 1:1, `ReplacementTransform` is not; (b) the TXT VM persists *global source-coordinate* offsets and chapter boundaries, so a length-changing rule misplaces saved positions/highlights **whenever the rule is active**, not only across an edit — this is a correctness defect, not a fidelity nicety; (c) the feature row's stated scope is "native **EPUB and MD**" — TXT was never in it. The correct fix is a real source↔display offset-map through `TXTReaderViewModel` (offset translate + chapter-boundary translate), the TXT bridges (restore + highlight ranges), and the persistence path — itself feature-class work. **TXT replacement rules stay deferred to Phase D (D-3)**, to ship with the offset-map.

**Rejected alternative F — run `ReadingModeMigration` detached like `WebDAVProfileMigrator`.** *Rejected (Gate-2 round-1 HIGH; confirmed genuine by round-2)* — see the "launch-time migration" precedent paragraph above. Detached scheduling is safe only behind actor serialization, which the per-book JSON store does not have. Synchronous-before-setup is the only safe shape without first building an actor wrapper around `PerBookSettingsStore` (which would be a larger, separate change).

**Rejected alternative G (v3) — claim `JSONSerialization` decode/re-encode preserves per-book files byte-for-byte.** v2 repeatedly described the `ReadingModeMigration` per-book rewrite as preserving other fields "byte-for-byte" / "byte-preserved" / "verbatim." *Rejected as factually wrong (round-2 Finding 2).* `JSONSerialization.jsonObject(with:)` parses bytes into a Foundation `Dictionary`; `JSONSerialization.data(withJSONObject:)` re-serializes from that dictionary. The round trip **cannot** reproduce the original byte stream — insignificant whitespace, indentation, key ordering, and number formatting (e.g. `1.0` vs `1`) are all free to change. What the round trip *does* guarantee is **semantic preservation**: every key the migration does not touch, and its value, survives into the rewritten file with identical decoded meaning. v3 replaces every "byte-*" phrasing with "semantic preservation of unknown keys/values" and re-specifies the test (decoded-content equality, not byte equality). See §6 Risk M and the §5 `ReadingModeMigrationTests` row.

## 7. Backward compatibility

- **`readerReadingMode` UserDefaults key.** Removed by `ReadingModeMigration.run` at launch (WI-5), synchronously before the store reads it. Users with `Unified` or `Native` selected: the key is deleted; rendering routes by `ReaderEngine` (always the native reader for the autonomous scope). No error surface, no migration prompt. A user who had `Unified` selected gets the native reader for that format — strictly better-rendered for every format except simple EPUB/AZW3, and the row's explicit design intent for EPUB.
- **Per-book `PerBookSettingsOverride.readingMode`.** The field is removed from the struct (WI-5) and stripped from on-disk per-book JSON files by the migration (WI-2 code, WI-5 launch wiring). The migration edits files as raw `JSONSerialization` JSON objects, so every other per-book key/value (font size, theme, etc.) is **semantically preserved** — the decoded content of each untouched field is identical after the rewrite. (The rewrite is *not* byte-identical: a JSON decode→re-encode round trip may re-order keys or change insignificant whitespace — round-2 Finding 2. This is harmless because the file is only ever consumed via `JSONDecoder`, never compared byte-wise.) An old per-book JSON that still carries a stray `"readingMode"` key (e.g. restored from a backup *after* the migration ran) decodes harmlessly — `Codable`'s synthesised `init(from:)` ignores unknown keys, and the next launch's migration re-run strips it. A per-book file that never had a `"readingMode"` key is left **untouched** by the migration (no needless rewrite).
- **WebDAV / iCloud backups.**
  - *New backups* (post-#54): `BackupSettingsKeys.all` no longer lists `readerReadingMode`; it is not snapshotted. Per-book backup blobs no longer carry `readingMode` (the field is gone from the struct the collector serializes).
  - *Old backups* (pre-#54): contain a `readerReadingMode` settings entry and per-book blobs with `readingMode`. `BackupDataRestorer.restoreSettings` writes the captured keys back generically — `readerReadingMode` lands as an orphan key, cleared by the *next* launch's `ReadingModeMigration.run`. Old per-book blobs decode into the trimmed `PerBookSettingsOverride` (unknown-key tolerance). **No backup format version bump is needed** — the settings/per-book sections are key-list-driven, not schema-pinned. (`BackupDataCollectorRestorerTests` covers this.)
- **SwiftData schema.** Unchanged. `ReadingMode` was never a SwiftData model; `ContentReplacementRule` is unchanged. No `VReaderMigrationPlan` / `SchemaV*` change.
- **Reading positions & highlights.** Stored as `Locator` / `AnnotationAnchor` — engine-agnostic. Switching a book from the (rarely-used) Unified renderer to the native reader does not invalidate them. Replacement rules in native **MD**: rules apply at open/reload time against the rendered text; `MDFileLoader.restoreOffset` clamps to `renderedTextLengthUTF16`, so positions re-derive consistently on a rule change (a re-derivation on reload, not a silent misplacement). Replacement rules in native **TXT** are NOT shipped (deferred to D-3) precisely because the TXT stack lacks the offset-map needed to keep source-coordinate positions valid across a length-changing rule.
- **Tap Zones (stated behavior change, Gate-2 MEDIUM).** The Unified-only Tap Zones configuration section is **removed** from `ReaderSettingsPanel` (WI-4), along with the `TapZoneStore` `@State` and sheet argument in `ReaderContainerView` (§2g). It was already invisible in normal use (shown only in Unified mode, bug #162). With Unified retired there is no native install site for `TapZoneStore`-configured tap-zone actions. **This is an intentional behavior change, not "all reader features unchanged":** users who had configured custom tap zones in Unified mode lose that configuration surface. Native readers continue to handle taps via their own gesture recognizers / JS click handlers (chrome toggle, page turn) — basic tap-to-turn and tap-to-toggle-chrome behavior is unaffected; only the *configurable* zone-action mapping is removed. Preserving configurable tap zones for native readers would be a separate native-tap-zone implementation feature; #54 documents the removal rather than rebuilding it. See §9.
- **Older app versions / older clients.** Forward-only. A device that syncs a new-version backup to an *older* app build: the older build defaults `readingMode` to `.native` when the key is absent (`ReadingMode(rawValue: ...) ?? .native`) — graceful. The trimmed per-book JSON simply lacks a `readingMode` key the old build reads as `nil` (inherit) — graceful. No crash, no data loss in either direction.
