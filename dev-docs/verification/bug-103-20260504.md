## Why no live device run

The bug is a pure concurrency race in `EPUBHighlightRenderer` + `HighlightCoordinator`. The fix is a signature change + a callback-routing change with no UI surface. The unit test uses real `EPUBHighlightRenderer` and real `HighlightCoordinator` (only `HighlightPersisting` is mocked, with deterministic handshake to force the race window). Live device verification would just exercise the same code path through a slower harness. Existing on-device EPUB highlight tests remain green.

## Observations

- **Codex caught the chapter-cross-wiring on the first round.** The original fix only handled the onInjectJS-swap race that bug #77 / #103 documented. Codex correctly identified that `currentHref` is also shared mutable state read across an await, and proposed the `forHref` threading. Without round 1, that subtle race would have shipped as a follow-up bug.

- **The handshake pattern is reusable.** `waitForFetchToBeArmed` + `pendingFetchGate` + `releaseFetchGate` is a clean way to deterministically pause a mock async method while the test triggers a concurrent action. Future race tests across the codebase can reuse this shape.

- **Default-impl extension on the protocol kept backward compat clean.** `func restore(records:)` now lives in an extension that calls `restore(records:forHref: nil, using: nil)`. Existing callers (`handleRemoval`'s re-render) work unchanged.
