## Root cause

`FoliateSpikeView`'s inner `Coordinator.userContentController(_:didReceive:)` registered handlers for many messages but only switched on `bridge-ready`, `book-ready`, and `error`. The JS bundle's `tap` message (sent on center taps) hit the `default: break` branch and silently no-oped.

The full `FoliateViewCoordinator` (production AZW3 path scaffolding) handles `tap` correctly via an `onTap` callback — but `ReaderContainerView` routes AZW3 to `FoliateSpikeView`, not through `FoliateViewBridge` → `FoliateViewCoordinator`. So the production AZW3 flow goes through the simpler Spike that doesn't forward `tap`.

## Fix

Two changes:

1. **Production**: added `case "tap": NotificationCenter.default.post(name: .readerContentTapped, object: nil)` to FoliateSpikeView.Coordinator's message router. ReaderContainerView already observes `.readerContentTapped` and calls `toggleChrome()` (line 125-127), so the chrome-toggle path completes.

2. **Refactor for testability**: extracted `Coordinator` from inside `FoliateSpikeWebView` (private struct) to a top-level `extension FoliateSpikeView`. Added `@MainActor func handleMessage(name:body:) async` so unit tests can exercise specific message branches without constructing a `WKScriptMessage` (whose initializer is internal to WebKit).

## Why no live device run

The bug is a missing case in a switch statement. Test exercises the actual production code path (`Coordinator.handleMessage`) with the actual notification name (`.readerContentTapped`) that `ReaderContainerView` actually observes. Adding a live device run would re-test the same code through a slower harness.

The chrome-toggle path's downstream behavior (`ReaderContainerView` calling `toggleChrome()` on receipt) was already device-verified for EPUB and is reused here unchanged.

## Observations

- **The Spike + production-coordinator split is technical debt.** Feature #42 plans to migrate AZW3 onto `FoliateViewBridge` → `FoliateViewCoordinator` (the production path). When that lands, `FoliateSpikeView` retires entirely. This bug is a forecast of what other "default: break" cases might exist in the Spike for messages that production cares about — Codex round 1 surveyed them and confirmed only `tap` was actively user-facing.

- **`queue: nil` notification delivery is the deterministic primitive for unit-test post/observe pairs.** The 50ms-sleep approach Codex flagged is an anti-pattern; the synchronous in-thread delivery has no flake mode at all.
