## Root cause

In paginated mode foliate-js renders a section as a 2-column page inside an
iframe ~2× the screen width and pages by shifting that iframe horizontally via
its element `left` (device-observed `left: -359` over a 402px host viewport) to
reveal one column at a time. The Bug #239 fix (`bd7564c7`) made the content-tap
handler post `{x: event.clientX, w: documentElement.clientWidth}` so paged
side-taps could route through `ReaderTapZoneRouter`. But `event.clientX` is in
the iframe's internal coordinate space (0..748) while
`documentElement.clientWidth` is a single column's width (374). On every
right-column page, `ReaderTapZoneRouter` saw `x/w ∈ ~1.0..2.0` and classified a
center tap as a right-zone tap → `.readerNextPage` instead of
`.readerContentTapped`. EPUB was unaffected because it renders in the top-level
document (no nested iframe → `clientX`/`clientWidth` are already host-relative).

The original reopen note hypothesized that feature #56 WI-11's bilingual
injection reloaded/replaced the doc and broke the click listener. Device
diagnosis disproved that: the bilingual path does in-place DOM `insertBefore`
and never reloads the doc, and the click listener already re-binds on every
Foliate `load`. The real cause is the coordinate-space mismatch above.

## Fix

`mapTapToHostViewport(doc, clientX)` in `foliate-host.js` maps the click back to
host-viewport coordinates: `hostX = clientX + frameElement.getBoundingClientRect().left`,
`hostW = frameElement.ownerDocument.defaultView.innerWidth`. Defensive paths: no
`frameElement` → `documentElement.clientWidth` (top-level / EPUB-style case);
unusable host width / cross-origin throw → returns `null` so the handler posts
the bare chrome-toggle `tap` (safe default). `foliate-bundle.js` rebuilt from
source via `build-bundle.sh` (esbuild 0.28.0).

## Observations

- The buggy-vs-fixed delta is purely the JS coordinate computation; the Swift
  `ReaderTapZoneRouter` math was already correct (its unit tests pass
  unchanged). The bug was the JS feeding it `clientX` from one coordinate space
  and `w` from another.
- Live verification used the real `webkit.messageHandlers.tap.postMessage`
  intercept, so the captured payload is exactly what the production WKWebView
  bridge receives — not a mirror of the math.
- Computer-use was unavailable (Screen-Sharing virtual-display host), so the
  verification used DebugBridge `eval` + a synthetic `click` MouseEvent
  dispatched at the on-screen center, which exercises the same production
  listener (closure over `mapTapToHostViewport`). A visual chrome-toggle
  screenshot was not reachable without CU; the notification-payload capture is
  a higher-fidelity signal than a screenshot anyway (it proves the exact
  `{x,w}` the router receives).
