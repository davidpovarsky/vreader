## Why simulated repro instead of live rclone

The fix prepares for a fresh rclone target's behavior. The current rclone deployment already has `VReader/uploads/tmp/` from earlier successful uploads (the one that was set up via `curl -X MKCOL` workaround during feature #47 verification). To do a live verification we'd need to either (a) reset the rclone server's data, or (b) point at a brand-new rclone instance. Both add infrastructure setup that doesn't increase confidence over the simulated test, which exercises the exact 409 path-rejection behavior rclone exhibits.

The mock's `simulateRcloneStrictParentRequirement` flag in `MockWebDAVTransport` (added in this PR) returns 409 from PUT when the parent path hasn't been MKCOL'd. That's the precise rule rclone enforces. Tests against this mock prove the fix's PUT lands without 409 and that the MKCOL ordering is correct.

## Observations

- **Race semantics**: dropped the planned NSLock — Swift 6 concurrency forbids `NSLock.lock()` in async contexts. Switched to `nonisolated(unsafe) var ensuredTempDir` with the rationale that the worst-case race is "two uploads each MKCOL the same path" — and MKCOL is idempotent on the transport (`try?` swallows the "already exists" error). The flag is set only after a successful pass, so a failed first MKCOL doesn't poison subsequent attempts.

- **`final class` vs `struct`**: had to convert `WebDAVBlobStore` from `struct` to `final class @unchecked Sendable` to hold the `ensuredTempDir` flag. Could have wrapped the flag in an `actor` or kept it in the transport, but: (a) the transport is shared across consumers and shouldn't accumulate per-blob-store state; (b) the actor hop would cost a microsecond per upload to avoid a benign double-MKCOL. The class with `@unchecked Sendable` + the documented benign-race rationale is the lightest correct option.

- **MKCOL ancestors deepest-last**: the implementation creates `VReader` → `VReader/uploads` → `VReader/uploads/tmp` in order. rclone accepts MKCOL on a deep path even if intermediate ancestors don't exist, but stricter servers (Apache mod_dav, Nginx WebDAV) require ancestor-first order. Walking all of them costs at most 3 round-trips on first call, then memoized.

- **Ancestor helper duplicated**: `nestedAncestors(of:)` is the same logic as `WebDAVProvider.nestedAncestors(of:)`. Kept it inline (private static) because exposing it on a shared utility would mean threading another import. Acceptable duplication — the function is 6 lines and the algorithm is structurally trivial.
