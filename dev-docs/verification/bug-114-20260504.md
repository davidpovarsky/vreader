## Why this fix instead of "make deleteBook unlink the file"

Two orthogonal options were on the table:

- **(a) Make `deleteBook` also remove the sandbox file** — symmetric, cleaner ownership story going forward, but doesn't help users who already have orphans from past deletes. Also widens the surface of `deleteBook` (a hot path) for a backup-restore concern.
- **(b) Make the materializer's already-local branch always import** — robust to orphans regardless of how they got there (delete bug, partial restore, manual filesystem prodding), localized to the restore code path, no widening of `deleteBook`.

We picked (b) because it converges all orphan-state paths to a consistent end-state. (a) can ride along later as a reduce-the-orphans-going-forward fix without conflicting.

## Why mock persistence (no live device run)

The fix lives in a single conditional inside `BookFileMaterializer.materializeOne` and exercises `BookImporter.importFile`'s existing dedupe-then-insert branches — both already device-verified during feature #46's full restore flow. The orphan precondition is created by hand on a live device only by deleting a book then attempting to re-restore — a rare flow that the unit test models exactly via `StubBlobReader` + `MockPersistenceActor.seed` (or absence-of-seed).

## Observations

- **Outcome mapping**: `reimportLocalFile` returns `MaterializeResult(outcome: .alreadyLocal)` even when the importer just inserted a fresh row. That's intentional — the user-facing accounting ("imported / already had it") cares about whether bytes transferred, not whether a row was created. A new outcome case (`.alreadyLocalReinserted` or similar) was rejected as ceremony for a transient state.

- **Defensive fingerprint check**: kept the post-import `fingerprintKey == entry.fingerprintKey` check even though preflight hash above already validated SHA. Cost is negligible (string compare) and it catches one historical class of bugs — extension/format confusion where the file's bytes hash to one thing but BookImporter computes a different fingerprint from the file path's extension.

- **Updated `materialize_existingLocalFileWithMatchingHash_skipsDownload`**: had to seed the persistence row in this test because the new code path reaches `BookImporter`. Without the seed the test would also exercise the orphan branch — useful but conflates the "happy path" assertion. Keeping happy-path and orphan tests separate keeps regressions readable.
