## Observations

- **Original bug was misdiagnosed.** Initial entry said "tap on `.remoteOnly` row doesn't trigger BookDownloadSheet." Diagnostic logging added in this PR's first commit (LibraryView per-guard observer logs) showed the entire tap → notification → enqueue → URLSession → didFinishDownload chain works correctly. The actual missing piece was `LazyDownloadFinalizer.finalize` — the WI-4b integration step that promotes the staged file into the library.

- **WI-4a (`BookFileImportFinalizer`) was correctly built but never wired.** The header comment of `BookFileImportFinalizer.swift:20` literally said `LazyDownloadCoordinator.swift (future, WI-4b consumer)` — a deferred wiring that was forgotten when feature #47 shipped. Nice example of how a comment promising a future wire-up can leak straight into a bug if no test pins the contract.

- **Why a separate `LazyDownloadFinalizer` instead of reusing `BookFileImportFinalizer`?** The materialize-all path goes through `BookImporter.importFile(.restore)`, which on dedupe (an existing `.remoteOnly` row with the same fingerprintKey) updates only `provenance` and returns `isDuplicate=true`. It doesn't move the file or touch `fileState`. That's the right semantics for materialize-all (which deletes its temp file after) but useless for lazy-download (which needs the file at the canonical sandbox path so the reader can open it). Documented in `LazyDownloadFinalizer.swift` header.

- **Cosmetic follow-up: missing cover after lazy-download promotion.** Because we promote directly via finalizer (skipping `BookImporter`), no cover image is extracted and the row keeps the placeholder cover icon. Minor — the book is openable, which is the bug #115 success condition. Tracking-worthy as a separate polish item if user feedback flags it.

- **Tailscale path latency was excellent.** 19.3 MB EPUB downloaded in 2.4s end-to-end (enqueue → didFinishDownload). No retry logic exercised in this verification — happy-path only.
