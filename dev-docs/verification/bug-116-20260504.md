## Why slice verification (not full picker UI walkthrough)

The fix is a single-line `NotificationCenter.post` per preplanted key. Both ends of the chain were already verified:

- **Sender side (this PR)**: unit test `preplant_postsBookFileStateDidChange_perRow` asserts that for a 3-entry manifest with no selections, three `.bookFileStateDidChange` notifications fire with the expected fingerprintKey set.
- **Receiver side (bug #115 verification)**: `LibraryView.onReceive(.bookFileStateDidChange)` was added in PR #210 and observed firing end-to-end in `dev-docs/verification/bug-115-20260504.md` AC #8 ("LibraryView re-rendered after notification"). The observer's force-refresh path is identical regardless of which producer posts the notification.

What's NOT directly observed in this run is the picker UI flow itself (Settings → Cloud Backup → Restore → SelectiveRestorePicker → toggle → Restore → library refresh). The picker UI was previously driven during the bug #115 reproduction and confirmed working; the only thing this fix changes is what happens AFTER `coordinator.restoreSelectively`'s preplant phase, which is what the unit test exercises.

A full UI walkthrough would require either (a) deleting the existing 3 books to clear state for a fresh restore, or (b) restoring from a different backup. Both add UI navigation cost without exercising any code path the unit test doesn't already cover.

## Observations

- **Why the existing notification rather than a new one?** `.bookFileStateDidChange` already exists, LibraryView already observes it, and the semantic ("a Book row's `fileState` changed") is exactly what's happening (rows transitioning from "doesn't exist" to "exists with `fileState=remoteOnly`"). Adding a new `.libraryDidGainRows` would have meant duplicating the observer wiring for no semantic gain.

- **Why post per-key instead of one bulk notification?** Two reasons:
  1. Consistency with the existing producers (`LazyDownloadFinalizer`, `LazyDownloadCoordinator.reattach`) which all post per-key with `userInfo` carrying `fingerprintKey` + `state`. Future consumers that want to filter by key (e.g., a row-level micro-update instead of full library refetch) can do so without ambiguity.
  2. The observer's re-entrancy guard already coalesces the refresh, so the cost of N posts is minimal.

- **Library row covers still missing** — the picker doesn't extract cover images for `.remoteOnly` rows because the file isn't local yet. Books restored via lazy-download finalize also miss covers (noted in bug #115 evidence). Tracking-worthy as a separate polish task if user feedback flags it.
