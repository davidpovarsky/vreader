## Why mock-only verification

All three bugs trip on rare-path conditions that don't cleanly reproduce on a live device:

- **#117** needs a fresh WebDAV server with MKCOL refusing for non-"already-exists" reasons (auth fail, server-side rate limit). Current rclone target accepts MKCOL on every path, so `mkcolInterceptor` simulation is the right repro.
- **#118** needs a SwiftData write to fail mid-pair (low memory, disk full, schema migration in flight). Live repro requires deliberately corrupted state. Unit tests cover both the atomic-write contract and the recovery path.
- **#119** needs a record-validation failure mid-batch. The unit test feeds a synthetic fingerprintKey-mismatch record to trip `PersistenceError.invalidContent` deterministically.

The `LazyDownloadReattachTests.reattach_canonicalFileOnDiskWithMatchingSHA_recoversToLocalAndSuppressesFailed` integration test is the strongest check — it wires a real `LazyDownloadFinalizer` into a real `LazyDownloadCoordinator`, seeds matching-and-non-matching rows, and asserts both branches behave correctly. That's the model for behavioral-WI verification when device repro is impractical.

## Observations

- **The audit caught 4 issues across 4 rounds.** Each round's finding was real — none were false positives. The first round's two findings were pre-existing in main; rounds 2 and 3 caught issues introduced BY my fix attempts. That's the value of the audit loop: it stops "fix introduces another bug" patterns from reaching merge.

- **`partialBulkInsert` enum addition** required threading through `ErrorMessageAuditor.sanitizePersistence`. The exhaustive-switch enforcement at compile time was the safety net — Swift refused to build until every switch handled the new case.

- **The TOCTOU window was real.** Between `await findBook(...)` returning and `await insertBook(...)` running, the actor CAN process another message — actor reentrancy is allowed at await points. Inlining into a single synchronous block (fetch + insert + save) closed the window because SwiftData's `try context.fetch/save` are non-async.

- **`tryPromoteFromDisk` is the missing link** between the lazy-download path and the persistence reconciler. Without it, a SwiftData write failure after the file move stranded valid bytes on disk. With it, the next launch's reattach pass detects the file and retries the promote — cheap because the bytes are already local.
