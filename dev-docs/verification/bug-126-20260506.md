## Commands run

```bash
# Build + install
DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer xcodebuild build \
  -project vreader.xcodeproj -scheme vreader \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' -configuration Debug
xcrun simctl install booted .../vreader.app

# Switch to native reading mode (EPUB defaults to unified-mode renderer for
# capabilities.contains(.unifiedReflow)). Native mode is required to
# exercise EPUBWebViewBridge / WKWebView at all.
DATA=$(xcrun simctl get_app_container booted com.vreader.app data | tail -1)
xcrun simctl terminate booted com.vreader.app
plutil -replace readerReadingMode -string native "$DATA/Library/Preferences/com.vreader.app.plist"
xcrun simctl launch booted com.vreader.app

# Reset → seed → open the mini-epub3 fixture
xcrun simctl openurl booted "vreader-debug://reset"
xcrun simctl openurl booted "vreader-debug://seed?fixture=mini-epub3"
KEY="epub:f284fd074ccd1d3c1a78985464d9e1be27975f4029f3c2ddef8428ca10684af4:2198"
xcrun simctl openurl booted "vreader-debug://open?bookId=${KEY}"

# Three eval probes
for js_pair in "arith:2 + 2" "title:document.title" "pcount:document.querySelectorAll('p').length"; do
  bridge="${js_pair%%:*}"
  js="${js_pair#*:}"
  js_b64=$(printf "%s" "$js" | base64)
  xcrun simctl openurl booted "vreader-debug://eval?bridge=${bridge}&js=${js_b64}"
done
```

## Observations

- **Routing trap that almost masked the fix**: EPUB defaults to `readingMode = unified` if no per-book override is set. In unified mode, `UnifiedTextRenderer` (TextKit) handles rendering and never instantiates `EPUBWebViewBridge`. The fix (which lives on EPUBWebViewBridge's coordinator) is invisible until the user / verifier switches to native mode. Real users who only read in unified mode will still see `evalUnsupported` — that's expected because there's no JS engine in TextKit. This is a correct response, not a regression.
- **iOS 26 simctl quirk**: `simctl spawn defaults write` writes to a host-level prefs domain rather than the app sandbox. Direct `plutil -replace` on the sandbox plist is the working path. (Not a bug fix; a one-line documentation point if the verification harness is rerun.)
- **The diagnostic log lines** added during debugging (`bug126: ...` in `makeUIView` and `didFinish`) were removed before the merge. The fix itself is silent — no production logging churn.
- The fix is **DEBUG-only** end-to-end: `activeEPUBWebView` is wrapped in `#if canImport(WebKit)` AND lives inside the `#if DEBUG` block of `DebugReaderRegistry`; the coordinator's setter is `#if DEBUG`-gated; the probe's jsEvaluator wire is inside the existing `#if DEBUG` `.onAppear` block. The Release-build symbol gate (`verify-release-no-debugbridge.sh`) continues to pass.
