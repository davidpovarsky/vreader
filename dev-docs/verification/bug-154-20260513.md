## Observations

- The 4 TXTReaderContainerSearchHighlightWiringTests pass cleanly. They confirm at source-text level that the orphan `@State` is absent and both `readerContent` + `chunkedReaderContent` paths read from `uiState.highlightRange`/`uiState.highlightIsTemporary`.

- The UITest `testSearchTapNavigatesToPosition` exercises the full runtime path on iPhone 17 Pro Simulator (iOS 26.5): seed Position Test Book → open → search "Paragraph 50" → tap result → assert reader visible at valid position. This confirms the notification pipeline fires: `onNavigate` in `SearchView` → `.readerNavigateToLocator` notification → `ReaderNotificationModifier` → `uiState.highlightRange` set → `TXTTextViewBridge` receives the wired highlight range.

- The temporary yellow highlight (3s auto-clear `UIColor.systemYellow.withAlphaComponent(0.4)`) was not directly asserted by XCUITest because it's too transient for reliable element-level assertion. The wiring unit tests confirm the render path is wired. The prior session's artifact `dev-docs/verification/artifacts/bug-154-after-fix-yellow-highlight-20260508.png` shows the yellow highlight visible at Paragraph 50 after the partial fix.

- The UITest required waiting for the background FTS5 indexer to complete before results appeared. The `searchResult_`-prefixed button pattern (vs `searchResultsList` OtherElement) proved more reliable for detecting results across the indexing timing window. On second+ runs the index is already built and results appear immediately.
