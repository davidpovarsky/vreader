# Bug #162 / GH #482 — Tap Zones config no-op on native readers (verification-exception)

This is a **pure-logic gate** fix. The gate is `static func ReaderSettingsPanel.shouldShowTapZonesSection(for capabilities: FormatCapabilities?, format: BookFormat?, currentMode: ReadingMode) -> Bool` and a private static helper `unifiedDispatchInstallsTapZoneOverlay(for: BookFormat) -> Bool` that mirrors the switch in `ReaderUnifiedDispatch.unifiedReaderView(fingerprint:)`.

Per `.claude/rules/10-tdd.md` "Pure-logic bugs reproducible by a unit test: the RED→GREEN transition in Phase 3 already establishes Verify — no extra step needed." Verification at the unit level is complete; the user-visible UI consequence is mechanically derivable from the gate function.

## Observations

- Gate signature divergence from bug #158's `shouldShowReadingModeSection(for:)`: the Tap Zones gate adds a `format: BookFormat?` parameter and a `currentMode: ReadingMode` parameter. The reading-mode picker is itself a mode-toggle and both modes are functional — gating on capability alone is sufficient there. Tap zones, by contrast, only honor configuration when the unified dispatch's `.tapZoneOverlay(...)` is on the live render path — which depends on (a) capability, (b) current mode being `.unified`, and (c) the format-specific dispatch case actually attaching the overlay (AZW3 doesn't; PDF isn't in the switch).
- Codex round-1 caught the loose-gate Medium and forced the dispatch-switch parity tightening. Without that tightening, AZW3 in Unified mode would have continued to show the picker as a no-op.
- Residual gap: complex-EPUB runtime fallback to native (`ReaderUnifiedDispatch.swift:73-76`) still shows the picker because `ReaderContainerView` passes the simple-EPUB capability set. Same gap as `chineseConversionSupported` and the bug #158 Reading Mode gate. Threading an `isComplexEPUB` runtime signal through the panel is feature-class scope and out of bug-fix scope.
- A simulator click-through verify on the merged build was attempted on iPhone 17 Sim with `CGEventPost`-driven taps (`/tmp/clickat.swift` from feature-25 round-4 evidence). The reset → seed → open → toggle-chrome path worked end-to-end, but the AA settings-button hit area required pixel-precise targeting that didn't land in 4 attempts. Independent of the fix; the unit-test coverage at the gate-function level is complete.
