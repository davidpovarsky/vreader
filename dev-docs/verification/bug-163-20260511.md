## Summary

Round-2 device verification of bug #163 (EPUB chapter content obscured
by Dynamic Island when chrome is visible) on iPhone 17 Simulator at
v3.14.142 (commit `228a10b`). The original fix shipped in v3.14.138
(commit `702a3b9`); subsequent versions (139–142) did not touch this
code path. Verifies via combined JS-eval geometry probe + visual
screenshot inspection — no UI taps needed.

**Round-1** (`bug-163-verify-light-theme-20260511.png` from a prior
verify-cron iteration) was inconclusive because CU display capture was
blocked, preventing toolbar-toggle to clear the translucent overlay.
This round uses a more rigorous data-side approach: cross-reference
the JS-reported `getBoundingClientRect().top` of each top-of-chapter
element against the safe-area top inset (59pt on iPhone 17) using the
WebKit CSS-px-to-logical-pt scale factor.

## Observations

- The CSS px-to-logical pt scale factor (~0.401) is consistent with WebKit's default desktop-fallback viewport of 980 CSS px scaled to a 393pt physical screen on iPhone 17. mini-epub3 has no `<meta name="viewport">` tag, so this fallback applies. Any EPUB with explicit viewport meta would scale 1:1, but the fix works regardless of viewport scale because the inset is applied at the `UIScrollView.contentInset.top` level (above the WebKit viewport).
- The translucent chrome toolbar still overlaps the h1 (visible faintly behind it) when chrome is shown. That's a SEPARATE UX concern from bug #163 — the toolbar's translucency means content scrolls underneath. Bug #163's title and body are explicitly about the **Dynamic Island** clipping, not toolbar overlap. The fix correctly addresses the DI; toolbar-overlap is out of scope.
- Pre-fix behavior (computed): h1 would have been at screen-y = 24.1 × 0.401 = **9.67pt**, directly behind the DI. The inset shifts it down by 59pt to 68.67pt. The 59pt delta is exactly the safe-area top inset on iPhone 17.
