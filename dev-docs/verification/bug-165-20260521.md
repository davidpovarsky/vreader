## Observations

- **Single-page chapter handling**: `mini-epub3.epub` has two chapters, each rendering as a single page in the WebView (`scrollWidth == viewportWidth == 980`). This means `totalPages == 1`, so page 0 is both the first AND last page. The router correctly classifies this — right-tap from a 1-page chapter wraps forward, left-tap wraps backward — as covered by `test_nextFromSinglePageChapter_isBothFirstAndLastPage` / `test_previousFromSinglePageChapter_isBothFirstAndLastPage`. The on-device verification implicitly exercises this path on every tap.
- **Multi-page in-chapter taps not exercised live**: `mini-epub3` does not have a chapter long enough to overflow into multiple pages, so the "tap-from-middle-of-chapter resolves to within-chapter `pageNavigator.nextPage()`" path is unit-tested only. The router is the single seam between in-chapter and cross-chapter behavior, so unit coverage is the right gate. A larger EPUB fixture would catch a regression that broke multi-page in-chapter handling — filed as informal follow-up.
- **§2.3 (continuous scroll mode) explicitly deferred**: §2.3 ("Chapters render one after the other in a single scrollable column" + lazy load + chapter divider + skeleton-pulse) is a fundamentally different rendering architecture than today's "one chapter per `loadFileURL`". Implementing it requires multi-chapter document stitching in the `EPUBWebViewBridge` plus a new lazy-load coordinator — feature-shaped, not bug-shaped. The bug row's primary symptom ("side-tap only toggles chrome") is the §2.2 path; users who picked scroll layout still see chapter-by-chapter navigation (with the same Contents menu they had before) — same behavior as before this fix. To be filed as a follow-up feature ("EPUB continuous cross-chapter scroll mode") referencing §2.3.
- **§2.2 bounce affordance ("subtle horizontal nudge animation") deferred**: design names the affordance but doesn't specify the curve / duration / haptic. Today both bounces collapse to no-op (matches the legacy clamp behavior). When the visual spec lands, the wiring is trivial — `handleSideTapNext` / `handleSideTapPrevious` already produce the `.bounceAtEndOfBook` / `.bounceAtStartOfBook` enum value at the right code point.
- **DebugBridge eval is the right test mechanism**: simulating the tap via `webkit.messageHandlers.contentTapHandler.postMessage({x, w})` bypasses the JS click-listener but invokes the same Swift coordinator handler the real click would. This proved that the entire chain — JS message → coordinator → `ReaderTapZoneRouter.dispatch` → `.readerNextPage` notification → `handleSideTapNext()` → `wrapForward()` → `viewModel.navigateNext()` → `contentURL` change → `EPUBWebViewBridge` `loadFileURL` — fires end-to-end.
