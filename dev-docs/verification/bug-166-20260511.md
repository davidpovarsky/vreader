## Summary

Device-side verification of v3.14.141 bug #166 **partial fix** (sub-piece
(a) only — slider-max raise from 32pt to 64pt). End-to-end signal proven
via DebugBridge JS eval + visual comparison screenshots. The
cross-format perceptual-calibration sub-piece (b) is OUT OF SCOPE for
this fix and remains open; this verification covers ONLY sub-piece (a).

**Net result**: the data-side fix (`TypographySettings.fontSizeRange
12...32` → `12...64`) flows correctly through `ReaderSettingsStore` →
`TypographySettings` → `EPUBReaderContainerView.readerContent`'s
`epubOverrideCSS(fontSize:...)` → injected `<style>` in the WKWebView,
and the body/p/h1 elements render at the new 64px size. Pre-fix, the
clamp would have capped the persisted value at 32px and the rendered
size would have been half of what's observed.

## Observations

- The JS-side probe returns exact `64px` (not a scaled value) — this matches expectation because `EPUBReaderContainerView.readerContent` passes `typography.fontSize` directly to `epubOverrideCSS(fontSize:)` which emits `font-size: 64` in the injected stylesheet.
- `window.innerWidth: 980` is the WebKit default desktop fallback (no viewport meta tag in mini-epub3 fixture). Logical scaling for the screen is handled by WebKit's auto-scale, but the CSS pixel value remains 64. This is the right behavior — the user setting controls CSS px, and WebKit handles the device-pixel-ratio scaling.
- The visual delta between 64pt and 32pt screenshots is dramatic and clear: at 64pt only 2-3 lines per column fit; at 32pt ~10 lines/column. Confirms the user-visible change is real and proportional.
- `h1` rendering at 128px (2× body) confirms the user-agent stylesheet h1 multiplier still works at the new bound; no caller-side clamp short-circuits it.
- The pre-existing `themeCSSIncludesLargeFontSize` test (32pt → `font-size: 32px`) plus the new `themeCSSAcceptsAppMaxFontSize64` test (64pt → `font-size: 64px`) bracket the AZW3/Foliate path. No regression risk on AZW3 from the raise.

## Residual

**Sub-piece (b) — cross-format perceptual inconsistency — REMAINS OPEN.**
TXT/MD use UITextView pt 1:1 with screen points; EPUB injects CSS px
which WebKit scales via the viewport; PDF and Foliate have their own
coordinate systems. The same numeric font-size value still renders at
different perceived sizes across formats. That's feature-class scope
(per-renderer measurement + unified pt→logical-size mapping) and is
NOT addressed by this raise. Bug #166 stays at `PARTIALLY FIXED` until
calibration work is planned + shipped. Recommended next step: file a
follow-up bug or split into a feature row with explicit acceptance
criteria for "same value, same perceived size across formats".
