## Observations

- **Seam contract is the strongest test boundary the bridge admits.**
  `EPUBWebViewBridge.applyScrollViewBackground(to:color:)` is unit-tested
  against a real `UIScrollView` (not a stub) â€” the test pre-sets `.red`,
  calls the seam, and asserts the resulting `scrollView.backgroundColor`
  matches the input. This locks the assignment behaviour for all 4 input
  classes (nil, light, sepia, dark) plus a custom `UIColor`. The remaining
  uncovered surface is the call-site wiring (`makeUIView` /
  `updateUIView`) and the OS-level paint of `UIScrollView.backgroundColor`
  during rubber-band â€” neither is realistically unit-testable without a
  full `UIViewRepresentableContext` mock or a physical-device pan harness.
- **`UIScrollView.backgroundColor` paint during rubber-band is a stable
  iOS guarantee** (since iOS 2.0): once the property is set, the OS
  paints it behind the content during overscroll. The fix can't
  logically fail unless (a) the call site never runs (covered by
  marker comments + code review), (b) the color value is wrong (covered
  by `ReaderTheme` tests), or (c) iOS itself changed how UIScrollView
  paints â€” none of which are vreader bugs.
- **Why mini-epub3 chapter 1 doesn't bounce**: `bodyHeight=874 <
  viewport=2131`, so `UIScrollView.contentSize <= bounds.size` and the
  scroll view is intrinsically non-bouncing. Forcing the document
  height via JS spacer + overflow override changed `documentElement.
  scrollHeight` (probe shows 6392) but the WKWebView's UIScrollView
  contentSize is computed from the layout pass and didn't update for
  the JS-injected spacer.
- **Synthetic gesture limit in iOS Sim**: even with Simulator
  frontmost (verified via `osascript`) and slow-stepped drags
  (60 steps Ă— 15ms), CGEventPost mouse drags do not deliver to
  WKWebView's UIScrollView pan recognizer. Same observation for
  `CGEventCreateScrollWheelEvent`. This is a tooling gap of the
  verify-cron harness, not a vreader bug. Earlier-session evidence
  files (`feature-25-20260510-round4.md`) showed CGEventPost taps DO
  work for click-targeted UI events, but pan/scroll gestures do not.
