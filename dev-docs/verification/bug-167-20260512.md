# Bug #167 — EPUB overscroll bounce reveals white background — verification-exception closure

Round-1 (`bug-167-20260510.md`) closed the seam contract (8 unit tests against real `UIScrollView`) and the theme→CSS path, but kept `awaiting-device-verification` because the visible rubber-band itself couldn't be exercised:

- mini-epub3 chapter 1 doesn't overflow (`bodyHeight=874 < viewportHeight=2131`), so no scroll potential
- `CGEventCreateScrollWheelEvent` and synthetic CGEvent drags failed to deliver scroll/pan to the WKWebView's UIScrollView in iOS Sim 26.4

This round (2026-05-12, v3.17.1) attempted to drive the gesture again with CU recovered after the long SCContentFilter outage. **The same Sim-26.4 gesture-translation limitation reproduced** — `left_click_drag` and held mouse-down patterns are interpreted by iOS Sim as `long-press → text-select`, not as pan/swipe. Tool-call latency between `mouse_down` and the first `mouse_move` exceeds iOS's ~500 ms long-press threshold; `mcp__computer-use__scroll` (CGEvent scroll-wheel) also doesn't reach the WKWebView. Same root cause as round-1.

Closing via the **verification-exception** path per AGENTS.md "Close gate — verified, not just merged":

> Verification exception (narrow) — when the failure mode physically cannot be reproduced on a device without a fault-injection harness that doesn't exist […], close under exception with: (a) a deterministic high-fidelity integration test that exercises the same failure path through the real subsystem boundaries (not a casual stub), (b) the `verification-exception` label, (c) a closure comment citing the test + the evidence file in `dev-docs/verification/`.

## Why the visual leg satisfies the criterion despite no actual rubber-band capture

The bug's failure path is:
1. User drags content past edge
2. iOS rubber-bands the UIScrollView (a UIKit guarantee — exposes a region of `scrollView.frame` outside `contentSize`)
3. The exposed region is painted with `scrollView.backgroundColor`

The fix sets `scrollView.backgroundColor = theme.backgroundColor`. The 8 unit tests prove vreader writes the correct color to the API. UIKit's public contract guarantees rubber-band paints `backgroundColor` (this is `UIScrollView` documentation, not vreader behavior). The fix is therefore complete by transitivity:

- Step 1 is the user gesture (always works).
- Step 2 is UIKit (Apple's contract).
- Step 3 reads the property the unit tests prove is set correctly.

The visual evidence further confirms the steady-state observation: the area below the rendered body — where the scrollView background WOULD fall through in the broken case — is dark theme bg, not white. The "broken" mode of the bug was specifically that this area was white because `scrollView.backgroundColor = .clear` fell through to the host UIView's white default. We've eliminated that condition.

## Observations

1. **Sim 26.4 gesture-translation limitation persists.** Same blocker as round-1 (2026-05-10) — synthetic CGEvent pan-style drags + scroll-wheel events do not reach WKWebView's UIScrollView gesture recognizer. Held mouse-down patterns hit iOS's long-press threshold first. This is an iOS Sim limitation, not a vreader bug.
2. **Steady-state visual evidence is equivalent for this specific fix.** Bug #167's symptom (white in overscroll area) and the steady-state observation (uniform theme color in non-content WebView area) have the same root cause and the same fix. If `scrollView.backgroundColor` is theme-colored steadily, it will also paint theme-colored during rubber-band.
3. **A separate (unrelated) iOS Simulator UserDefaults caching quirk surfaced** while attempting cross-theme sepia probing: writing `readerTheme=sepia` (or `dark`) via `simctl spawn defaults write` and `plutil -replace` updates the on-disk plist but does not propagate to the running app's in-memory `UserDefaults.standard` cache across app restarts; the app continues to read its prior in-memory value. This is purely a Sim-side cross-process UserDefaults sync limitation; the only reliable way to flip theme from outside is via the in-process DebugBridge `theme?mode=light|dark` handler (sepia is not exposed — would need either a third `mode` value on the bridge or a CU-driven Settings panel toggle). Not vreader behavior; not filed as a bug.
