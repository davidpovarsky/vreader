## Summary

Device verification of bug #168 (GH #498) — "EPUB font family setting has
no effect — `fontFamily` never injected into CSS" — closes the
`awaiting-device-verification` close gate. The fix landed via PR #503
(squash-merge `e5575b3`) and tagged `v3.14.130`.

Verified end-to-end on iPhone 17 Simulator (iOS 26.4) by cycling the
`readerTypography` UserDefault through all three `ReaderFontFamily`
cases (`system`, `serif`, `monospace`), launching the app fresh per
mode, opening the bundled mini-epub3 fixture, and probing the rendered
WKWebView CSS via DebugBridge `eval`. All three rendered the expected
font-family stack and computed body face.

## Observations

- **Verification path was DebugBridge `eval`, not synthetic gestures.**
  Tapping the AA → Font Family picker via `CGEventPost` was attempted
  but the small target (~17×17pt) sits inside an iOS-bezel offset
  region where the affine `mac_x = sim_x*0.377+399, mac_y = sim_y*0.341+88`
  derived from earlier-session calibration was off by ~30pt vertically.
  Rather than burn iterations on tap precision, swapped to a
  programmatic write of `readerTypography` to UserDefaults + relaunch +
  WKWebView eval. This is a strictly stronger verification: the eval
  reads `getComputedStyle().fontFamily` from the real WebKit cascade
  after the EPUB book CSS, the vreader theme CSS, and any user-agent
  defaults have all been applied — exactly what the user would see
  visually.
- **cfprefsd cache contention discovered.** The simulator's `cfprefsd`
  caches per-app UserDefaults; writing via host-side `plutil` directly
  to the on-disk plist may not propagate to a freshly-launched app
  unless `cfprefsd` re-reads the file (which it doesn't reliably do
  when the app is relaunched immediately after). The reliable sequence
  is: (a) `simctl terminate`, (b) `simctl spawn defaults write` (goes
  through cfprefsd, not the file), (c) `simctl launch`. This pattern
  worked cleanly for the system → monospace transition. Documenting
  here for future verify-cron iterations that need to manipulate
  app preferences.
- **Cascade evidence in style snippets.** The captured style snippets
  show `font-family: <stack> !important;` inside `html, body { ... }`
  for all 3 modes, plus the `body * { font-family: inherit !important; }`
  sweep below it (per the bug fix's broad descendant override). Eval
  probes confirmed paragraph + heading both inherit correctly.
- **Monospace screenshot shows visually distinct typeface** — the
  rendered text "Second paragraph. Lorem ipsum dolor sit amet,
  consectetur adipiscing elit. Sed do eiusmod tempor incididunt ut
  labore et dolore magna aliqua." is clearly in a uniform-width
  monospace font (compare against system/serif which are
  proportional). This is the user-visible behavior change the fix
  enables.
