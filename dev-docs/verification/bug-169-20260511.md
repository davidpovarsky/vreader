## Summary

Round-1 verification of bug #169 (AI Settings expanded sections don't
render after flipping the AI Assistant toggle ON until app relaunch)
at v3.14.144 (commit `506b1c2` shipped in v3.14.140; later commits 141–144
did not touch `AISettingsViewModel.swift`).

**Result: PARTIAL** — high-fidelity Observation-tracker unit test
passes (the same `withObservationTracking` mechanism SwiftUI uses to
subscribe to `@Observable` properties), but UI gesture drive to flip
the actual toggle was blocked this session by a CU MCP capture outage
(`CU display unavailable` despite a real monitor attached and
`screencapture -x` succeeding at OS level). Issue stays open with
`awaiting-device-verification` label until a session can drive the
toggle UI.

## Observations

- The baseline screenshot at v3.14.144 confirms the build is correctly
  installed and the Library view renders. The Settings gear icon is
  visible in the upper-left; the AI toggle path is reachable.
- The high-fidelity test `toggleNotifiesObservationTracker` uses
  `withObservationTracking { _ = vm.isAIEnabled } onChange: { ... }` —
  this is the identical Observation-framework API that SwiftUI uses
  internally to subscribe a `View.body` evaluation to its observed
  properties. If the unit test's `onChange` fires after the toggle
  setter writes, SwiftUI's body invalidation also fires (same mechanism;
  there is no intervening gate). This is the strongest possible
  in-test fidelity for an `@Observable` re-render bug short of running
  the actual SwiftUI runtime.
- The remaining gap for `FIXED` closure is purely user-visible-gesture
  confirmation: tapping the toggle in the live Settings sheet and
  watching the API Key / Provider Configuration / Data & Privacy
  sections appear without close/reopen.
