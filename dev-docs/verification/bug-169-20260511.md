## Summary

End-to-end verification of bug #169 (AI Settings expanded sections
don't render after flipping the AI Assistant toggle ON until app
relaunch) at v3.14.144 (commit `506b1c2` shipped in v3.14.140;
later commits 141–144 did not touch `AISettingsViewModel.swift`).

**Result: PASS.** Round-1 (this iteration's earlier pass) confirmed the
high-fidelity Observation test gate via `withObservationTracking` (same
mechanism SwiftUI uses internally). Round-2 (this iteration's later
pass after CU-substitute technique was located) drives the actual
toggle gesture in the live Settings sheet via `osascript`-derived mac
coords + `CGEventPost`, confirming visible end-to-end behavior: API
Key / Provider Configuration / Data & Privacy sections appear and
disappear in lock-step with the toggle state, in a SINGLE sheet
session, with no close/reopen and no kill+relaunch.

The CU MCP outage (`CU display unavailable`) was bypassed by querying
the Simulator process's Accessibility tree directly (the technique
documented in `dev-docs/verification/feature-38-20260510-round3.md`).

## Observations

- The simulator initially showed toggle ON because the app's prior
  state (model `inclusionai/ring-2.6-t:free` configured, API Key pin
  set) was preserved across `simctl install` — `defaults delete` of
  the flag key didn't affect the in-process FeatureFlags state, only
  removed the explicit override at the disk level. The relevant
  state-machine test is the toggle's behavior under user interaction,
  not the disk-default-vs-runtime mismatch.
- The 3-screenshot cycle pins the lock-step:
  1. **toggled-on (initial)**: API Key + Provider Configuration visible
  2. **toggled-off (mid-cycle)**: only "About" section below "Enable AI Assistant" (no API Key, no Provider Configuration)
  3. **toggled-on (final)**: API Key + Provider Configuration reappear identically to (1)
- Pre-fix, step (3) would have shown the toggle as green but the conditional sections NOT reappearing (the failure mode the bug body documents). Post-fix observes them reappearing.
- The CU-substitute technique (osascript accessibility-tree query → mac-space coords → CGEventPost) is the canonical fallback when MCP `screenshot` reports `CU display unavailable` while real screen capture is healthy. Documented in feature-38 round-3 evidence; reused successfully here.
