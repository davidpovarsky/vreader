# Bug #180 / GH #614 — TXT continuous-scroll reader verification

## Verification class: verification-exception (high-fidelity integration test)

Bug #180 is a UI/behavioral bug. The fix renders chaptered TXT in
Scroll layout as one continuous scrollable surface (reuse the chunked
`UITableView` windowing) with `currentChapterIdx` DERIVED from scroll
offset. The user-observable acceptance — "scroll from the end of
Chapter 1 straight into Chapter 2 with no jump, no cascade, no
chrome-button tap" — requires driving a continuous cross-chapter
scroll gesture and observing the result.

**On-device headless verification is genuinely infeasible on this
host.** Documented blockers (all hit and recorded during this run):

1. **computer-use is DOWN** — the host is virtual-display-only;
   `mcp__computer-use__screenshot` is unavailable per the bugfix-cron
   environment.
2. **`vreader-debug://` URLs are gated behind an iOS confirmation
   dialog.** Every `xcrun simctl openurl "$SIM" "vreader-debug://…"`
   raised iOS's "Open in 'vreader'?" system confirmation dialog
   (screenshot artifacts below). `simctl openurl` does not auto-confirm
   it; the URL never reaches the app's `onOpenURL` handler, so
   `reset` / `seed` / `snapshot` produced no DebugBridge output
   (`Library/Caches/DebugBridge/` stayed empty).
3. **The Simulator window is not AX-accessible on this host.**
   `osascript`/`System Events` returns "Can't get window 1 of process
   Simulator (-1719)", so the `sim-drive-fallback` toolkit cannot
   resolve tap coordinates for the confirmation dialog.
4. **`clickat.swift` (CGEventPost) cannot be aimed** without the AX
   window rect, and races Terminal-window focus. Coordinate-guess taps
   from screenshot geometry did not land on the dialog's "Open" button.
5. The DebugBridge `eval` command runs JS in a **webview** reader only
   — TXT renders in a `UITextView`/`UITableView`, not a webview — and
   the `snapshot` command's `position` field is unpopulated for TXT
   (`partial: ["selection", "position"]`). Even with the dialog
   bypassed, there is no headless primitive to drive a TXT scroll
   gesture or read a TXT scroll offset.

These blockers match the bug's own scope note and the fix-issue
prompt's anticipation ("DebugBridge reader-mount nav has been flaky
headlessly for siblings"). They are confirmed tool-unavailability, not
a discipline lapse — see `## Commands run` for the reproducible
evidence.

Per AGENTS.md "Close gate — verification exception (narrow)", this bug
is verified via a **deterministic high-fidelity integration test that
exercises the same failure path through the real TXT reader subsystem
boundaries** — not casual stubs.

## Observations

- The fix's root-cause correctness is fully covered by the integration
  suite: the rejected PR #681 design (boundary-detect-then-swap, which
  jumped + cascaded to the wrong chapter position) is *removed*, not
  patched — `TXTScrollBoundaryChapterNavTests` (which pinned the
  rejected behavior) is deleted, and the `didScrollPast*` protocol
  callbacks, the coordinator's boundary block, and the
  `isChapterNavInFlight` serializer are all gone. With no boundary
  predicate, the cascade is structurally impossible.
- The continuous surface speaks one coordinate space — document-global
  UTF-16 — end to end: chunk offsets, `currentOffsetUTF16`, restore
  offset, and persisted highlight ranges. The chapter-local hop that
  the single-`UITextView` Paged path needs is simply not present, so
  the rejected design's failure mode (chapter-local↔global round-trip
  during scroll) cannot recur.
- Codex audit (3 rounds, thread `019e3e68`): round 1 found 1 High
  (`shouldOpenContinuous` would route all scroll-mode TXT to
  continuous) + 1 Medium (restore trusted the saved idx) + 1 Low; all
  fixed. Round 2's Medium was withdrawn after evidence that the
  `chunkedReaderContent` path was a dead fallback. Round 3: clean,
  ship-as-is. Audit log:
  `.claude/codex-audits/fix-issue-614-txt-continuous-scroll-v2-audit.md`.
- Headless TXT-reader verification on this host is a recurring gap —
  there is no DebugBridge primitive for a TXT scroll gesture, and the
  custom-URL-scheme confirmation dialog blocks DebugBridge entirely
  when `simctl openurl` is the only driver and CU is down. A future
  improvement would be a `vreader-debug://scroll?offset=N` command or
  pre-granting the URL scheme — tracked informally; not in this bug's
  scope.
