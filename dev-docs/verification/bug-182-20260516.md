# Bug #182 / GH #621 — Post-fix verification (round 2 outcome: PARTIAL)

## Result summary

The round-2 fix in v3.22.19 (commit `1fc950e`) addresses one of two
defects on the search-result-tap-to-highlight path:

1. **Snippet markup → plain-text matchers**: FIXED. `SearchHitToLocatorResolver`
   now strips `<b>...</b>` and `...` ellipsis markers from
   `hit.snippet` before passing to `Locator.textQuote`. Verified
   end-to-end: when the cleaned text is fed to `window.find()`, the
   matcher succeeds and the `.vreader_search_highlight` span is
   injected. (Evidence below.)
2. **Deferred-eval pipeline silence in user-tap flow**: NOT YET RESOLVED.
   The user-triggered cross-chapter search-tap still leaves
   `document.querySelectorAll('.vreader_search_highlight').length === 0`
   after navigation completes. The same JS that succeeds via direct
   `vreader-debug://eval` does NOT appear to execute via the
   pendingHighlightJS stash + didFinish consume pipeline.

Per the close-gate rule "verified, not just merged," GH #621 must
remain open with `verification-blocked` because the user-visible
symptom (no yellow highlight on cross-chapter tap) persists despite
the merged fix.

## Observations

The asymmetry between Step B (count=0 via user-tap) and Step C
(count=1 via direct injection) is decisive: **the same JS payload
that the post-fix resolver SHOULD be sending is in fact runnable
against the live chapter 2 DOM** (Step C proves this). The fact that
Step B observes no span means one of:

- (a) `pendingHighlightJS` is being cleared between stash time
  (`updateUIView`) and consume time (`didFinish`) — a race with the
  immediate-eval tail branch that I couldn't reproduce by analysis
  but the symptom strongly suggests.
- (b) `pendingHighlightJS` arrives at `didFinish` but
  `evaluateJavaScript` runs against a stale webview (e.g., the
  outgoing chapter-1 webview's `didFinish` fires AFTER the new
  chapter-2 webview is wired up, consuming the new chapter's
  `pendingHighlightJS` against the OLD DOM).
- (c) The resolver-fix JS itself never reaches the container's
  `pendingHighlightJS = js` line — i.e., `locator.textQuote` is nil
  at the moment the search-tap notification fires. This would happen
  if the resolver's call site (search ViewModel) caches the locator
  pre-fix.

To distinguish among (a)/(b)/(c), the next iteration needs a
sentinel: inject the search-highlight JS through a probe that writes
`window.__vreader_lastSearchTap = { text, timestamp }` at the start,
so the deferred-eval consume path can be verified by reading the
sentinel post-tap.

The 144-line content seam at chapter 2 (chapter rendering above the
chrome / DI; visible only when chrome is hidden) is a SEPARATE
rendering issue, not part of Bug #182. Filing follow-up if not
already tracked.
