## Verdict

Bug #201's core acceptance — **users can now create a Highlight in AZW3/MOBI books** — is verified end-to-end. The fix's scope (the missing `case "selection":` + the action-sheet → persist → live-paint chain) works as designed.

The visual re-paint on reopen is a distinct missing capability that was never in Bug #201's stated repro or fix direction. The issue body's "survives reopen" language is satisfied at the data layer (Annotations panel + `highlightCount` confirm the row persists). The visual gap is now tracked as Bug #207 / GH #765 and will be picked up by a future bugfix-cron iteration.

`result: pass` for Bug #201's scope; Bug #207 carries the residual visual-restore work.

## Observations

- The Bug #201 fix lights up the entire selection → persist → paint chain on AZW3 for the first time. The new `.foliateRequestAnnotationJSCreate` create-token observer in the Coordinator (sibling of Bug #199's delete-token) does what it claims — the overlay paints immediately without waiting for a reload.
- The action sheet shows the selection preview ("most" in this run) and the Highlight + Cancel buttons. iOS's default WKWebView menu also appears in parallel for the first long-press; on subsequent long-presses (with the prior selection still partial) it tends not to re-appear. Neither blocks the vreader flow.
- The visual restore-on-reopen gap is the next slice to close for AZW3 highlight UX. The path is straightforward: extend `FoliateSpikeView.Coordinator.handleMessage` with `case "create-overlay":` (mirrors Bug #201 pattern) and emit a batch of `FoliateHighlightRenderer.addAnnotationJS` calls for the saved highlights on each section.
- Tested in v3.23.16's just-merged state on iPhone 17 Pro Sim (iOS 26.5) under Debug. No crashes, no `lastError` entries from DebugBridge across the run.
