# Bug #217 (GH #839) — ReplacementTransform regex no-op under dispatch-pool saturation

Verification-exception closure. The production failure mode cannot be
reproduced by interacting with the app UI; it is verified by a
deterministic high-fidelity test that drives the exact failure path
through the real `ReplacementTransform` against a real saturated
`DispatchQueue.global()`.

## Acceptance criteria

The bug: `ReplacementTransform.applyRegexRule` dispatched the regex match
to `DispatchQueue.global()` and blocked the caller on a 1 s
`DispatchSemaphore`. Under dispatch-pool saturation the work item could
not get a thread within the window, the wait spuriously timed out,
`matches` became `[]`, and the rule returned the input unchanged — a
silent no-op of the content-replacement rule.

| # | Criterion | Observed | Result |
|---|---|---|---|
| 1 | A regex replacement rule still applies when `DispatchQueue.global()` is fully CPU-saturated | `regexRule_appliesUnderGlobalQueueContention` saturates every pool thread (`activeProcessorCount * 4` spinners) for ~1.6 s — longer than the old 1 s timeout — then runs the `\d+`→`#` rule on `"page 1 of 100"` through the real `ReplacementTransform.transform()`; result was `"page # of #"`. Passed after 1.600 s. | pass |
| 2 | The dispatch-pool / semaphore-timeout machinery is removed from `applyRegexRule` | `applyRegexRule` now calls `regex.matches(in:options:range:)` synchronously on the calling thread; `DispatchQueue.global()`, `DispatchSemaphore`, `DispatchWorkItem`, and the `regexTimeoutSeconds` constant are all gone (PR #841 diff). | pass |
| 3 | The always-on regex regression guards — the exact tests that flaked under the parallel suite when #217 was discovered — pass on the merged build | `regex_multipleMatches` and `replace_regex_groupCapture` both passed; all 15 `ReplacementTransformTests` passed (`✔ Test run with 15 tests in 1 suite passed`). | pass |
| 4 | RED established — the contention test fails on the pre-fix commit | Codex audit log records the empirical RED: on the pre-fix commit the `\d+`→`#` rule returned `"page 1 of 100"` unchanged and the test FAILED; post-fix it PASSES. | pass |

## Observations

This is a verification-exception, not a device-verification, closure. The
production symptom — a content-replacement rule silently no-op'ing — only
occurs while the global dispatch pool is saturated at the exact moment the
rule runs. That condition cannot be created by tapping around the app: it
requires programmatically pinning every `DispatchQueue.global()` worker
thread. So a real high-fidelity test substitutes for a device repro.

`regexRule_appliesUnderGlobalQueueContention` is that test, and it is not a
stub: it spins `activeProcessorCount * 4` CPU-bound work items on the real
`DispatchQueue.global()` and then exercises the real
`ReplacementTransform.transform()` — the same code path the production
failure hit. The Codex audit log records that on the pre-fix commit the
test FAILED (rule no-op'd), confirming it reproduces the bug; on merged
`main` it PASSES in 1.600 s because the fixed code matches synchronously
and has no dependency on the dispatch pool.

The test stays gated opt-in (`VREADER_RUN_CONTENTION_TESTS=1`) in normal
runs because a faithful pool-starvation repro must CPU-saturate the
default-QoS pool and is antisocial to the parallel scheduler — no QoS
setting can both starve the pool the buggy code used and spare neighbours.
For this verification the gate was enabled by injecting the env var into
the `.xctestrun`. The everyday regression guard is the always-on pair
`regex_multipleMatches` + `replace_regex_groupCapture`, both green here.

Residual (accepted, noted in the bug row): a pathological user-authored
regex is no longer apply-time-bounded. The removed timeout was only weakly
protective — the caller blocked up to 1 s anyway and `DispatchWorkItem.cancel()`
cannot preempt a running `regex.matches`. A real guard belongs at
rule-creation-time pattern validation, not apply-time; out of scope here.
