# Bug #232 / GH #960 — Chunked TXT bridge clears a temporary search highlight on a new search + user scroll

Verification of the fix shipped in v3.36.8 (PR #968, merge commit
`e2960bb`). Closed under the close-gate **verification-exception** clause
(`AGENTS.md`): the precise failure-mode delta is sub-3-second and below
the available device-screenshot sampling rate. The qualifying evidence
is a deterministic high-fidelity integration test driving the real
subsystem boundaries, plus an on-device no-crash / search-nav pass.

## Observations

- **Why verification-exception, not full device repro**: the temporary
  search highlight has a 3 s auto-clear timer. The bug's behavioral delta is
  "the highlight clears *immediately* on a new search / user scroll vs. it
  *lingers* until the 3 s timer fires." Both observing "highlight visible"
  and observing the sub-3-second clear delta require sampling the screen
  faster than ~3 s after a search-result tap. A computer-use
  tap→screenshot round-trip is ~3-5 s; even `xcrun simctl io screenshot`
  fired right after a CU tap lands ~1-2 s late, and across 4 rapid captures
  every frame was post-auto-clear (identical byte size). This is the same
  limitation the Bug #154 / GH #443 verification trace documented
  ("computer-use screenshots... routinely land past the 3s auto-clear
  window"). The failure mode is not physically observable at the available
  sampling rate.
- **What the high-fidelity test covers**: `TXTChunkedSearchHighlightClearTests`
  is not a casual-stub unit test. It constructs real
  `TXTChunkedReaderBridge.Coordinator` instances, applies highlights through
  the real `applyHighlight`, posts a real
  `NotificationCenter.default.post(name: .searchHighlightClear)` (the exact
  notification `SearchViewModel` posts on a new query), drives the actual
  `scrollViewDidScroll(_:)` `UIScrollViewDelegate` callback with real
  `UIScrollView` instances, and asserts on the real coordinator state +
  the `onTemporaryHighlightCleared` callback. It exercises the same code
  paths the production failure hits. RED was proven by neutering the helper.
- **On-device confidence add**: the merged build's chunked reader opened,
  search-tap navigation worked twice, and — critically — the new
  `Coordinator.init` `.searchHighlightClear` observer + `deinit` +
  `clearTemporaryHighlightIfNeeded` all executed live (each new search query
  posts `.searchHighlightClear`), with zero crash / hang. A retain-cycle
  deinit bug, a bad `MainActor.assumeIsolated`, or a force-unwrap in the new
  code would have surfaced during these interactions.
- **Brittle for next time**: any future device-verification of a
  temporary-highlight behavior needs either a DebugBridge command to
  freeze/extend the auto-clear timer, or a DebugBridge query to read the
  TXT chunked bridge's highlight state (`eval` only reaches webview readers,
  not the `UITableView` TXT path). Worth a harness feature.
