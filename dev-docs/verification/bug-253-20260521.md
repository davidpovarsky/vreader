# Bug #253 — DebugBridge `present?sheet=…` sheet-presenter

Pre-FIXED slice verification (Gate 5 / `/fix-issue` Phase 6a) of the new
`vreader-debug://present?sheet=<toc|highlights|ai|settings|bookmarks>[&tab=<...>]`
command. Verified CU-free using `xcrun simctl openurl` + `simctl io screenshot`
(OS-level capture, no computer-use needed — which is the whole point of this
fix, since CU is structurally unavailable on this host).

## Observations

- The fix does exactly what the bug asked for: a previously-unreachable
  (CU-free) verification surface — the open sheet — is now driveable from
  `simctl openurl`. The TOC + Highlights + Settings sheets render their real
  content; the snapshot confirms the command is accepted (lastError null).
- The `DebugSnapshot` schema does NOT expose presented-sheet state, and `eval`
  only reaches webview bridges (EPUB/AZW3), so the *presence* of a SwiftUI
  sheet is confirmed via OS-level `simctl io screenshot` rather than the
  snapshot/eval JSON. That's sufficient and CU-free — `simctl io` works on the
  booted simulator regardless of computer-use availability.
- The AI gate behaving as a no-op (vs an empty sheet) when no provider is
  configured matches the chrome AI button precisely — important so a verify run
  that forgets `provider?action=add` fails loudly (no AI content) rather than
  showing a misleading empty AI sheet.
- Brittle for next time: the seeded book's `fingerprintKey` is content-derived;
  reading it from `ZBOOK.ZFINGERPRINTKEY` is the reliable way to get the
  `open?bookId=` value (don't hardcode).
