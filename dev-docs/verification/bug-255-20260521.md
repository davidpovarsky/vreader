# Bug #255 — DebugBridge `ai` action command (pre-FIXED verification)

**presented** AI sheet exposes, so the AI-response-card render states become
CU-free verifiable. This is the last harness leg blocking Feature #65 rows
3/6/11 + Feature #69 criteria 7-8.

## Commands run

```bash
SIM=1FAB9493-B97E-48F0-96C7-44A8E5AAA21E

# Build + install the worktree DEBUG binary (path resolved via -showBuildSettings,
# not a global find, to avoid picking a sibling worktree's build).
xcodebuild build -project vreader.xcodeproj -scheme vreader -configuration Debug \
  -destination "platform=iOS Simulator,id=$SIM" -derivedDataPath build/issue-1112
xcrun simctl install "$SIM" build/issue-1112/Build/Products/Debug-iphonesimulator/vreader.app

# Launch with BOTH flags (--enable-ai is a no-op without --uitesting: the
# AITestOverride.forceAvailable assignment lives inside `if config.isUITesting`).
xcrun simctl launch "$SIM" com.vreader.app --uitesting --enable-ai

# Seed + open a TXT book (DebugBridge seed computes the real file SHA).
xcrun simctl openurl "$SIM" "vreader-debug://reset"
xcrun simctl openurl "$SIM" "vreader-debug://seed?fixture=war-and-peace"
xcrun simctl openurl "$SIM" "vreader-debug://open?bookId=txt:bd8285a8…:1705"
xcrun simctl openurl "$SIM" "vreader-debug://settle?token=t1"

# Configure the OpenRouter provider (key url-encoded from .secrets/.env, never logged).
xcrun simctl openurl "$SIM" "vreader-debug://provider?action=add&name=OR&kind=openAICompatible&endpoint=https://openrouter.ai/api/v1&apiKey=<ENCODED>&model=openai/gpt-4o-mini&active=true"

# Present the AI sheet, then FIRE the action (the Bug #255 command under test).
xcrun simctl openurl "$SIM" "vreader-debug://present?sheet=ai&tab=summarize"
xcrun simctl io "$SIM" screenshot bug-255-prefix-ai-idle-20260521.png        # idle baseline (Section chip)
xcrun simctl openurl "$SIM" "vreader-debug://ai?action=summarize&scope=chapter"
# (wait ~12s for the OpenRouter stream)
xcrun simctl io "$SIM" screenshot bug-255-prefix-ai-summarize-chapter-20260521.png   # Chapter chip + summary card

xcrun simctl openurl "$SIM" "vreader-debug://ai?action=chat&text=Who%20wrote%20this%20book%3F"
xcrun simctl io "$SIM" screenshot bug-255-prefix-ai-chat-20260521.png        # Chat tab + assistant bubble
```

Every `snapshot` after `provider?action=add`, `present`, and `ai?...` showed
`lastError: null`.

## Observations

- The before/after contrast is the cleanest evidence: the idle screenshot has
  the **Section** chip filled; after `ai?action=summarize&scope=chapter` the
  **Chapter** chip is filled and the summary card is present. This single pair
  proves both the scope-chip selection (Feature #69 crit 7) and the
  `.complete` card render (Feature #65 row 3).
- The chat fire switched the panel to the Chat tab and produced a real
  assistant reply — Feature #65 row 6.
- The AI panel is a SwiftUI sheet, not a WKWebView, so `snapshot`/`eval` can't
  read its card text directly (those target the reader webview). The
  authoritative AI-card evidence is therefore the screenshot, which is fine —
  the command's job is to make the card *renderable* CU-free, and a `simctl io
  screenshot` is CU-free.
- Gotcha confirmed for future runs: `--enable-ai` alone is inert; you need
  `--uitesting --enable-ai`, and `--uitesting` swaps the SwiftData store so a
  bridge `reset` + `seed` is needed to land a book.
