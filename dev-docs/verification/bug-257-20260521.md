# Bug #257 — DebugBridge `open?position=N` host-side seek

The `vreader-debug://open?bookId=…&position=N` URL parsed + resolved the
position but the host-side seek was a documented deferred no-op — opening the
book did not move the reader, and `snapshot` always reported `position: null`.
This blocked CU-free verification of any flow needing a non-zero reading
position (Feature #69 criterion 8 — populated Book-so-far).

The fix wires the seek through the production `.readerNavigateToLocator` path
and surfaces the live reading position into the DebugBridge probe so
`snapshot.position` reflects where the reader actually landed.

## Observations

- The war-and-peace DEBUG fixture is a tiny 1705-byte / 1705-char ASCII file
  that renders in **paged** TXT mode. The seek scrolls to the *page* containing
  the requested offset, and `TXTReaderViewModel.updateScrollPosition` reports
  the **page-top** offset — so `position=1000` lands on the page whose top char
  is 949. This is correct paged-reader behavior, not a seek defect: the readback
  is the genuine reading position, and distinct non-zero positions are now
  reachable CU-free (the whole point of the bug). A continuous-mode book would
  land exactly on the requested offset.
- Two OSLog gotchas worth recording for the next harness run: (1) the default
  `simctl spawn … log show` level drops `.info` lines — pass `--info --debug` or
  DebugBridge `log.info(...)` output is invisible; (2) `simctl openurl` exits 0
  even when LaunchServices is still gating the scheme, so the
  `grant-debug-scheme-approval.sh` pre-grant (bug #123) is mandatory before the
  first URL or the handler never fires.
- The `--uitesting` in-memory store wipes between launches, so reset → seed →
  open → snapshot must run in a single launch session (it does here).
- `position` dropped out of the snapshot `partial` array once the live position
  was wired — itself a signal the field is now populated rather than
  best-effort-nil.
