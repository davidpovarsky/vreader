fix (`onAdvance` per-tick callback bridging the auto-turn timer to
`syncPagedState()` + position persist) makes the MD paged view advance and the
persisted position climb monotonically with no user interaction — the round-9
criterion-5 failure is gone.

## Root cause + fix

`AutoPageTurner.scheduleTimer()` advanced `navigator.nextPage()` (mutating the
navigator's internal page) but never synced the observable
`uiState.pagedCurrentPage` that drives `NativeTextPagedView`'s explicit
`currentPage` render param + `snapshot.position`. The sync
(`TextReaderUIState.syncPagedState()`) was called ONLY from the `.readerNextPage`
observer, which the timer bypasses.

**Fix shape chosen: (b), not (a).** The bug row's fix-direction (a) — have the
timer POST `.readerNextPage` — is FATAL: that observer ALSO calls
`autoPageTurner?.pause()` (Bug #131's "manual turn pauses auto-turn" semantics),
so routing the timer through it halts auto-advance after a single tick. Instead,
`AutoPageTurner` gained an `onAdvance: (@MainActor () -> Void)?` callback fired
after each timer-driven `nextPage()`. `TextReaderUIState.updateAutoPageTurner`
installs it to run `syncPagedState()` + a container-supplied
`onAutoAdvancePersist` (which calls `viewModel.updateScrollPosition`) — the SAME
sync the `.readerNextPage` observer does, MINUS the `pause()`. This routes the
timer's advance through the single existing sync method without re-entering the
pause path.

## Observations

- **The fix is verified by the same CU-free method that caught the bug** in
  round-9: position-delta over time. Pre-fix `0/0/0/0`; post-fix monotonic
  `1085→3523→4758→7258→8440`. The harness cleanly distinguishes "advances" from
  "doesn't" with no computer-use.
- **Auto-stop at last page works**: position plateaus at `8440` (≈ document end,
  6 pages) once the turner reaches the last page — `AutoPageTurner`'s last-page
  guard (`nav.currentPage >= nav.totalPages - 1`) returns before `nextPage()`, so
  `onAdvance` correctly does NOT fire there.
- **Build-tooling gotcha (not part of the fix)**: an INCREMENTAL rebuild silently
  drops the injected `vreader-debug` URL scheme (the inject build-phase has no
  declared outputs, so Xcode skips it on incremental builds, and the regenerated
  Info.plist loses the types). Symptom: `simctl openurl` → error 115. A clean
  build restores it. Verification builds for DebugBridge-driven repro should use
  `xcodebuild clean build` (or re-run the inject phase) when iterating in one
  DerivedData dir.
- **Scope**: MD verified live. TXT shares `NativeTextPagedView` + the
  `TextReaderUIState.updateAutoPageTurner` path, so the fix wires TXT
  symmetrically — but `FormatCapabilities.autoPageTurn` is MD-only (TXT lost it
  via bug #157), so TXT auto-turn isn't reachable at runtime today. The TXT wiring
  is defensive for a future re-enable; not separately verifiable now.
