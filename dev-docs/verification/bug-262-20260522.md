## Observations

- The `mini-azw3` fixture (Project Gutenberg "The Masque of the Red Death") ships
  a real 3-entry TOC with `filepos:` hrefs. Foliate-js `view.goTo(target)` accepts
  these `filepos:` hrefs directly — confirming the CFI-then-href target resolution
  in `FoliateNavSeek.navigationTarget` is correct for AZW3 (the TOC entries carry
  hrefs, not CFIs).
- The whole TOC pipeline (`book-ready` JS → `parseBookReady` → `FoliateBookInfo.toc`
  → `FoliateTOCConverter`) already existed; the only break was the
  `onBookReady: (String) -> Void` boundary dropping everything but the title. The
  fix is purely additive wiring — no new parser, no new UI.
- The `.readerPositionDidChange` produce on the live path was the subtle half:
  pre-fix only the dead `FoliateReaderContainerView` posted it, so AZW3 AI-panel
  context + the DebugBridge `snapshot.position` were both blind. The snapshot
  showing a live, navigation-responsive position is the proof it now fires.
- CU (computer-use) was unavailable during this run (`screenshot` returned
  "CU display unavailable"), so the row-tap front-half (`.readerNavigateToLocator`
  → `.foliateRequestSeekTarget`) was verified by unit tests + the `goTo` back-half
  was verified live; `simctl io screenshot` (which does not need CU) captured the
  populated TOC sheet. The full chain is unit-tested end-to-end in
  `FoliateTOCNavWiringTests`.
