# Bug #263 — DebugBridge `seed-sessions` command (verification)

CU-free pre-FIXED verification of the new `vreader-debug://seed-sessions`
command. The bug: the reading dashboard (Feature #58) reads `ReadingSession`
SwiftData records, but the harness could seed only books, so the dashboard
rendered all-zero data CU-free — blocking Gate-5 verification of Feature #58
criteria b/c/d. The fix adds a DEBUG-only command that seeds a deterministic
6-session spread.

## Acceptance criteria

| # | Criterion | Observed | Pass |
|---|---|---|---|
| 1 | Before the fix, the harness leaves `ReadingSession` empty after seeding a book | After `reset` + `seed?fixture=war-and-peace`: 1 `ZBOOK` row, `ZREADINGSESSION` count = **0** — the exact bug symptom reproduced | PASS |
| 2 | `seed-sessions?book=<key>` inserts synthetic `ReadingSession` rows via the real persistence boundary | After `seed-sessions?book=<war-and-peace key>`: `ZREADINGSESSION` count = **6**, all attached to the correct fingerprint key | PASS |
| 3 | Sessions are anchored across the standard windows so per-window totals differ | The 6 sessions' `startedAt` are **0.2 / 3.0 / 15.0 / 60.0 / 120.0 / 300.0 days ago** — one per bounded band (today / 7d / 30d / 90d / 180d / year-all) | PASS |
| 4 | Each session carries the requested per-session duration | All 6 sessions `ZDURATIONSECONDS = 600` (the default) | PASS |
| 5 | Window totals are non-zero and strictly increasing (so the dashboard renders distinct data per window) | Derived from the seeded `startedAt` spread: today=600s, 7d=1200s, 30d=1800s, 90d=2400s, 180d=3000s, all=3600s — non-zero, strictly increasing | PASS |
| 6 | The per-book `ReadingStats` aggregate is refreshed (Library-list sort parity) | `ZREADINGSTATS`: `ZTOTALREADINGSECONDS = 3600` (6×600), `ZSESSIONCOUNT = 6` for the seeded key | PASS |
| 7 | Today band is midnight-safe + no future `lastReadAt` | Today session anchored 0.2 days (~5h) ago = midpoint of elapsed day, inside `[startOfDay, now)`; clamp keeps every `endedAt ≤ now` (covered by unit test `…todayBandSurvivesMidnightEdge` at 00:10) | PASS |

## Commands run

```bash
SIM=61149F0E-DC18-4BE2-BB37-52659F1F4F62
APP="build/verify-1138/Build/Products/Debug-iphonesimulator/vreader.app"

# Clean DEBUG build into a fresh derivedDataPath (Bug #259: incremental
# rebuilds drop the injected vreader-debug URL scheme).
DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer xcodebuild build \
  -project vreader.xcodeproj -scheme vreader -configuration Debug \
  -destination "platform=iOS Simulator,id=$SIM" -derivedDataPath build/verify-1138

xcrun simctl install "$SIM" "$APP"
xcrun simctl launch "$SIM" com.vreader.app

# Reproduce the bug: seed a book, confirm no sessions.
xcrun simctl openurl "$SIM" "vreader-debug://reset"
xcrun simctl openurl "$SIM" "vreader-debug://seed?fixture=war-and-peace"

STORE="$(xcrun simctl get_app_container "$SIM" com.vreader.app data)/Library/Application Support/default.store"
sqlite3 "$STORE" "SELECT ZFINGERPRINTKEY, ZTITLE FROM ZBOOK;"          # 1 book
sqlite3 "$STORE" "SELECT COUNT(*) FROM ZREADINGSESSION;"               # 0  (bug)

# Apply the fix: seed sessions for the real war-and-peace key.
KEY="txt:bd8285a80f01df96dedd20a02178043afb85c0b499127e300baf57b7f1ed7508:1705"
ENCKEY=$(python3 -c "import urllib.parse,sys;print(urllib.parse.quote(sys.argv[1],safe=''))" "$KEY")
xcrun simctl openurl "$SIM" "vreader-debug://seed-sessions?book=$ENCKEY"

# Confirm 6 sessions across the bands + the stats refresh.
sqlite3 "$STORE" "SELECT COUNT(*) FROM ZREADINGSESSION;"              # 6
sqlite3 "$STORE" "SELECT ZBOOKFINGERPRINTKEY, ZDURATIONSECONDS,
  ROUND((strftime('%s','now') - (ZSTARTEDAT + 978307200))/86400.0,1)
  FROM ZREADINGSESSION ORDER BY ZSTARTEDAT DESC;"                     # 0.2/3/15/60/120/300 days, 600s each
sqlite3 "$STORE" "SELECT ZTOTALREADINGSECONDS, ZSESSIONCOUNT FROM ZREADINGSTATS;"  # 3600, 6
```

## Observations

- The OSLog `[DebugBridge]` lines did not surface via `log show`/`log stream`
  on this simulator (a known iOS-26 sim log-persistence quirk for `.info`),
  so verification was done by inspecting the SwiftData store (`default.store`)
  directly — the same posture the Bug #264 row used. This is actually the
  strongest possible evidence for this bug, since the bug lived precisely at
  the persistence boundary ("no mechanism seeds `ReadingSession`").
- The today band landed at 0.2 days (~5h) ago because the command ran around
  mid-morning local — the midpoint-of-elapsed-day anchor (Codex round-1 fix)
  scales with the time of day, which is why it's always inside the calendar
  day rather than a fixed `now − 1h` that would slip past midnight.
- The dashboard's window totals are computed from these `ReadingSession` rows
  by `ReadingStatsAggregator` (verified exhaustively + deterministically by the
  4 new `PersistenceActorStatsReadTests` seam tests, which drive the SAME
  aggregator over the SAME container and assert the exact 600/1200/.../3600
  ladder + the strictly-increasing invariant). The on-device store inspection
  confirms the inputs to that math are correct.
