# Bug #265 — AZW3/MOBI reading position not saved/restored on reopen

**DEVICE VERIFICATION FAILED.** The merged fix (v3.39.13, `FoliatePositionRestoreController`
wired into `FoliateBilingualContainerView`) does NOT restore the reading position
on reopen of an AZW3 book. The reader resumes at the START — reproducing the
user's exact reported symptom ("azw3 cannot resume the process after reopen it").
The bug is **NOT fixed**; it must be reopened for rework.

This was discovered because `DebugFixtureCatalog` DOES ship a real AZW3 fixture
(`mini-azw3`, line 48 — the summary's "no AZW3 fixture" was outdated), and the
`seek?fraction=` command (Bug #267) drives the live Foliate path, so the
end-to-end repro IS reachable CU-free.

## Observations

- **The failure reproduces the user's symptom exactly**: open AZW3 → read on → leave → reopen → resumes at the start.
- **The teardown genuinely happened** (step 3 snapshot shows `txt | '0'` — war-and-peace opened, so the AZW3 `FoliateBilingualContainerView.onDisappear` flush ran).
- **`fraction`-based seek works** (open `/6/2!` → seek 0.6 → `/6/10!`); the reopened reader's position is byte-identical to the start CFI (`/6/2!…:29`), so the restore path produced **no seek at all** (not a wrong seek).
- **`open?bookId=…&position=<CFI>` also fails to relocate the AZW3 reader**: the log shows `open: posted navigate-to-locator seek for resolved position`, but the snapshot stays at `/6/2!`. (This CFI-seek fires ~0.4s after open — possibly before the AZW3 render-complete first-relocate — so it is NOT a clean isolation of CFI-seek; treat as suggestive, not conclusive.)
- **The persistence subsystem is sound**: `FoliatePositionPersistenceIntegrationTests` (added this session) drives `FoliatePositionRestoreController → ReaderPositionService → real in-memory PersistenceActor` and the save→reopen→restore round-trip passes. So the gap is NOT the controller/store; it is the **live restore application** — either (a) the save never reaches the store on the live path (the `.readerPositionDidChange`→`handlePositionChange`→`scheduleSave` chain or the save gate), or (b) the restore seek (`triggerPositionRestoreIfNeeded` → `.foliateRequestSeekTarget` with the saved CFI) is posted but does not relocate the Foliate reader.

## Next diagnostic step (for the rework)

The components (`FoliateBilingualContainerView+Position.swift`, `FoliatePositionRestoreController.swift`, `ReaderPositionService.swift`) have **no logging**, so the live save/restore path is opaque CU-free. Add temporary `Logger` calls to: (1) `handlePositionChange` (did a gated relocate arrive?), (2) `flush`/`saveNow` (did the save fire? with what locator?), (3) `loadRestoreTarget` (did it return a target on reopen?), (4) the `.foliateRequestSeekTarget` observer in `FoliateSpikeView` (did the seek apply?). Rebuild, re-run the repro above, and the empty-vs-populated log will localize save-side vs restore-side. Hypothesis worth testing: since `goToFraction` works but the CFI `goTo` may not relocate at render-complete time, restoring via the saved **fraction** (`.foliateRequestSeekFraction`, the proven channel) instead of the CFI target may be the fix.
