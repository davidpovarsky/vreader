# Bug #265 — AZW3/MOBI reading position not saved/restored on reopen

**FIXED + device-verified** (after the first fix attempt failed device verification
and was reworked). The reader now resumes at the saved position on reopen.

`commit_sha` above is the pre-rework `main`; the rework lands in the PR that
references this file (its merge commit on `main` is the authoritative SHA).

## Arc: first fix FAILED → diagnosed → reworked → PASS

1. **First fix (v3.39.13)** wired `FoliatePositionRestoreController` into the live
   Foliate path. Device verification on the bundled `mini-azw3` fixture (CU-free,
   via `seek?fraction` + DebugBridge) showed reopen still resuming at the START.
2. **Diagnosis** — added permanent OSLog to the live wiring (it had none, which is
   why the first fix shipped broken). The log localized it precisely:
   - SAVE works: `flush: saving cfi=/6/10! progression=0.616`.
   - Restore-target load works: on reopen `loadRestoreTarget: saved cfi=/6/10! → target=epubcfi(/6/10!…)`.
   - But the CFI restore-seek (`.foliateRequestSeekTarget` → `goTo(filepos-CFI)`) NEVER relocated the reader — subsequent relocates stayed at `/6/2!`. `goToFraction` (the bottom-scrubber channel) DOES work.
3. **Rework** — restore via the saved `fraction` over `.foliateRequestSeekFraction`
   (the proven channel), with the CFI as fallback; re-assert across a short window
   (4 × 700ms) so the cross-section seek lands once foliate-js finishes paginating;
   short-circuit to immediate gate-open when there is nothing to restore.

## Observations

- The first fix's failure was invisible to unit tests (the controller's persistence
  round-trip was sound — proven by `FoliatePositionPersistenceIntegrationTests`).
  Only end-to-end device verification + instrumented logging exposed that the live
  CFI restore-seek doesn't relocate the Foliate reader. Strong argument for the
  close-gate device-verification requirement.
- Root cause: AZW3/MOBI use filepos-anchored CFIs that foliate-js `goTo` can't
  resolve reliably; `goToFraction` (whole-book progress) works. Restore now uses it.
- Settles the #267 / GH #1157 concern too: `seek?fraction` reaches a distinguishable
  position (the fixture has multiple sections), so the restore IS device-verifiable
  CU-free now.
