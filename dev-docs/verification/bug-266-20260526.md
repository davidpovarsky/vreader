# Bug #266 — Bilingual translation misaligned to the wrong paragraph

**Verified via the close-gate verification-exception** (deterministic high-fidelity
tests at the real pairing boundary). `commit_sha` is the most recent `main` at
verification time; the #266 fix shipped earlier (v3.39.14).

## Why the exception path

The bug's symptom is a **wrong** pairing (paragraph N's translation injected under
paragraph N+2). The fix's guarantee is structural: `BilingualPairing.translationsByBid`
pairs blocks↔segments **only** on an exact 1:1 count match; ANY mismatch returns an
empty map → the renderer paints source-only. A wrong (partial / drifted) pairing is
therefore **impossible to produce** — the production code path either pairs 1:1 or
shows source-only.

The intended device verification (render a nested-block EPUB + run AI translation +
observe correct alignment) is **CU-free-blocked**: it needs a nested-block EPUB
fixture (verification debt tracked in #268) AND the AI translation path, which is not
CU-free-drivable (same class as #267, #243's AI-harness gap). A device test would
only re-confirm the structural impossibility the unit tests already prove at the
exact production boundary — so per AGENTS.md, the high-fidelity tests satisfy the
close gate.

## Observations

- The fix converts the failure from "wrong translation under a paragraph" (misleading)
  to "no translation for that paragraph" (source-only) — strictly safer. The structural
  1:1 guarantee makes the original symptom unreproducible.
- **Residual (tracked in #268, NOT #266)**: nested `<pre>`-blank-lines + mixed-content
  `<blockquote>` chapters still fall back to source-only more than necessary (the two
  unit models diverge). That is a completeness gap, not a wrong-pairing gap — #266's
  symptom (wrong paragraph) is fully eliminated.
