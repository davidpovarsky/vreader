# Bug #302 — EPUB (Readium) tapping a highlight cannot edit it — Gate-5b close-gate verification

Readium-default-flip regression (sibling of #299): tapping an existing highlight in the
default Readium EPUB engine never opened the edit popover. The gap was total — no producer
(highlights are Readium `Decoration`s created `isActive:false`; no decoration-activation
handler) AND no consumer (`ReadiumEPUBHost.coreBody` never attached the unified popover
presenter). The fix wired both ends: the adapter's `attach` registers
`observeDecorationInteractions(inGroup: "highlights")` → maps the decoration id back to the
highlight UUID via the pure `tapEvent` helper → posts `.readerHighlightTapped`, and
`ReadiumEPUBHost+Body` attaches `.unifiedHighlightPopoverPresenterIfAvailable(...)` (the
designed feature #64 popover). It was then REOPENED for a re-arm gap (tap worked ONCE but a
second same-highlight tap was a no-op) and re-fixed by resetting the popover view-model on
interactive (swipe-down) dismissal so `.onChange(of: viewModel.presented)` re-fires.

Verified CU-free by re-running the shipped Swift Testing suites on merged main (v3.59.0)
as part of a targeted re-run of the shipped suites (35 via -only-testing) that returned `** TEST SUCCEEDED ** (35 shipped suites, 0 failures)`. This is
a **VERIFICATION-EXCEPTION** close — see Observations for the device-irreproducibility
reason and why the cited suites drive the same production code path at the real subsystem
boundary.

## Observations

**VERIFICATION-EXCEPTION close.** Tapping a Readium highlight to edit it is a WKWebView
decoration tap that is **not synthesizable CU-free** — the decoration activation fires inside
the navigator's web content and (Bug #316) reports a `.zero` sourceRect, so the on-screen hit
target is at an unknown position that neither idb HID gestures nor the DebugBridge can target
deterministically. The original single-tap fix WAS device-verified on 2026-06-02 (Readium
EPUB → create highlight → tap → unified edit popover opened), but the REOPENED re-arm path
(second same-highlight tap) is the hard-CU-free piece, which is why the row carried
`awaiting-device-verification` after the re-arm fix.

The three shipped suites drive the real edit-popover wiring at the production subsystem
boundaries rather than through stubs:

- `ReadiumDecorationHighlightAdapterTests.attach_registersHighlightsTapObserverOnce` exercises
  the **producer** — the actual `observeDecorationInteractions(inGroup:)` registration that
  was entirely missing pre-fix.
- `ReadiumDecorationTapEventTests` drives the **decoration-tap-event mapping** — the pure
  `tapEvent` helper that converts a Readium decoration id + rect/point into the
  `.readerHighlightTapped` payload, including the id-validation and `.zero`-degrade edges.
- `HighlightPopoverViewModelTests.handleTap_afterDismiss_reArmsSameHighlight` drives the
  **consumer re-arm seam** — the exact `.onChange(of: viewModel.presented)` reset that the
  REOPENED second-tap regression was about; it fails on the pre-fix code by construction.

These cover producer + event-mapping + consumer-re-arm — i.e. both ends of the previously
total gap plus the reopened regression — at the same boundaries the production failure hits.
`status_target: FIXED`.
