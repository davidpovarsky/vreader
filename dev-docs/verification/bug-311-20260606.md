# Bug #311 — Re-translate chapter progress fabricated/coarse (pins at 50%) — Gate-5b verification

The "Re-translate chapter" `ReTranslateProgress` sheet (spinner + bar + % + ETA) shows
its progress bar — but pre-fix the bar pinned at 0.5 for the entire (longest) translate
phase because `ChapterReTranslateViewModel` stepped `progress` 0.0 → 0.25 → 0.5 → 1.0 and
the translate was a single opaque `await runner.translateForRetranslate(...)` with no
per-chunk callback, so it read as "stuck." The fix threads a real per-chunk
`onChunkProgress: (@Sendable (Int,Int)->Void)?` callback from `ChapterTranslationService`
through `ChapterReTranslating` into the VM, mapping the real N-of-M committed-chunk count
into the 0.5→0.95 band (1.0 on flow completion); the ETA is the design's own
`(1-progress)*18s` heuristic, now driven by real progress (no UI change, Rule-51 clear).

Verified CU-free by re-running the shipped Swift Testing suite on merged main (v3.59.0).
`ChapterReTranslateViewModelTests` exercises the fix at the REAL
`ChapterReTranslateViewModel` boundary (real VM, real `translateProgress` mapping
function, real run-generation/`.running` guard) against an injected `ChapterReTranslating`
runner mock that fires real per-chunk ticks — the appropriate end-to-end substitute for
this provider-call-driven progress defect (see Observations for the verification
exception).
