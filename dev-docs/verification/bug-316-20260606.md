# Bug #316 — Readium EPUB highlight popover opens as the barren bottom-sheet fallback, not the designed anchored card — Gate-5b verification

Readium EPUB highlight taps presented the unified popover as the full-width
bottom-sheet fallback (tall empty note body for no-note highlights) instead of
the Feature #64 canonical compact card anchored to the tapped word. Root cause:
`ReadiumDecorationHighlightAdapter.attach` posted `.readerHighlightTapped` with
`sourceRect: event.rect`, but Readium's `observeDecorationInteractions`
`event.rect` is nil for these decorations → `tapEvent` degraded to
`sourceRect: .zero`, and `HighlightPopoverModifierBody` falls back to the sheet
form for `.zero`. The fix is two-part: (1) `tapEvent` now also reads
`OnDecorationActivatedEvent.point` and derives a 1×1 anchor at the tap point when
`rect` is nil (non-zero `sourceRect`); (2) `ReadiumEPUBHost+Body` supplies the
navigator's view as the popover host so `resolvedForm` keeps a non-zero rect at
`.card` instead of degrading it back to `.sheet`.

This is a **VERIFICATION-EXCEPTION** close (see Observations): the symptom is a
Readium-rendered WKWebView decoration tap whose target is not in the AX tree and
not synthesizable on this harness. Verified CU-free by re-running the shipped
Swift Testing suite on merged main (v3.59.0) — it drives the SAME point-anchor →
card resolution path the production tap hits, at the real `ReadiumDecoration` /
`HighlightPopoverPresenter` boundary.

## Observations

**VERIFICATION-EXCEPTION.** The symptom — tapping a no-note Readium highlight
opens the popover as the barren full-width bottom-sheet (tall empty note body)
instead of the compact anchored card — is produced by a tap on a Readium
decoration painted **inside the navigator's WKWebView content**. That hit target
is not exposed in the accessibility tree, so it cannot be addressed by
DebugBridge, XCUITest-synthesized accessibility taps, or a CU/idb HID tap with
any reliability on this harness (the same Readium-decoration-tap irreproducibility
that left Bug #302 `awaiting-device-verification`). There is therefore no
device-reproduction path available CU-free.

The shipped `ReadiumDecorationTapEventTests` suite stands in as the high-fidelity
integration check: it exercises the production seam end-to-end through the real
subsystem boundaries — the precise `rect`-nil → tap-point → non-zero 1×1 anchor
mapping in `ReadiumDecorationHighlightAdapter.tapEvent`, then the precise
sheet-vs-card decision in `HighlightPopoverPresenter.resolvedForm` (fed by
`HighlightPopoverPresenter.content`). Before the fix that path yielded `.zero` →
`.sheet`; after the fix it yields a non-zero point-anchor + (with the navigator
host view now supplied by `ReadiumEPUBHost+Body`) `.card`. The two-half test
(`hasHostView: true → .card`, `false → .sheet`) specifically pins the Gate-4
Codex round-1 finding that the non-zero rect alone was insufficient without the
host view. The only hop not in the test is the Readium-internal
decoration-activation → `point` delivery — exactly the portion the exception
covers. The targeted re-run of the shipped suites (35 via -only-testing) on merged main returned
`** TEST SUCCEEDED ** (35 shipped suites, 0 failures)`, so the regression guard ships green on v3.59.0.
