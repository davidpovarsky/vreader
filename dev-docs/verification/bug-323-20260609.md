## Observations

- The reopened freeze had a DIFFERENT mechanism than the v3.59.13 fix (composer-vs-
  save): per-token main-thread re-render churn in `consumeStream`. The fix coalesces
  streamed deltas (`StreamCoalescer`) so the `@Observable` transcript re-publishes at
  a capped rate instead of per token.
- The mock provider's chat reply is short, so the *freeze itself* (which needs a long
  real-provider reply to saturate the main thread) is not mock-reproducible — that
  aspect is proven by the deterministic `StreamCoalescerTests` (per-token publishes
  collapse to a bounded count). The device smoke confirms the reported FUNCTIONAL
  symptom ("2nd message stuck, no reply") is gone: both messages reply and the
  composer stays responsive.
- Codex audit caught a real regression risk (drain skipped on a thrown stream) —
  fixed + tested before this verification.
