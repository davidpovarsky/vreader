# Bug #333 — long-chapter re-translate timeout mislabeled "You appear to be offline"

**Verification-exception**: a real provider request timeout (`URLError.timedOut`)
cannot be reproduced on a device without a fault-injection harness. The fix is
verified by high-fidelity integration tests that drive the SAME code paths the
production failure hits (the real `translate()` boundary + the real ViewModel
error formatter), not casual stubs.

## Acceptance criteria

| Criterion | Test (drives the real path) | Result |
|---|---|---|
| A request timeout maps to a distinct timeout error, NOT `.offline` | `ChapterTranslationServiceTests.timedOutURLError_throwsTimedOutNotOffline` — injects `URLError(.timedOut)` through `translate()`; `mapTransportError` runs for real → `.timedOut` | ✅ |
| Host-unreachable maps to `.providerFailed`, NOT `.offline` | `...cannotConnectToHostURLError_throwsProviderFailedNotOffline` | ✅ |
| Genuine connectivity still maps to `.offline` | `offlineURLError_throwsOfflineNotProviderFailed` (pre-existing, still green) | ✅ |
| The UI shows a "timed out" message, not "You appear to be offline" | `ChapterReTranslateViewModelTests.submit_timeout_showsTimedOutNotOffline` — real `errorMessage` renders the timeout copy, asserts NOT "offline" | ✅ |
| No regression in the bilingual prefetch path | `BilingualReadingViewModelRetryUnitTests` green (a `.timedOut` becomes `.failed`/retry, not silent-offline-fallback) | ✅ |
