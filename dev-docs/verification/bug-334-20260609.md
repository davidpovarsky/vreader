# Bug #334 — Foliate (AZW3) bilingual translation never injects (loading shimmer stuck)

CU-free device verification of the fix via the Feature #77 `bilingual` DebugBridge
harness on a real AZW3 (`mini-azw3` fixture, Foliate engine).

## Acceptance criteria

|---|---|---|---|
| Bilingual enable renders the loading shimmer | `loading:9` | `loading:4` (matches block count) | ✅ |

```bash

xcrun simctl openurl "$SIM" "vreader-debug://open?bookId=<azw3-key>"
xcrun simctl openurl "$SIM" "vreader-debug://seek?fraction=0.45"
# enable bilingual with a FRESH (uncached) target language:
xcrun simctl openurl "$SIM" "vreader-debug://bilingual?action=enable&lang=Norwegian"
# eval the foliate-frame DOM (loading / decoration / mock counts) during + after:
xcrun simctl openurl "$SIM" "vreader-debug://eval?bridge=azw3&js=<base64 frame counter>"
```

The frame counter walks `foliate-view.renderer.getContents()[].doc` and counts
`.vreader-bilingual-loading`, `[data-vreader-bid]`, and `MOCK译` occurrences.

## Observations

- Root cause was `bilingualSectionText` returning whole-body `textContent` (no
  paragraph boundaries) so the translate's segment count collapsed to ~1 while the
  enumerate counted N leaf blocks → the `blocks.count == segments.count` pairing
  guard failed → empty inject map → stuck shimmer.
- The fix shares a single `bilingualLeafBlockElements` selector between the
  enumerate and the section-text extraction, so the two counts can't diverge.
- Verified across a single-paragraph page (`blocks:1 → mock:1`) and a multi-paragraph
  body section (`blocks:4 → mock:4`). The fresh-language requirement matters: cached
  translations persist by fingerprint, so each verification used an unused target
  language to force a real translate.
- The DURING window cleanly shows `loading:N, mock:0` (shimmer up) and the POST shows
  `loading:0, mock:N` (replaced) — the transient cycle the original bug never reached.
