# Bug #352 — Bilingual ON breaks EPUB reading-position restore

## Summary

With bilingual interlinear ON, a Readium/paged EPUB reopened at an
EARLIER position within the right chapter — observed by the user as a
deep-in-chapter position reopening at the chapter start (translations
rendered, so it was pure position drift, not translation loss).

Mechanism: the navigator restores to the saved fraction-based progression
against the SOURCE-ONLY spine and fixes the page; the bilingual inject
then grows content under that page, so the held page shows earlier
content. Fix: re-assert the saved locator after the restore-landing
spine's first inject (one-shot href-guarded `BilingualRestoreReassertGate`),
with a 0.25s WKWebView-reflow settle so the navigate re-resolves against
the final injected height.

## Observations

- The decisive fix detail was device-found: an IMMEDIATE re-assert landed
  ~1 page short because the inject's WKWebView reflow is asynchronous —
  `navigate(to: progression)` resolved against the pre-reflow height. A
  0.25s settle before the navigate makes it land exactly.
- The re-assert is one-shot and href-guarded: it fires only on the
  restore-landing spine's first inject, re-checks the live href at fire
  time, is held in a cancellable task cancelled by all 6 disable/reposition
  paths, and disarms if the user leaves the restore chapter before the
  inject (Codex rounds 1–5, all resolved/accepted).

## Known limitation

The re-assert uses the saved fraction-based progression, exact to first
order; a residual from non-uniform interlinear injection is theoretically
possible but was below observable in device testing once the reflow-settle
landed (exact restore). Anchor-exact (CFI/block-id) restore is a tracked
follow-up (Codex Medium-1, accepted).
