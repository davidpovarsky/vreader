## Observations

- **CU is structurally down for the third consecutive verify-cron
  session.** The host is virtual-display-only (no real monitor
  attached). `mcp__computer-use__screenshot` returns
  `CU display unavailable`. The `sim-drive-fallback` toolkit's
  CGEventPost taps also do not translate into iOS touches on this host
  â€” `osascript` AX queries against the Simulator window return only
  the window-chrome elements (close / minimize / Home / Save Screen /
  Rotate / status text), and the iPhone display group has zero
  child accessibility elements (the iOS-side accessibility tree is not
  exported through the macOS AX layer). Confirmed by tapping a known
  point (Settings gear at the documented top-left) and observing no
  state change in the resulting screenshot. This is the
  **same condition** documented in `feature-65-20260519.md` and
  `feature-69-20260519.md`.
- **XCUITest is the working CU-free path.** XCUITest synthesizes
  touches via the iOS accessibility API directly through the simctl
  device â€” it does NOT depend on the macOS-side AX bridge that's
  broken on this host. The test passes deterministically in 50
  seconds.
- **The `--enable-ai` flag was load-bearing.** Without it, the AI
  Chat toolbar button is hidden by `AIReaderAvailability.isAvailable`
  (the three production gates: feature flag + API key + consent),
  none of which an XCUITest sandbox can satisfy. Bug #237 / GH #975's
  fix (`AITestOverride.forceAvailable` seam, consumed from
  `--enable-ai` in `VReaderApp`) is what unblocks reaching the
  surface. Without that fix, this verification path would not be
  possible.
- **`.empty` seed is enough.** Bug #93 is a Library-level chat fix,
  not reader-context-dependent. An empty library lets the test focus
  on the toolbar â†’ sheet â†’ dismiss â†’ reopen lifecycle without
  book-grid noise.
- **The user-message-append happens BEFORE the AI request.**
  `AIChatViewModel.sendMessage(_:)` line 83 appends the user message
  to `messages` synchronously, then line 91 fires the network call.
  Even though the XCUITest environment has no valid AI provider
  configured (no `--enable-ai` provider seed in the `--uitesting`
  in-memory store), the user-message-survives-dismiss assertion still
  works because the bubble renders from the local append â€” no
  successful AI roundtrip required.
- **Why this is a "verification-exception" close, not a
  "verification-blocked" close.** The bug's failure mode CAN be
  reproduced inside an XCUITest at the real subsystem boundary
  (`LibraryView` SwiftUI `.sheet` content closure â‡’ `AIChatViewModel`
  lifecycle). The integration test exercises the exact production code

  close-gate language: "a deterministic high-fidelity integration
