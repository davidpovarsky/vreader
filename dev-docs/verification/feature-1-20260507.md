## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 14 passing tests give **strong Gate-4-class evidence** for the
data + ViewModel + haptic layer:

- **`BookmarkListViewModel` state machine** — load / add / remove /
  toggle / rename / edge cases. The ViewModel is the boundary
  between SwiftUI's BookmarkListView and the SwiftData-backed
  `PersistenceActor`. Tests cover all the public mutators.
- **`PersistenceActor` bookmark CRUD** — 8 tests against an in-
  memory `ModelContainer` exercise the full insert / fetch /
  update / delete cycle. This is the same code path the
  production app uses, just with an in-memory backing store.
- **Bug #42 rename regression covered** — the rename mutator path
  has its own test in the mutation suite. If the regression
  recurred, the test would fail.
- **Haptic feedback** — `BookmarkFeedback` fires
  `UIImpactFeedbackGenerator(.light)` on add. The test pins the
  feedback intensity (light, not medium / heavy / rigid).
  Misclassified intensity would feel wrong in the user's hand
  but pass a "function got called" check.

What's deferred: the actual gesture path (swipe row → Delete
button → tap → row removes; long-press → context menu → Rename
→ alert with text field → Done). Pure UIKit/SwiftUI plumbing.

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Bookmark row count was hand-counted as 14.** xcodebuild's "8
  tests in 1 suite" final line undercounts because of the runner-
  bucket split (Swift Testing tests are reported by `swift-testing`
  not the `XCTestObservationCenter`). Same noise pattern noted in
  feature #31's evidence.
- **PersistenceActor uses an in-memory ModelContainer** —
  `ModelConfiguration(isStoredInMemoryOnly: true)` so each test
  starts with a fresh store. This is the project-standard pattern
  per `.claude/rules/10-tdd.md`.
- **Toggle is the user-facing primitive** —
  `toggleBookmark(at: locator)` is the action wired to the
  reader's "add bookmark" button. It adds when absent, removes
  when present. Tested directly.
- **Bug #42 was the rename bug** — the original BookmarkListView
  had no rename action; bug #42 added it via context menu. The
  rename path is now tested.
- **Tap-to-scroll is cross-feature** — tapping a bookmark in the
  list scrolls the reader to that locator. The Locator round-trip
  is exercised under each format's locator suites (TXT / MD / PDF
  / EPUB), already verified in #11 / #17 / #12 / #38 evidence.
