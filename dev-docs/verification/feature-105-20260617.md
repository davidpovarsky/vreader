# Feature #105 — Android Spike B: CJK WebView reader benchmark — verdict

Instrumentation-first (no UI automation — the iOS verify stack does not transfer)
de-risking of **ADR-0001 Risk 2**: can Readium-Kotlin 3.3.0 render a real
1000+-spine CJK novel with acceptable scroll smoothness, bounded memory, stable
rendering, and working position/selection restore — **measured, not assumed** —
before committing to the Android WebView-engine plan? All legs measured on the
android-35 arm64 emulator across WI-1..WI-3; this file is WI-4 (the verdict).

## Metric baselines (250-chapter sweep, real 1042-ch CJK corpus)

**Scroll smoothness** — 28,498 motion-bounded frames, **jank 0.23%** (rubric ≤5%),
p50/p90/p99 = 16.67ms (steady 60fps), max 50ms. Frame window bounded by Readium
locator stabilization (not a fixed timer). **PASS** — and well within the
`p90 ≤ iOS×1.5` bar (iOS runs the same book at ~16.67ms p90).

**Memory / eviction** — host process flat 170–210MB; the WebView **renderer**
process (measured separately via `dumpsys meminfo`, attributed by baseline-PID
diff) ramps to a **~1.1GB high-water by ch~90–130 then EVICTS down to a
580–870MB oscillation** for the second half — bounded, NOT monotonic; peak total
~1.29GB; **zero OOM**. **PASS** (eviction works). The ~1.1GB renderer high-water
is a low-RAM-device watch-item (see obligations).

**Renderer stability** — **zero** WebView/Chromium renderer crashes, zero
render-process-gone, zero FATAL over the sweep. **PASS**.

**Position restore (WI-3, synthetic controlled-offset CJK fixture)** —
chapter-level restore + Locator JSON round-trip are **FAITHFUL** (exact href +
progression; the save→JSON→restore path the backup relies on preserves all
fields); **selection round-trips exactly** (programmatic DOM selection surfaced by
`currentSelection()` with the exact CJK paragraph text). **Fragment-level
restore is APPROXIMATE** — the target paragraph restores on-screen in the top
third (~18% / ~2 paragraphs below the top), not pinned same-paragraph.

## Verdict + engine decision

**Readium-Kotlin 3.3.0 scroll mode is VIABLE as the Android v1 reader engine.**
The hardest, most-bug-bearing iOS surface (continuous-scroll CJK: #83/#85
windowing, the #329 eviction saga, #349/#352 restore) is handled by
Readium-Kotlin out of the box at 60fps with working eviction and zero crashes —
without re-implementing the iOS coordinator. **Decision: proceed with
Readium-Kotlin scroll for the Android reader; do NOT reopen the engine strategy.**

No FAIL on the engine-blocking legs (memory/renderer). The two imprecisions are
**recorded engine-hardening obligations** for Phase 3 "re-hardening" (per the
plan: "a scroll/anchor miss is a hardening obligation, not necessarily a strategy
reopen"), NOT blockers:

1. **Renderer memory high-water ~1.1GB** on the 1042-chapter CJK book. Bounded
   and oscillating (eviction works), no OOM on the emulator, but a watch-item for
   low-RAM devices — Phase 3 should validate on a 3–4GB device and consider a
   tighter resource-cache budget.
2. **Fragment-restore ~2-paragraph precision** on CJK (target lands ~18% below
   the viewport top, not same-paragraph-exact). The iOS #352 bar is exact;
   Android v1 will need position re-hardening for exact CJK restore. Chapter +
   JSON + selection fidelity are already exact, so resume-to-chapter and
   backup/restore round-trips are sound today.

## Minimally-automatable Android verification recipe

The iOS verify stack (rule 47 / `cron-prompts/verify.md` / tdd-guardian /
computer-use) does NOT transfer. The Android lane is **instrumentation-first**:

```bash
source /tmp/android-env.sh                      # JDK 17 + Android SDK on PATH
spikes/android-reader-bench/run-bench.sh        # WI-2 scroll/memory/stability (250ch)
SUITE=anchor spikes/android-reader-bench/run-bench.sh   # WI-3 anchor/selection probes
CHAPTERS=14 SCROLLS=2 spikes/android-reader-bench/run-bench.sh   # quick smoke
```

It builds + installs (`am instrument`), pushes the real corpus + the synthetic
`mini-cjk.epub`, runs the instrumentation suite, pulls `metrics.json`, and scans
logcat for renderer crashes — emitting a single `BENCH RESULT: PASS|FAIL` line.
**The cron CAN drive an Android emulator end-to-end** (the ADR's biggest open
unknown, resolved in WI-1). This is the seed of the Phase-2 Android verification
story.
