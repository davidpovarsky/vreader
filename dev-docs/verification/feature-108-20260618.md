# Feature #108 — source-bytes canonical fingerprint for converted-Kindle

Closes the #354 obligation: iOS persists the SOURCE-bytes canonical identity
(`azw3:{sha256_of_source}:{bytes}`) for converted-Kindle books as the
cross-platform identity, carried through backup.

## Verification class

Non-UI, **latent** feature (the cross-platform divergence it prevents cannot
manifest without an Android client, which does not exist yet). Verified through
**high-fidelity integration tests at real subsystem boundaries** (the actual
`BookImporter` + libmobi converter + `PersistenceActor` + backup
collector/restorer + the SchemaV10 disk-backed migration — not stubs), plus an
on-device migration smoke confirming the additive SchemaV10 column lands without
data loss.

## Observations

- The Gate-4 audit caught that WI-2 could not ship without WI-3 (a backup taken
  between them silently loses the field) — so WI-3 was folded in, making this the
  complete feature in one PR. The audit also found 4 other issues (off-main hash,
  hash-before-convert, dedupe backfill, Phase-1 existing-row backfill), all fixed.
- Design choice that keeps risk low: `fingerprintKey` (converted-EPUB) stays the
  local primary, so rendering + the blob-identity invariant are untouched; the
  source key is an additive cross-platform identity. Existing converted books
  grandfather to nil (their source bytes were discarded) and can backfill on
  re-import.
- The on-device launch over an existing library confirms the additive SchemaV10
  lightweight migration runs without data loss (unlike #109's shape-identical V10,
  this one genuinely fires).
