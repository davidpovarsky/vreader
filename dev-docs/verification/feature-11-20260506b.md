## Summary

Round 2 of feature #11 (EPUB highlighting). The first round (`feature-11-20260506.md`) verified persistence + count-exposure slices via direct ZHIGHLIGHT INSERT against `default.store`. Render-layer slices (selection-gesture → JS-bridge → CSS-Highlight-API) were DEFERRED in round 1 because computer-use (display) was unavailable.

Bug #126 (`vreader-debug://eval` for EPUB) shipped in v3.14.13 as PR #307 + #308. Round 2 uses the new eval capability to verify the **JS-side bridge readiness** — what part of the render-layer can be observed without UI gestures.

**Verified** (this round):
- vreader's highlight bridge global (`window.__vreader_createHighlight`) is loaded and is a function on first DOM-ready.
- WKWebView `selectionChanged` message handler is registered.
- CSS Highlight API is available in this WebView (`'highlights' in CSS` → true).
- `document.readyState === 'complete'` after EPUB load.
- DOM content is correct (`document.querySelectorAll('p').length === 3` for mini-epub3 chapter 1).
- **Programmatic CSS highlight insertion round-trips**: `CSS.highlights.set('vr-test', new Highlight(range))` returns true on `CSS.highlights.has('vr-test')` and the key appears in `CSS.highlights.keys()`. Range text matches the expected EPUB content (`"This is th"`).

**Still deferred** (this round): the user-flow `selection gesture → menu tap "Highlight" → __vreader_createHighlight call → CSS.highlights.set on the selected Range`. That requires UI driving (text-selection gesture + edit-menu interaction) which DebugBridge doesn't synthesize. Computer-use display would be needed; not available in the current loop session.

## Observations

- **The bridge-readiness slice is now headlessly verifiable**. Before bug #126 shipped, every `vreader-debug://eval` returned `evalUnsupported(format: "epub")`, so the only way to know if `__vreader_createHighlight` was loaded was to drive the selection gesture and watch what happened. After bug #126, JS state in the EPUB WebView is observable without UI gestures — a real expansion of the verification surface.
- **The user-flow gap remains real**. Selection gestures require synthesized touches. Even with eval available, we can't verify that "user taps Highlight in the menu → bridge correctly maps the selection range to a CFI → CSS.highlights.set is called with the correct Range" without running through the UIKit edit-menu chain.
- **CSS Highlight API works correctly in this WebView** even though the production code uses `__vreader_createHighlight` (which itself wraps CFI/Range arithmetic). Direct `CSS.highlights.set` round-trips fine — confirming the rendering primitive is available, only the gesture wiring is unobservable.
- Per SCHEMA partial semantics: feature #11 status stays DONE pending the gesture-driven slice. The bridge-readiness slice is now PASS, narrowing what's deferred from "all render-layer" to "gesture-driven flow only".
