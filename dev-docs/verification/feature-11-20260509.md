## Summary

Round-3 verification of feature #11 (EPUB text highlighting and
note-taking) closed the long-deferred **gesture-driven user-flow slice**:
text-select via long-press → custom "Highlight" menu action → data
persistence → side panel listing.

The half that's still missing is **visual render**: the highlight
saves correctly to the data layer (visible in the Highlights tab as
"🟡 some") but the EPUB page does NOT paint a yellow background on
the highlighted range. Reproducible across themes (Sepia, Light) and
across book close+reopen. Filed as **bug #159 / GH #472** (EPUB/*
Medium).

Per verify-cron scope, this round files but does not fix. Feature #11
status stays DONE; VERIFIED is gated on bug #159 fix (CSS Highlight
API paint pipeline). Combined with round-1's persistence + count
slices, round-2's bridge-readiness probes, and round-3's gesture +
data-layer confirmation, the only remaining gap is the visual paint —
a well-scoped follow-up.

## Observations

- **Gesture path works end-to-end up to data persistence.** The
  long-press triggers the iOS selection state, the custom VReader
  menu (Highlight / Add Note / Copy) renders correctly alongside
  the standard system menu, the tap on Highlight clears the
  selection, and the side panel's Highlights tab correctly shows
  the new row.
- **Paint pipeline is the gap.** The bridge-readiness probes from
  round-2 (`__vreader_createHighlight` defined, `selectionChanged`
  handler registered, programmatic `CSS.highlights.set` round-trip
  with correct rangeText) all PASS. The data persistence proves
  that the gesture eventually calls into the persistence layer.
  But the WKWebView page does not visibly paint the yellow
  background on the highlighted range.
- **Root-cause hypotheses** (not yet code-read in detail):
  1. Missing `::highlight(...)` CSS rule injection — the
     `Highlight` is registered in `CSS.highlights` but no stylesheet
     defines a paint property for the named highlight, so the user
     agent renders nothing.
  2. JS path divergence — the `__vreader_createHighlight` call from
     the gesture menu uses a different path than the bridge-readiness
     probe, and that path constructs the Range incorrectly (e.g.,
     CFI-anchored range doesn't match live DOM).
  3. CSS theming conflict overriding background. Light theme also
     reproduces, so theme-conflict (3) is less likely.
- **Bonus regression checks (consistent with prior fixes)**:
  - Bug #158 picker gating: Reading Mode picker IS shown for EPUB
    (capability-gated correctly post-#158).
  - Bug #128 native-mode no-op: "Lorem ipsum" rendered verbatim in
    Native mode for mini-epub3 (rule does NOT apply per #128).
  - Bug #128 banner copy: side panel banner contains "currently
    EPUB, MD, AZW3 — not TXT or PDF".
- **Cross-format note**: Foliate (AZW3/MOBI) highlight overlay
  restoration is documented in feature row 11 as "deferred to
  WI-7". The Foliate path may exhibit a similar gap but uses a
  different render mechanism (Foliate-js handles its own highlight
  paint inside the iframe). Out of scope for this round.
