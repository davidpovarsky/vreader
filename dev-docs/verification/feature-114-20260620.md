# Feature #114 — Android backup & WebDAV restore UI — VERIFIED

All five designed Compose surfaces from the committed `vreader-backup-webdav.jsx` (#1767 design,
landed in #1771) implemented, wired to a `BackupViewModel` behind a UI-oriented `BackupService`
seam, and verified end-to-end via instrumented Compose tests on a real Android emulator. Per the
Gate-2 scope guard, the UI ships DEBUG-reachable only (no production wiring / fake data); the
real WebDAV+restore backend is a separate future feature behind `BackupService`.

## Observations

- The UI is rendered in VReader's own vocabulary (paper/dark tokens, Source-Serif titles,
  #8c2f2f / dark #d6885a accent, rounded-14 cards), recreated 1:1 from the design's `UI.light`/
  `UI.dark` token values — not a Material re-skin (rule 51).
- The five surfaces shipped across WIs `android/v0.5.3`→`v0.6.0` (foundation, then one surface
  per WI), each with its own Gate-4 Codex audit (some multi-round — the High in WI-3b caught the
  full-screen-vs-bottom-sheet chrome gap; the WI-2/WI-4 Mediums caught server-specific error/
  success copy).
- DEBUG-isolation (Gate-2 High): `BackupDebugActivity` (adb-launchable) + `PreviewBackupService`
  live in `src/debug`, so no fake servers/backups can reach a production build.

## Known limitations (separate future features)

The real **WebDAV client + restore-import pipeline + backup collector** (the `BackupService`
implementation) is out of this feature's scope — it ships separately and is the next backend
slice (the design note is explicit the backend is non-design-gated). A **production Settings
entry point** needs its own committed design (a `needs-design` issue when wanted).
