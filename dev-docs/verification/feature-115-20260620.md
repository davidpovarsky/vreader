# Feature #115 — Android PDF reader (base page-view) — VERIFIED

The fourth reader capability under the #110 Android driver (EPUB ✓ → TXT ✓ → MD ✓ → **PDF**),
implementing the committed design `vreader-pdf-reader.jsx` (#1766) — the canonical
continuous-scroll layout over `android.graphics.pdf.PdfRenderer` page bitmaps. Verified on a
real Android emulator.

## Observations

- The canonical continuous-scroll layout (the design's #1766 decision) renders `PdfRenderer`
  page bitmaps lazily per visible item on a neutral viewer backdrop; off-screen bitmaps are left
  for GC (synchronous `Bitmap.recycle()` at the composable boundary races Compose's draw and
  crashes — a deliberate, Codex-confirmed v1 decision).
- `PdfRenderer` lifecycle is handled safely: `renderPage` + `close` serialized through one
  `Mutex`, `Page.close()` in `finally`, the document closed off-main on the process scope.
- Resume mirrors the proven TXT pattern with `page` for `charOffsetUTF16` (legacy `Locator(page)`
  envelope → `ResumeResolver → Canonical`); clamped to a valid page.
- Real PDF fixtures don't exist (the documented real-books-first exception "no real PDF today"),
  so hand-built synthetic PDFs are used; large-file performance / encrypted-unlock / the paged
  toggle / the page-jump overlay stay unverified (designed follow-ons — see below).
- The new Swift-free; pure Android. (A stale `project.pbxproj` lesson does not apply here.)

## Known limitations (designed follow-ons, deferred on platform/API constraints)

The **paged-mode** Display toggle, the **page-jump** thumbnail/scrubber overlay, and the
**encrypted-unlock** password flow are designed (`vreader-pdf-reader.jsx`) but deferred: stock
`PdfRenderer` password support is API-35-only (minSdk 26), and paged/page-jump are navigation
enhancements over the canonical continuous-scroll read. Large-PDF (600-page) memory/scroll
performance is unverified until a real PDF fixture exists. AZW3 remains separately deferred
(libmobi).
