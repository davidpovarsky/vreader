## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 138 passing tests give **strong Gate-4-class evidence** for the
TOC slice and the MD subsystem that backs it:

- **`TOCBuilder.forMD`** — 25 tests cover the regex-based heading
  extraction path (H1-H6), fenced code block skipping, UTF-16
  offset computation, and edge cases (empty input, no headings,
  inline markdown in heading text).
- **Round-trip integration** — `MDOffsetRoundTripTests` exercises
  the offset → Locator → offset path that TOC entries depend on
  for "tap heading → scroll to position". If this doesn't round-
  trip, the TOC tap-to-scroll feature breaks silently.
- **Parser + types + renderer pinned** — TOC builds on top of the
  whole MD subsystem; the renderer and parser tests in this slice
  guard against regressions that would manifest as wrong heading
  offsets or missing entries.
- **Canonical hash + determinism** — same MD input produces same
  parsed output run-to-run; the TOC depends on this for stable
  scroll positions across app launches.
- **Bug #23 regression covered** — position-save guards prevent
  overwriting the user's last-read position during restore. If
  this regressed, "tap heading → scroll" would race against
  position restore.

What's deferred: the UI gesture path (open MD file → TOC list opens
→ tap heading → scroll lands at correct position) needs the
simulator with a display. Pure data-layer behavior is fully
exercised below.

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **The TOC slice (25 tests) matches the row count.** The whole
  feature is tightly scoped to the heading-extraction logic; the
  surrounding 113 MD-subsystem tests support TOC indirectly by
  pinning the parser/loader/renderer the TOC relies on.
- **Fenced code-block skipping is the load-bearing edge case.**
  Without it, a `# heading` inside a fenced code block would
  generate a false TOC entry. Tests cover this directly.
- **UTF-16 offsets matter because UITextView uses UTF-16
  indexing.** A heading offset miscomputed in UTF-8 would scroll
  to the wrong position for any document containing CJK or
  emoji before the heading. The tests pin UTF-16 alignment.
- **TOC works in both rendering modes** — native (UITextView with
  attributed string) and unified (TextKit 2 reflow). The unified-
  mode reflow tests guard the alternative path.
- **Front-matter metadata** — `MDMetadataExtractor` pulls title
  from YAML front-matter when present, falling back to the first
  H1. This affects how the TOC list's section header renders.
- **No live AI / network / UI dependencies** — feature #12 is a
  pure parsing feature, so the test slice gets close to full
  coverage without UI gesture; the only deferred items are the
  scroll-to-position visual confirmation and large-document
  performance.
