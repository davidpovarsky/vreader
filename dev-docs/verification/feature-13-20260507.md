## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 44 passing tests give **strong Gate-4-class evidence** for the
ViewModel + integration + availability + cache + configuration
layers:

- **`AIAssistantViewModel` state machine** — 11 tests cover the
  full lifecycle: idle → loading → complete | error |
  contextError | disabled | consentRequired. Plus per-action
  dispatch (summarize / translate / vocabulary / askQuestion).
  This is the canonical surface a SwiftUI panel binds to.
- **`AIReaderIntegration` end-to-end (with mock provider)** — 12
  tests drive the same path the production app drives in the
  reader: extract context, gate, call service, display in panel,
  handle errors, hide button when disabled, truncate over-long
  content. Crucially covers PDF + EPUB formats explicitly so the
  per-format `AIContextExtractor` divergence is pinned.
- **`AIReaderAvailability` gating** — 8 tests pin the entry-point
  visibility logic (flag + key + consent). Live-transition test
  ensures revoking consent during a session flips visibility — the
  bug class to watch is "AI button stays visible after consent
  revoke until app relaunch".
- **`AIRequest.cacheKey` determinism** — 7 tests pin the cache-key
  contract. Important load-bearing piece: AI cache misses cost
  real money, and the cache key must distinguish (user prompt,
  target language, context text, fingerprint) reliably. The "nil
  fingerprint uses general prefix" test pins the boundary
  between general (no-book) and book-specific cached responses.
- **`AIConfiguration` persistence** — 6 tests cover round-trip,
  default fallback on empty storage, default fallback on corrupt
  data (must not crash), and partial-config forward-compat.
- **Bug #92 regression cross-reference** — the encoding fix for
  non-UTF-8 TXT books is what made AI summarize show real
  content instead of garbled bytes. Covered in bug #92's own
  evidence; the `AIReaderIntegration` summarize tests would fail
  if the regression returned because they assert the AIService
  request payload contains real text.

What's deferred: the UI gesture path itself (Reader → AI button →
action → response in panel; live AI-toggle hides button). Needs
real display + a real or stubbed network round-trip; the in-process
tests use a mock provider.

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Two ViewModels for two AI surfaces.** `AIAssistantViewModel`
  drives the *reader-side* AI panel (per-book context, action
  types: summarize / translate / vocabulary / askQuestion).
  `AIChatViewModel` (verified separately in #15) drives the
  multi-turn *chat* surface (Library toolbar entry + per-book
  chat). Same `AIService` backend. Different ViewModels because
  the state machines diverge: assistant is one-shot
  request/response per action; chat is multi-turn history.
- **Per-format summarize is non-trivial.** `AIContextExtractor`
  has format-specific paths for PDF (text per page),
  EPUB (text per spine item), and TXT/MD (chunk slicing).
  `AIReaderIntegration` tests cover PDF + EPUB explicitly. The
  TXT/MD path is implicit in the "long content truncated" test
  (long-form text exercises the chunk slicer).
- **Live availability transition is critical.** If the user
  revokes consent in Settings while a reader is open, the AI
  button must disappear without a relaunch. The test
  `availableTransitionsAcrossConsentRevoke` pins this — easy to
  regress because it's a rare interactive path.
- **AIConfiguration tolerates corrupt data.** The corrupt-data
  test (`loadReturnsDefaultOnCorruptedData`) exists because the
  config is JSON-persisted via UserDefaults; if future schema
  drift produces unparseable data, the app must still launch.
  Without this guard, an AI-config schema bump could brick the
  reader.
- **Bug #92's fix is upstream of #13's user-facing claim.**
  Before bug #92, summarize on a GBK-encoded TXT (Chinese web
  novel) produced mojibake into the AI request, returning a
  meaningless summary. The fix detects encoding before
  extraction; #13's "Device verified" line in the tracker
  refers to that GBK summarize round-trip, which is now stable.
- **Cache key includes context text — a load-bearing choice.**
  Two summarize requests on the same book at the same chapter
  with the same prompt MUST hit the cache; but if the user
  edits the chapter (highlight changes the surface? unlikely,
  but theoretically), or if the extractor returns slightly
  different text after a future update, the cache must miss.
  The test `differentContextTextsProduceDifferentKeys` pins
  this contract.
- **`AIReaderAvailability` and `AIService` gates duplicate the
  same checks at different layers.** UI gating in
  `AIReaderAvailability` decides whether to show the button;
  `AIService` re-checks at request time. Belt-and-suspenders.
  Both layers are tested.
