## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 121 passing tests give **strong Gate-4-class evidence** across
the full chat stack:

- **`AIChatViewModel` end-to-end** — request building, streaming
  token accumulation, message-history mutation, cancel handling, and
  book-context propagation all exercised through the production
  ViewModel against a mocked URLSession (only the network wire is
  mocked; the rest is real).
- **`AIService` + `AIProvider`** — request shape, OpenAI-compatible
  payload, header/auth handling, response parsing all tested. This is
  the same code path summarization (#13), translation (#18), and
  general chat (#15) flow through.
- **Consent + availability gating** — the four-step check
  (feature flag → consent → API key → configuration) is verified in
  exact order, including live reflection when the flag toggles
  mid-flight. This prevents the class of bugs where chat fires
  before the user has accepted consent.
- **Cache correctness** — `AIRequestCacheKey` deterministic-on-input
  + `AIResponseCache` eviction prove repeated prompts don't
  re-bill the OpenAI-compatible endpoint.
- **Book-context extraction** — `AIContextExtractor` tested directly
  against fixture book texts; the chat ViewModel calls into it for
  every message that has a non-nil `bookFingerprint`.

What's deferred: the **chat UI flow + a real OpenAI-compatible
endpoint round-trip**. Streaming SSE parse against a real server
would catch endpoint-side surprises (e.g., provider-specific token
formats), and the UI gestures (open AI panel → switch to Chat tab →
keyboard input → tap Send) are pure UIKit/SwiftUI plumbing that
needs a display. The data + state + service layers are exercised end
to end below the network seam.

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Chat shares the AI stack with summarize / translate / general
  chat.** `AIService`, `AIRequestCacheKey`, `AIResponseCache`,
  `AIConsentManager`, `AIReaderAvailability` are all shared. Verifying
  chat exercises the same code paths the other AI features depend on
  — strong cross-feature confidence.
- **Streaming is part of the production contract.** The chat
  ViewModel accumulates SSE tokens into the in-flight message rather
  than waiting for a complete response. The streaming-path tests
  prove the accumulator behaves correctly, but the SSE parse is
  fixture-driven; a real provider could still surprise us.
- **Consent is checked BEFORE API key** — the documented order is
  feature flag → consent → API key → configuration. The
  `consentCheckedBeforeApiKey` test pins this so the consent dialog
  always wins the race against an API-key-missing error message.
- **`featureFlagToggleLiveReflectsInService`** matters because the
  AI feature flag is a settings toggle the user can flip at runtime;
  if a stale value were cached, the flag wouldn't take effect until
  app restart. Tested explicitly.
- **Book-context window is `AIContextExtractor`'s job.** The chat
  ViewModel doesn't reach into the reader; it asks the extractor
  for "context at locator X" and the extractor handles the window
  size + offset math. Keeps the chat ViewModel renderer-agnostic.
- **`AIAssistantViewModel`** is the orchestrator that the AI panel's
  three tabs (Summarize, Chat, Translate) share — its tests cover
  the dispatch + state transitions across modes.
