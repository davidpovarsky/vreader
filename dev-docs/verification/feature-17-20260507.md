## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 157 passing tests give **strong Gate-4-class evidence** across the
full PDF feature stack:

- **Real `PDFDocument` + `PDFAnnotation` + `PDFPage` boundary** — the
  annotation-bridge tests construct in-memory PDF documents (no
  mocking of PDFKit) and exercise real `addAnnotation` /
  `removeAnnotation` calls. The 0-1 normalized rect round-trip is
  verified with actual `PDFPage.bounds(for: .mediaBox)` math.
- **`PDFReaderViewModel` end-to-end** — 40 tests across 10 suites
  covering state transitions, position-service integration, password
  flows, background/foreground lifecycle, and edge cases. This is the
  same code path the UI drives.
- **Locator + PositionService integration** — round-trip tests prove
  page → `Locator.page` → page is stable across the document, and
  that debounced saves coalesce rapid page changes correctly.
- **Session tracker integration** — open / read / close /
  background / foreground cycles drive the `ReadingSessionTracker`,
  verified via `PDFSessionIntegrationTests` and
  `PDFBackgroundForegroundTests`.

What's deferred is the **PDFView native render layer** — actually
seeing the highlight overlay rendered on the page, observing the
selection tap interaction route through `UIMenuController`, and
verifying that the theme change propagates to the page background
color. PDFKit's `PDFView` is a UIKit class that renders via Core
Graphics; visual verification needs the simulator running with a
display + a UI gesture harness (neither available this iteration).

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **The "44 tests" line in the row was stale.** Counting today, the
  PDF feature has **157 tests across 17 suites** — annotation-bridge
  alone has 42, the ViewModel has 40 across 10 suites, page navigator
  has 26, progress helpers have 31. The per-row count was accurate at
  WI-008 and grew as edge cases landed; updating the row count is
  out of scope this iteration.
- **PDFAnnotationBridge tests don't mock PDFKit.** The tests build
  real `PDFDocument` instances in memory, call the real
  `addAnnotation` / `removeAnnotation`, and verify the document state
  through `PDFPage.annotations`. This is one of the more
  integration-flavored test surfaces in the codebase.
- **Normalize/denormalize rect round-trip is the load-bearing
  invariant.** Highlights are persisted as 0-1 normalized rects so
  they survive page-size differences (zoom level, rendering
  resolution). The round-trip identity test + the clamp/zero-size
  edge cases make this safe.
- **Multi-line selection produces multiple annotations** (one per
  visual line rect) — verified directly. This matches PDFKit's
  selection-rect behavior across line wraps.
- **`restoreHighlights` is defensive across malformed records** —
  skips nil PDF anchor, negative page index, and out-of-range page
  index. Prevents corrupted persistence from crashing the reader.
- **Background/foreground lifecycle is exercised twice** — once at
  the ViewModel level (`PDFReaderViewModelLifecycleTests`) and once
  at the integration level (`PDFBackgroundForegroundTests` builds a
  full session + drives notifications). Catches the class of bugs
  where the position is saved on background but not restored on
  foreground (or vice versa).
