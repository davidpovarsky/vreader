# Feature #17 — Round 2 device-verify (PDF reader UI slices)

## Summary

Round 1 (2026-05-07, v3.14.37) ran 157 unit tests across 17 suites
covering the PDF stack (annotation bridge, page navigator, ViewModel,
position service, locator, session, lifecycle) — all passing. Four UI
gesture / native render slices were deferred at that time:

1. Long-press → drag selection → Highlight menu
2. Tap Highlight → annotation renders in `PDFView`
3. Tap existing highlight → context menu → change color → re-renders
4. Theme switch → PDF page background flips (light/dark/sepia)

Round 2 (this round) addresses these UI slices on iPhone 17 Pro Sim
at v3.22.19 against an EXISTING-IN-LIBRARY PDF (the 4-KB DeviceKit
framebuffer mask PDF imported during Feature #59 round-2 verification).
**Result: 1 of 4 deferred slices verified PASS; 3 still deferred** —
the text-selection-dependent slices (1, 2, 3) need a text-bearing PDF
in the library, which is logistically blocked this iteration by
Bug #197 (library not auto-refreshing post-import — see Feature #59
round-2 evidence).

gestures meaningfully; the test fixture available in the library this
session (DeviceKit framebuffer mask, ~4KB, no selectable text) yields
only the degraded Copy/Select All popover.

## Commands run

### Step A — Open PDF, verify PDF reader chrome (criterion: PDF view loads)

- Tap PDF row in library → vreader opens PDF in PDFView.
- Chrome shows: back arrow, search, bookmark, contents, AA (settings).
- Bottom: scrubber + "Page 1 of 1" + page counter "1 / 1".

→ Implicit slice: PDF reader UI loads. PASS.

### Step B — Theme switch (criterion 4)

- Tap Light → Light dot fills; visible PDF page background lightens

→ **Criterion 4: theme switch → PDF page background flips: PASS**.

Artifact: `dev-docs/verification/artifacts/feature-17-r2-pdf-theme-light-20260516.png`.

### Step C — Long-press gesture (criterion 1, attempted)

- Long-press near center of PDF content area (around x=263, y=350,
  press-hold 2 seconds).
- iOS native popover appears: "Copy | Select All".
- No "Highlight" option in the popover.

→ **Long-press gesture wiring fires** — iOS popover appears. The
  popover degrades to default Copy/Select All because the PDF
  fixture has no selectable text (DeviceKit framebuffer mask is
  graphics-only). The Feature #17 custom "Highlight" menu item
  would only become available once `PDFSelection` carries
  user-selected text. This slice cannot complete here.

→ **Criterion 1: deferred** (needs text-bearing PDF). The gesture
  half is verified; the menu-item-wiring half needs real text.

Artifact: `dev-docs/verification/artifacts/feature-17-r2-pdf-long-press-popover-no-text-20260516.png`.

### Step D — Subsequent slices (criteria 2, 3) — deferred

Because criterion 1 cannot complete here, criterion 2 (tap Highlight
→ annotation renders) is gated and cannot proceed. Same for criterion
3 (which requires an existing rendered annotation to tap on).

## Observations

- The Feature #59 round-2 PR landed Share Sheet → vreader → PDF
  import as a working code path (DB row appears, but library doesn't
  refresh — Bug #197). To unblock Feature #17's text-selection
  slices, a future verify round needs either:
  - A text-bearing PDF imported via Share Sheet → cold-restart vreader
    → re-tap from refreshed library, OR
  - A bundled text-bearing PDF added to `DebugFixtureCatalog` for
    seed-based testing.
- The DeviceKit framebuffer mask PDF used here is geometrically simple
  (1 page, mostly opaque black). It was an opportunistic fixture from
  the Feature #59 verify session — useful for chrome + theme slices,
  not for text-selection slices.
- The popover content "Copy | Select All" matches PDFView's default
  edit menu when no text is selected. If Feature #17's
  `PDFAnnotationBridge` ever needs to extend the edit menu, that
  wiring is best verified with a controlled-content PDF where the
  selection range is known.

## Conclusion

1 of 4 deferred slices closes (criterion 4: theme + PDF bg). The
remaining 3 (text-selection-dependent: long-press → Highlight → render
→ color change) need a text-bearing PDF in the library to complete.
Feature row stays **DONE**; cannot flip to VERIFIED until the three
remaining slices are verified.

Next iteration's reasonable scope: either (a) bundle a small
text-bearing PDF fixture and rerun the three slices, or (b) wait for
Bug #197 to fix so the existing Share Sheet path can deliver a
text-bearing PDF into the library without cold-restart friction.
