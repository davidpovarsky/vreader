## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 92 passing tests give **strong Gate-4-class evidence** across
the translation feature:

- **`AITranslationViewModel` end-to-end** — the production ViewModel
  is exercised through the full state machine: idle → translating →
  translated, with the language-picker mutating target language,
  the bilingual view rendering both texts, and reset clearing
  state. Only the URLSession boundary is mocked.
- **CJK input** — `cjkSourceTextTranslates` proves the offset/text
  handling holds for Chinese/Japanese/Korean source text. Translation
  is a likely surface for double-byte-encoding bugs; covered.
- **Cache + cost-control** — translation is one of the more frequent
  AI calls (every word lookup if the user enables it). The cache
  tests prove repeated lookups don't re-bill.
- **Consent + feature-flag gating** — both gates are tested
  explicitly (`featureDisabledShowsError`, `consentRequiredShowsError`),
  matching the documented order in `AIReaderAvailability` (covered
  by feature #14 evidence).
- **Bug #95 regression context** — the row's "Device verified" note
  records that the Define→Translate flow was exercised on a real
  device after the fix. This iteration does not add a new
  isolated regression test for #95 specifically (initial tab
  behavior is harder to unit-test without driving the SwiftUI
  sheet); the device-verify on the row stands.

What's deferred: re-translation when the language picker changes
mid-sheet against a real endpoint, and a full 9-language end-to-end
pass. The data + state + service layers below the network seam are
exercised end to end.

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Translation entry points are two**: from the dictionary's
  Translate menu (Define→Translate sheet, originally bug #95) and
  from the AI panel's Translate tab. Both go through the same
  `AITranslationViewModel`. The dictionary entry was the bug #95
  device-verify scope; the AI panel entry is exercised through the
  ViewModel tests.
- **9 supported languages** — `supportedLanguagesContainsExpected`
  pins the list. Adding/removing a language without updating this
  test would break — desirable since the feature surface depends
  on the exact language set.
- **Different language → different cache entry** — important for
  cost: translating "Pierre" to French and then to Spanish must
  hit the API twice (different cache keys) but translating "Pierre"
  to French again must be a cache hit.
- **CJK is tested explicitly** — given the project's CJK-encoding
  history (bugs #92, #99, #143, etc.), every text-handling feature
  needs a CJK fixture. Translation has it.
- **Consent + feature-flag tests live in the translation suite even
  though the gating logic lives in `AIReaderAvailability`** — the
  ViewModel re-checks at translate time so a user who revokes
  consent mid-session can't keep translating.
- **Long content is handled** — `longContentHandled` covers the
  case where the user has selected a paragraph rather than a word;
  no truncation surprise vs. the API's max-token limit.
- **The "row already device-verified" framing is accurate** — the
  feature row explicitly notes "Device verified: Select word →
  Translate → opens Translate tab", which corresponds to the bug
  #95 fix path. Real UI gesture verification was done at that time.
