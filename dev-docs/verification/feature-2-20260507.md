## Observations

- **The "feature" is mostly the bug fix it was resolved by.** Bug #43
  added the yellow-background highlight + auto-clear timer to the
  search-tap-navigation path that already existed. The pipeline grew
  more substantial than the bug-fix label suggests:
  - `EPUBHighlightBridge.searchHighlightJS` ‚Äî JS injection that
    creates a CSS Highlight on the EPUB DOM at the matched text.
  - `HighlightableTextView.clearSearchHighlight` ‚Äî TXT/MD path; clears
    the temporary highlight from the `NSLayoutManager`.
  - `PDFViewBridge.searchHighlightText` ‚Äî PDF path; uses PDFKit's
    own findString + setSelection to mark the location.
  Each is tested in its format's own evidence file.
- **`SearchHighlightDismiss` covers the auto-clear half (feature #5).**
  25 tests pin scroll/tap/new-search dismiss triggers + idempotency
  + persistence preservation (the dismiss must not delete the user's
  permanent highlights, only the search-temporary one).
- **`HighlightedSnippet` covers the search-result-list half (feature
  #3).** Multi-word query bolding in the list view; a separate
  surface from the in-reader yellow background but visually
  contiguous ‚Äî the user sees both at search time.
- **Feature #5's 2026-05-04 verification was inconclusive on the
  visual half.** That run noted "search-result-tap navigation lands
  the matched text past the visible viewport so the visual highlight
  cannot be confirmed in the rendered region". That gap blocks #2
  and #5 equivalently ‚Äî it's a navigation-centring question, not a
  highlight-rendering one.
