## CU-driven repro on iPhone 17 Pro Sim (iOS 26.4) at v3.14.92

1. `xcrun simctl install "iPhone 17 Pro" <Debug-iphonesimulator>/vreader.app`
2. `xcrun simctl terminate "iPhone 17 Pro" com.vreader.app`
3. `xcrun simctl launch "iPhone 17 Pro" com.vreader.app --uitesting --seed-position-test --reset-preferences`
4. Open Position Test Book → reader (TXT, 100 paragraphs).

### Navigation half (bug #153 fix verified)

5. Tap **search** toolbar (magnifier icon at ~y=200).
6. Search modal opens, "Preparing search…" then ready.
7. Paste **"Paragraph 50"** into search field via clipboard (CU `type` had iOS soft-keyboard issues — pasted via `cmd+v` after `write_clipboard`).
8. Search returns 1 result: "Paragraph 50: This is test content for verifying rea…" with "Section 51" sub-text.
9. Tap the result row → search dismisses, reader scrolls to "Paragraph 50" with the matched line clearly inside the viewport (per bug #153's 25% top-headroom contract).
10. Captured at `feature-2-r2-after-search-tap-paragraph50-20260508.png`.

### Render-and-auto-clear half (bug #154 surfaced)

11. **Observed**: no yellow background highlight on "Paragraph 50" in the viewport, even when the screenshot is taken immediately after the result tap (within ~1 second — well before the 3s auto-clear timer would fire).
12. Re-opened search → result still shows "Paragraph 50" → tapped again → captured `feature-2-r2-immediately-after-tap-20260508.png` (timestamped 22:57:45.27 — same paint frame as the tap).
13. **Expected** (per feature contract + bug #43 fix + `HighlightableTextView.highlightColor = UIColor.systemYellow.withAlphaComponent(0.4).cgColor` at `HighlightableTextView.swift:27`): translucent yellow background on the matched range "Paragraph 50" for ~3s, then auto-clear via the timer at `TXTTextViewBridge.swift:144-152`.
14. Code-read confirms the wiring is broken: `TXTReaderContainerView.swift` has TWO `highlightRange` states (line 62 orphan `@State`; line 78 `uiState.highlightRange` set by `ReaderNotificationModifier.swift:34-47`). The bridge reads the orphan; the notification handler writes the other.

Filed as **bug #154** (`Refs #443`); detailed root-cause + fix sketch in the issue body and `docs/bugs.md` row 154. Per cron scope, NOT fixed this iteration.

## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

Two halves of the deferred slice:

- **Scroll half** — bug #153 fix means the matched text is reliably in
  the visible region after a search tap. This round confirms it works
  on the live simulator.
- **Render half** — the temporary yellow highlight that should mark
  the matched range NEVER appears on TXT reader, because of an orphan
  `@State` wiring in `TXTReaderContainerView.swift`. The notification
  handler updates a `TextReaderUIState` field that nothing reads on
  the TXT path, while the bridge reads a sibling `@State` that
  nothing writes. MD reader is correctly wired (`uiState.highlightRange`
  at `MDReaderContainerView.swift:312`); only TXT regressed.

The feature contract from `docs/features.md` and bug #43's fix is "yellow background on matched range, auto-clears after 3s". The render half is non-negotiable for the contract; with it broken, this round cannot flip to `pass`. Per cron scope, the bug is filed (#154 / GH #443) but NOT fixed in this iteration — bug-fix cron handles fixes.

`result: partial`. Tracker status stays `DONE`.

## Observations

- **Scroll half + render half are decoupled**. Bug #153 fixed scroll; bug #154 captures the orphaned `highlightRange` wiring. They surfaced in the same workflow but live in different code paths (`TXTOffsetMapper.scrollOffsetForVisibleMatch` vs `TXTReaderContainerView` `@State` plumbing).
- **MD reader is fine**. `MDReaderContainerView.swift:312` passes `uiState.highlightRange` to its bridge — the correct wiring. So MD search-tap should still render the highlight properly. Only TXT regressed. (Not exercised this round; recommended as a follow-up confirmation in bug #154's fix verification.)
- **CU `type` action vs iOS soft keyboard**. `mcp__computer-use__type` only registered the first character ("A") of "Paragraph 50" on the iOS simulator before iOS keyboard prediction interfered. Worked around with `write_clipboard` + `cmd+v`. Documenting as a CU+iOS-keyboard interaction note (third encounter of CU+SwiftUI quirks this session).
- **Result: `partial`** — scroll half passes (bug #153 fix verified); render half fails (bug #154 filed). Status stays `DONE`. The fix for #154 is a 5-line @State-wiring change; once shipped, the auto-clear timer half (currently blocked) becomes observable in the same flow.
