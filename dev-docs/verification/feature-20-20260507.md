## Observations

- **#20 was bundled into #6's WI-001.** The "persist library view
  preferences" feature naturally includes "reset preferences" as
  the inverse operation. Splitting into a separate row was the
  product / planning choice; the implementation lives in the same
  ViewModel.
- **The "Default" sort order is hard-coded.** The reset path doesn't
  read a user preference for "what's the default" — it returns to
  the documented hard-coded default (`addedAt` descending, per
  current convention). This means the test contract is simple:
  after reset, `sortMode == .addedAt && sortOrder == .descending`.
- **Reset is the bug-class boundary for "stuck preferences".** If a
  preferences-storage migration drops a user into an unusable
  state, "Reset to Default" is the recovery affordance. Without it,
  the only escape is delete-the-app — not great. The reset path is
  thus a load-bearing UX safety valve, even though it's tagged
  Low-priority.
