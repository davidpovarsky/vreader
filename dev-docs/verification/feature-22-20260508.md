## Summary

text in search result list — Medium priority). Round 1
(`feature-22-20260507.md`, `result: partial`) covered the
`HighlightedSnippet` pure-string transformation with 15 unit
tests: empty-query/empty-snippet edge cases, single-word
matching, case-insensitive matching, FTS5 markup tag stripping,
no-match plain text, multi-word query (with all-occurrences
highlighting), duplicate-word handling, whitespace-only safety,
extra-spaces handling, separate-vs-coalesced bold runs, regex
special-char escaping.

Round 1 explicitly deferred: "End-to-end UI: `SearchResultRow`
renders the `AttributedString` with bold runs visible".

This round runs that deferred slice end-to-end on iPhone 17 Pro
Sim (iOS 26.4) at v3.14.89 with CU display capture. **Result:
pass.** The bold runs render correctly across single-result and
multi-result scenarios; case-insensitive match works at the UI
layer; the search-result list renders the `HighlightedSnippet`
output as visually-distinct bold weight.

Status moves from `DONE` to **`VERIFIED`**.

## CU-driven repro on iPhone 17 Pro Sim (iOS 26.4) at v3.14.89

1. `xcrun simctl install "iPhone 17 Pro" <Debug-iphonesimulator>/vreader.app`
2. `xcrun simctl terminate "iPhone 17 Pro" com.vreader.app`
3. `xcrun simctl launch "iPhone 17 Pro" com.vreader.app --uitesting --seed-position-test --reset-preferences`
4. Open Position Test Book → reader.

### Single-word digit match

5. Tap search button → search panel slides up.
6. Tap field, type `100` (single keys).
7. Search returns 1 result. Snippet renders as:
   - **"Paragraph `100`: This is test content for verifying rea..."** with `100` rendered in bold weight (full 3-character run).
   - Sub-text: "Section 101" (segment label, regular weight).
   - Captured at `feature-22-r2-single-result-bold-100-20260508.png`.

### Multi-result word match (case-insensitive)

8. Clear field (tap × clear button).
9. Type `Reading` (iOS auto-capitalizes the R; underlying query is "Reading"; the source text contains lowercase "reading").
10. Search returns 200+ results (each paragraph contains "reading" at least twice).
11. Result list shows multiple snippets, each with the matched word **rendered in bold weight**:
    - "...h 1: This is test content for verifying **reading** position persistence. The reader should..."
    - "...eader should remember where you stopped **reading** and restore the scroll position when re..."
    - (continues for paragraphs 2, 3, 4, 5...)
12. Captured at `feature-22-r2-multi-result-bold-reading-20260508.png`.

The case-insensitive contract from round 1's
`caseInsensitiveMatch` unit test holds at the UI layer — query
"Reading" (capital R) matches and bolds lowercase "reading" runs
in the snippets.

## Observations

- **Bold weight is visually clear.** The `HighlightedSnippet`
  renders matches as iOS system bold attribute, not a
  background-color highlight. At simctl 3x screenshot scale the
  weight difference is unambiguous; on a real iPhone 17 Pro
  display at 1x reading distance it will also be clear.
- **Case-insensitive match is end-to-end.** Round 1's unit-test
  `caseInsensitiveMatch` predicted this; confirmed at the UI
  layer with iOS auto-capitalization producing "Reading" while
  source text uses lowercase "reading". The bolding still finds
  every match.
- **Section label correctly NOT bolded.** The secondary "Section
  N" line below each snippet is rendered in gray secondary
  weight via SwiftUI's `.font(.subheadline)` + `.foregroundStyle
  (.secondary)` — it doesn't go through `HighlightedSnippet` so
  no spurious bold runs appear there even when the segment number
  partially overlaps with a query term (e.g., searching "1" would
  not bold "Section 1" because it's a separate label).
- **Multi-result rendering is performant.** 200+ results loaded
  without observable delay; the FTS5 backend + per-result
  `HighlightedSnippet` invocation is fast enough that the list
  scrolls smoothly. (Performance characterization not the
  primary slice but worth noting.)
- **`--reset-preferences` was required** because prior rounds
  (#32 just before this) left UserDefaults in a state that put
  the reader in Unified mode + custom background, which
  interfered with this round's chrome-toggle / search-button
  reachability. With reset, the reader started in default Native
  + Light theme.
- **Result: `pass`** — all deferred UI-rendering slices from
  round 1 confirmed at runtime. Status moves to `VERIFIED`.
