## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 211 passing tests give **strong Gate-4-class evidence**:

- **Full rule-engine surface tested** — 4 rule types
  (`BSSearchRule`, `BSBookInfoRule`, `BSTocRule`, `BSContentRule`),
  both regex + CSS evaluators, the rule engine that orchestrates
  them. This is the largest test surface in the codebase outside
  the reader itself.
- **Legado JSON import is exhaustively tested** — 5 sub-suites
  for import / export / edge cases / compatibility classification /
  performance. Compatibility classification is the load-bearing
  bit: imported sources land in one of three buckets (standard
  Legado, Yima dialect, unsupported) and the user is told why an
  unsupported source can't be used.
- **Encoding detection covers GBK / Big5 / UTF-8 / ISO-8859-1** —
  Chinese web novels are the primary use case and they ship in
  any of these encodings. The detector's job is to read
  `<meta charset>` first and fall back to byte heuristics.
- **The pipeline + HTTP client run against fixture HTML** — search
  → book-info → TOC → chapter-content can be exercised end-to-end
  with a mocked URLSession returning fixture pages. This is the
  closest thing to a live integration test that doesn't need a
  real web server.
- **Bug #100 + #101 fixes are covered** — the
  `modelContext.save` pattern + SchemaV4 BookSource entity are
  both load-bearing for import to actually persist. Tests on the
  container + persist+fetch flow guard the regression.
- **The row already device-verified** — "Device verified: Import
  JSON → sources visible → search works" stands as Gate-5-class
  evidence for the import + search UI flow.

What's deferred: live scraping against a real Legado-compatible
source URL (would need a known-good source + network) and the
multi-source parallel search stress test.

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **211 tests / 20 suites is the largest verifiable surface in
  the AI/scraping side of the codebase**, on par with the reader
  bridges. The book-source feature has more edge cases than most
  features because it's parsing third-party JSON (Legado) and
  scraping arbitrary HTML; defensive tests pay back here.
- **CSS + regex evaluators are kept separate** — Legado sources
  use both CSS selectors and regex patterns. Each evaluator has
  its own test suite. The rule engine picks the right one per
  rule type.
- **Compatibility classification is the user-facing lever** — the
  importer puts each source in standard / Yima-dialect / unsupported
  buckets so the user sees why a source they pulled off the
  internet doesn't work. Without this they'd just get silent
  failures.
- **Encoding detection is the CJK gate** — Chinese web novels
  ship in GBK or Big5 about as often as UTF-8. The detector reads
  `<meta charset>`, falls back to BOM detection, then to byte
  heuristics. Same family of bugs as feature #4 (TXT auto-detect).
- **Chapter cache exists for offline reading** — once a chapter
  is fetched, the user can re-read offline. Tests cover read /
  write / eviction / per-source isolation so a noisy source
  doesn't push another source's chapters out of cache.
- **`SourceSharingService`** uses clipboard / share-sheet flows;
  the service tests cover the JSON shape that gets shared but
  not the UI clipboard read.
- **The HTTP client is mocked at the URLSession boundary** — same
  pattern as `AIService` and `WebDAVClient`. Catches ~95% of
  protocol bugs without needing a live server.
