## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> **Live integration tests against the actual backend** counts as
> verification. Unit + protocol tests alone are NOT verification —
> they cover the audit gate (Gate 4), not the integration gate
> (Gate 5).

The 26 passing tests give Gate-4-class evidence on the **data layer**:
zone math, persistence, modifier wiring, action mapping. They do
**not** drive a SwiftUI runtime — i.e., they don't prove that an
actual tap event in `TapZoneModifier`'s `onTapGesture` callback
flows through `TapZoneDispatcher.dispatch` to a NotificationCenter
post, and that the right reader observer (`ReaderContainerView`'s
`onReceive(.readerNextPage)` etc.) handles it correctly.

UI integration verification stays deferred until display recovers OR
a SwiftUI test harness (XCUITest with the `tapZoneOverlay`
accessibility identifier the modifier already exposes) is wired.

`result: partial` per SCHEMA. Tracker status stays `DONE` (already
there); no flip to `VERIFIED`.

## Observations

- **Boundary math is exhaustive.** The boundaries at X = ⅓ totalWidth
  and X = ⅔ totalWidth are tested at exact (`leftBoundary` X=330,
  `rightBoundary` X=660) AND just-past
  (`pastLeftBoundary` X=334, `pastRightBoundary` X=667) coordinates.
  The semantics are: `< 1/3` → left, `< 2/3` → center, `>= 2/3` →
  right. So X=333 lands in left (since 333/1000 = 0.333 < 0.3333),
  X=334 lands in center, X=666 lands in center (666/1000 = 0.666 <
  0.6666), X=667 lands in right. The test fixtures match this exact
  semantic.
- **Defensive math.** Zero width returns `.center` rather than
  crashing; negative X clamps to `.left`; X > totalWidth clamps to
  `.right`. Production crash safety is solid.
- **Codable persistence is real.** `TapZoneStore` reads + writes JSON
  to UserDefaults under key `readerTapZoneConfig`. Codable
  round-trip tests confirm the schema is stable, so future
  customizations will deserialize old user configs correctly.
- **Modifier excludes bottom 100pt by default.** That's the
  documented bug-#63 fix — the bottom safe area passes touches
  through to `ReadingProgressBar` and `ReaderBottomOverlay` rather
  than capturing them in the tap zone overlay.
- **The dispatcher path is mechanical.** `TapZoneDispatcher.dispatch`
  is a 4-case switch with no logic — each `TapAction` maps to one
  `Notification.Name` post (or no-op for `.none`). I verified by
  code-read; not unit-tested directly. A future "fires correct
  notification" suite would cover this trivially with an observer.
- **Native-mode safety is documented.** `TapZoneOverlay.swift:3-7`
  explicitly warns: "Only used for the UNIFIED renderer (SwiftUI-
  native scroll views). Native mode readers (UITextView, WKWebView,
  PDFView) handle taps via their own UITapGestureRecognizer / JS
  click handler. Do NOT apply tapZoneOverlay to native readers — the
  Color.clear overlay blocks scroll gestures from reaching UIKit
  (bug #70)." This is enforced by application-level discipline (not
  applying the modifier) rather than by a build-time check; the
  comment + bug #70 history is the contract.
