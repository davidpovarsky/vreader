## Summary

Gate-5 round-7 verification of feature #25 ("Configurable tap zones"),
attempting to close the in-reader tap-gesture leg that rounds 1-6 all
deferred. Approach: a CU-free pure-XCUITest suite
(`vreaderUITests/Verification/Feature25TapZonesVerificationTests.swift`,
4 tests) on iPhone 17 Pro Simulator (iOS 26.5).

**Outcome — terminal `partial`.** The verification did not close the
deferred slice; it discovered that **feature #25's user-facing capability
has been deliberately removed by feature #54**, so there is no in-reader
tap-zone behaviour left to verify and the row cannot flip to `VERIFIED`.

Feature #54 ("Remove Native/Unified reading mode toggle"):

- **WI-3 (PR #883, v3.31.7)** deleted `ReaderUnifiedDispatch.swift` — the
  *sole* `tapZoneOverlay(...)` install site. (Confirmed: no file in
  `vreader/` references `ReaderUnifiedDispatch` or `unifiedReaderView`;
  `ReaderContainerViewEngineDispatchTests` enforces the deletion.)
- **WI-4 (PR #886, v3.31.9)** removed the Tap Zones section AND the
  `TapZoneStore` wiring from `ReaderSettingsPanel` + `ReaderContainerView`.
  (Confirmed: `grep` finds zero `tapZoneSection` / `Tap Zones` /
  `TapZoneStore` references in either file; the regression-guard suite
  `ReaderSettingsPanelTapZonesGateTests` — 5 tests — locks this in.)

Net state on origin/main (`dab2dc3`): the `TapZoneConfig` / `TapZoneStore`
/ `TapZoneModifier` *types* survive as dead code (still unit-tested by
`TapZoneTests.swift`), but `tapZoneOverlay()` is applied to **no view**,
no settings UI exposes the left/center/right pickers, and no reader zones
taps by horizontal position. The four native bridges (TXT/MD via
`TXTTextViewBridge`, EPUB, PDF, Foliate) each post `.readerContentTapped`
unconditionally — they ignore `TapZoneConfig` entirely. This was already
true on every native renderer per Bug #162 (the overlay was *always*
Unified-only); feature #54 then deleted Unified mode, removing the last
place tap zones were live.

The 4-test suite verifies this current observable truth CU-free and is
all-green; what it cannot do is verify "tap right zone → next page" —
that contract no longer exists.

## Observations

- **The verification premise was obsolete before this round started.**
  The feature #25 row's most-recent rounds (5/6) were chasing a multi-page
  fixture to close the Left/Right zone slice. By the time that fixture
  landed, feature #54 had already deleted the entire tap-zone surface.
  This is the documented hazard of long-deferred verification slices: the
  feature can be refactored out from under the open verification debt.
- **The 4-test suite is genuinely CU-free and reliable.** XCUITest
  synthesizes its own coordinate taps + swipes — no computer-use. After
  one authoring iteration to remove a heavy per-iteration
  `panel.debugDescription` accessibility-snapshot call (which caused
  `testmanagerd` snapshot timeouts) and to widen the panel-open wait from
  5 s to 20 s (the `ReaderSettingsPanel` `List` is slow to realize on a
  cold simulator), the suite is 4/4 green on two consecutive full runs
  (~82 s each).
- **`test_verify_feature_25_taps_are_positionally_uniform_no_zoning` is
  the load-bearing behavioural assertion.** It synthesizes a tap at the
  left-third (dx 0.17) and the right-third (dx 0.83) of the EPUB reader.
  Feature #25's whole premise is that a tap's *horizontal position*
  selects an action (default config: left → previousPage, right →
  nextPage). The test observes that BOTH taps produce the identical
  chrome-toggle transition (`readerBackButton` disappears, then
  reappears) — i.e. every in-content tap routes to the bridge's own
  `.readerContentTapped` regardless of x. That is the direct, on-device
  proof that positional zoning is no longer wired.
- **`TapZoneOverlay.swift`'s own file header still says "IMPORTANT: Only
  used for the UNIFIED renderer"** — but the Unified renderer is gone, so
  the file is entirely dead code with a stale header. The
  `ReaderSettingsPanelTapZonesGateTests` header explicitly anticipates a
  "separate feature-#25 cleanup" to retire these types. That cleanup is
  product/triage scope, not verification scope — flagged here, not done.
- **No bug filed.** The removal is intentional and tracked (feature #54
  row + WI-4 PR #886 + the gate-test suite). A verification round filing
  a "tap zones don't work" bug would be wrong — they were *removed on
  purpose*. The correct action is tracker re-classification, which is a
  maintainer decision left to the user.
- **No feature #25 / #54 production code was changed.** The only repo
  additions are the verification test file, the regenerated
  `project.pbxproj`, this evidence file, and the version bump.
