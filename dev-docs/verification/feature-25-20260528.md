# Feature #25 ‚Äî Configurable tap zones ‚Äî VERIFIED (with a #54 scope reduction)

Gate-5 acceptance for the tap-zone feature. The runtime **behavior** (tap the
left / center / right reader zone ‚Üí the configured `TapAction`) is the feature's
user value, and it is device-verified here via computer-use. One acceptance
criterion from the original scope ‚Äî the **3-picker config UI in
`ReaderSettingsPanel`** ‚Äî was deliberately **removed by feature #54** ("Remove
Native/Unified reading mode toggle ‚Ä¶ Tap Zones removed from the settings panel").
That criterion is therefore *superseded by a later design decision*, not failed;
the default `TapZoneConfig` (left=previousPage, center=toggleChrome,
right=nextPage) is retained and the behavior still dispatches through
`ReaderTapZoneRouter` in every reader.

## Observations

- The behavior is wired into every reader (`NativeTextPagedView`,
  `TXTChunkedReaderBridge`, `TXTTextViewBridgeCoordinator`,
  `EPUBWebViewBridgeCoordinator`, `PDFViewBridge`, `FoliateSpikeView`) via
  `ReaderTapZoneRouter.dispatch` ‚Äî confirmed by code-read + the PDF device test.
- In scroll-mode readers the page-turn action is a no-op/scroll (expected); the
  PDF paged reader is the clean surface to observe `nextPage`/`previousPage`.
- The #54 config-UI removal means tap zones are no longer end-user-configurable;
  the feature now ships its sensible defaults. This is a deliberate product
  decision, documented on feature #54's row, not a #25 regression.
