## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as verification.
> Unit + protocol tests alone are NOT verification.

The 101 passing tests give **strong Gate-4-class evidence** across
the full TTS stack:

- **AVSpeechSynthesizer wiring** — `MockSpeechSynthesizer` mocks
  the OS API to test state-machine reactions to delegate callbacks
  (`willSpeakRangeOfSpeechString`, `didFinish`, etc.). The
  production `TTSService` logic is exercised end-to-end through
  the same delegate callbacks the system delivers.
- **HTTP cloud TTS** — `MockURLSession` mocks the network boundary
  to test request shape (POST body, headers, voice/text payload),
  response parsing, chunking, cancellation, caching. All **production
  `HTTPTTSProvider` code** is exercised; only the wire is mocked.
- **Foliate TTS bridge** — JS-string generation tested directly
  against expected JS payloads (`initTTSJS`, `startTTSJS`,
  `nextTTSJS`, `prevTTSJS`, `setMarkJS`). The Foliate-side
  evaluation is a WKWebView concern that needs a runtime to verify.

What's deferred is the **audio integration** layer — actually
hearing audio play, observing text highlights animate at the
right offset, and the UI playback controls. Those need the
simulator running with a UI gesture harness + audio session
(neither available this iteration).

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Three TTS providers, one TTSService.** Production has
  AVSpeechSynthesizer (system), HTTPTTSProvider (cloud), and
  FoliateTTSAdapter (AZW3/MOBI WebView bridge). Each has its own
  test suite; all three integrate through the same `TTSService`
  state machine.
- **Caching matters for HTTP TTS economics.**
  `HTTPTTSProviderCacheTests` proves repeated synthesis of the
  same text+voice doesn't re-bill the cloud API. Important for
  users who replay sentences or scrub backwards.
- **Foliate bridge tests are direct JS-string comparison.** The
  Swift side generates `initTTSJS()`, `startTTSJS(offset:)`, etc.
  as JS source; tests assert the generated string matches the
  expected Foliate-js API call shape. The actual evaluation in
  WKWebView is a separate runtime concern.
- **CJK extraction tested.** `textExtraction_fromReflowableSource_cjkText`
  covers Chinese/Japanese/Korean text where the offset → range
  mapping uses different boundaries than English.
- **Bugs #96 + #97 are referenced in the row but not isolated as
  individual regression tests.** Their fixes are absorbed into the
  state-transition + position-tracking coverage. A future improvement
  would be a one-test-per-historical-bug regression discipline, but
  that's outside this slice.
