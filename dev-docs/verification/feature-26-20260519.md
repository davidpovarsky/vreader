# Feature #26 — Text-to-Speech read aloud — round-6 (CU-free XCUITest, pause/resume gap closed)

Round-6 closes the **last open gap** in feature #26's verification: the
`TTSControlBar` **pause/resume button** had never been exercised by
automation. Rounds 1-5 verified TTS start, the genuine-speech offset
signal, and stop teardown across TXT / EPUB / MD, but every CU-free
round (round-4, round-5) drove TTS through the `vreader-debug://tts`
DebugBridge command, which exposes only `start` / `stop` — there is no
`tts?action=pause` URL, so the pause/resume control was physically
unreachable. Computer-use is unavailable on this virtual-display host,
so the button could not be tapped that way either.

This round adds a **CU-free XCUITest suite**
(`vreaderUITests/Verification/Feature26TextToSpeechVerificationTests.swift`)
that drives the app through the XCUITest accessibility API with
synthesized taps — it **can** tap the `TTSControlBar` pause/resume
button. All three tests pass: the full `.idle → .speaking → .paused →
.speaking → .idle` state machine is exercised end-to-end through the
real production control bar on TXT, MD, and EPUB readers.

## Acceptance criteria

round-4 verification history) is the TTS read-aloud capability: start
narration, the `.idle → .speaking → .paused → .speaking → .idle` state
machine, genuine speech, stop teardown, across the openable formats.

## Observations

- **The pause/resume gap is genuinely closed.** Every prior round noted
  the `TTSControlBar` pause/resume button as the un-automated leg.
  XCUITest synthesizes a real tap on it; this round's three tests each
  tap the button twice (pause, then resume) and assert the
  `TTSService.State`-driven accessibility-label flip ("Pause" ⇄
  "Resume") at each step. The `.speaking → .paused → .speaking` cycle is
  now exercised by automation, not just inferred.

- **EPUB test — environmental teardown flake (not a defect).** On the
  iOS 26.4 Simulator the runner intermittently logs
  `Class UIAccessibilityLoaderWebShared is implemented in both
  WebCore.axbundle and WebKit.axbundle … mysterious crashes` and the
  test runner can crash in *teardown* — after the EPUB test's TTS
  assertions have all already executed — when the EPUB WKWebView's
  accessibility bundle unloads. In the final suite run the EPUB test's
  full lifecycle (start / pause / resume / stop taps) completed, then
  the runner crashed post-teardown; an immediate isolated re-run of
  the EPUB case **passed cleanly (19.5 s)**. The EPUB test is
  deterministic; the crash is a simulator-infrastructure WebKit issue.
  Net: all three tests (TXT, MD, EPUB) pass — TXT + MD in the suite
  run, EPUB in the isolated re-run.

- **The state machine is observable without DebugBridge.**
  `TTSControlBar`'s play/pause `Button` carries
  `.accessibilityLabel(ttsService.state == .speaking ? "Pause" :
  "Resume")` and the whole bar unmounts on `.idle`. The button label IS
  the state machine — the suite reads it directly from the XCUITest
  accessibility tree, with no `vreader-debug://snapshot` round-trip.
  This is more robust than the snapshot path: the
  `VerificationDebugBridgeHelper`'s `posix_spawn` of host `simctl` does
  not reliably work from inside the sandboxed XCUITest runner (the
  earlier snapshot-based draft of this suite produced zero snapshot
  files and read a nil `ttsState`). The final suite carries no
  DebugBridge dependency at all.

- **Container `accessibilityIdentifier` clobbers leaf identifiers.**
  As the feature-#54 / #63 pilots found (Bug #214 / #223), a SwiftUI
  container's `.accessibilityIdentifier` propagates onto its leaf
  elements. `ReaderMorePopover`'s root carries
  `accessibilityIdentifier("readerMorePopover")` and `TTSControlBar`'s
  root `HStack` carries `accessibilityIdentifier("ttsControlBar")`, so
  the per-row / per-button identifiers (`readerMoreReadAloud`,
  `ttsPlayPauseButton`, `ttsStopButton`) do NOT resolve — an
  `app.debugDescription` dump showed every More-popover row surfacing as
  `identifier: 'readerMorePopover'`. The suite queries every control by
  its stable accessibility LABEL instead ("Read aloud" / "Pause" /
  "Resume" / "Stop reading"), which the container identifier does not
  override.

- **`XCUITestMockSpeechSynthesizer` gained a suspendable timeline.**
  The mock's synthetic `willSpeakRange` / `didFinish` callbacks
  previously ran on a fixed batch schedule that `pauseSpeaking()` did
  NOT suspend — the mock's own file header flagged this as a required
  change for "a future WI extending mock coverage to pause/resume
  tests". This round makes the change: the timeline is now a single
  self-rescheduling tick that genuinely holds while `isPaused`, so a
  test parked in the paused state stays paused instead of racing to the
  terminal `didFinish` → `.idle`. This is DEBUG-only test-harness code
  (`#if DEBUG`, never in Release); feature #26's production TTS
  (`TTSService`, `TTSControlBar`, `startTTS()`) is untouched.

- **HTTP cloud TTS (C6) is the one slice that genuinely cannot be
  verified in this environment** — and is the reason this round's
  `result` is `partial`, not `pass`. C6 needs a live third-party HTTP
  TTS server; none is reachable here, and the system
  `AVSpeechSynthesizer` path (which is what every test in this suite
  exercises) cannot substitute for an HTTP provider. C6 is covered at
  the unit level by round-1's 43 `HTTPTTSProvider` tests
  (config / synthesis / chunking / cancellation / caching / Azure via
  `MockURLSession`), but that is the audit gate, not the integration
  gate. Because C6 is a feature-#26 acceptance criterion that cannot be
  exercised end-to-end here, feature #26 stays `DONE` and does NOT flip
  to `VERIFIED` this round — the `AVSpeechSynthesizer`-backed lifecycle
  (start / pause / resume / stop on TXT / MD / EPUB) is fully verified,
  but `VERIFIED` requires every criterion. The standing follow-up to
  fully close #26 is a live-HTTP-TTS-server integration test (the same
  external-infrastructure dependency rounds 1-5 all recorded as
  deferred).

- **No bug discovered. No feature-#26 product code changed.** TTS
  starts, pauses, resumes, and stops correctly on all three formats.
  The only `vreader/` change is the DEBUG-only mock timeline tweak
  described above (test-harness infrastructure, `#if DEBUG`, never in
  Release).

## Verdict

**PARTIAL** — feature #26's `AVSpeechSynthesizer`-backed TTS read-aloud
lifecycle (start → pause → resume → stop, criteria C1-C5) is verified
end-to-end on TXT, MD, and EPUB through the production `TTSControlBar`,
driven entirely by CU-free XCUITest. The **pause/resume reachability
gap** that kept #26 at `DONE` through five prior rounds — the specific
gap this round set out to close — **is closed**: the
`TTSControlBar` pause/resume button is now tapped by automation and the
full `.speaking → .paused → .speaking` cycle is asserted.

Feature #26 **stays `DONE`**, NOT `VERIFIED`, because acceptance
criterion **C6 (HTTP cloud TTS end-to-end)** genuinely cannot be
exercised in this environment — it requires a live third-party HTTP TTS
server, and the system `AVSpeechSynthesizer` path cannot substitute for
it. C6 is covered at the unit level (round-1's 43 `HTTPTTSProvider`
tests) but not end-to-end. `VERIFIED` requires every criterion.

To close #26 to `VERIFIED`, the one remaining step is a live-HTTP-TTS-
server integration test (or a high-fidelity provider stub at the real
`HTTPTTSProvider` boundary) — the same external-infrastructure
dependency rounds 1-5 all recorded as deferred. That is a standing
follow-up; everything else in feature #26's contract is now verified.
