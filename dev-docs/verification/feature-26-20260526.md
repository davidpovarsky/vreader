# Feature #26 — Text-to-Speech read aloud (round 7 — final acceptance)

**VERIFIED.** Round 7 closes the SOLE remaining gate — acceptance criterion **C6
(HTTP cloud TTS end-to-end)** — which Feature #72 (HTTP cloud TTS provider
integration) implemented and verified. Every other criterion was device-verified
in rounds 2–6. With C6 now satisfied, #26 flips DONE → VERIFIED.

## Why C6 is now satisfied

Through rounds 1–6, #26 stayed at `DONE` for one reason: C6 ("read-aloud works via
a cloud TTS provider end-to-end") was UNIMPLEMENTED — `HTTPTTSProvider` was
orphaned, never wired into `TTSService` (this was later filed as Bug #270). The
#26 row recorded "VERIFIED gated solely on a live-HTTP-TTS-server integration test."

**Feature #72** (VERIFIED 2026-05-26, v3.39.29) implemented exactly that wiring —
`HTTPSpeechSynthesizer` adapts `HTTPTTSProvider` into the live `TTSService` path,
selected by `defaultSynthesizer()` when a valid `HTTPTTSConfig` exists — and
verified it end-to-end via a high-fidelity integration test through the real
`TTSService → HTTPSpeechSynthesizer → HTTPTTSProvider` stack (the AGENTS.md
close-gate verification-exception, since no live third-party HTTP TTS server is
available). That test (`vreaderTests/Integration/Feature72CloudTTSIntegrationTests.swift`)
IS #26's C6 end-to-end verification: a configured cloud endpoint speaks, per-chunk
HTTP round-trips drive playback, willSpeakRange advances the reader offset, and
failure stops cleanly to idle.
