## Observations

- **The transform engine is independent of where it's plugged in.**
  `ReplacementTransform` is a pure value-type transform over text;
  the bug class to watch (and the one bug #128 surfaced) is "the
  reader pipeline didn't wire it in for one rendering path". The
  unit suite cannot catch that â€” it pins the engine, not the
  wiring. Both layers must be checked separately, which is why
  this slice is partial even though all 14 tests pass.
- **Offset-map correctness is the load-bearing piece for highlights.**
  When the user has highlights at byte offsets 100â€“120, and a
  replacement rule changes "Mr." to "Mister" earlier in the text,
  every highlight downstream of the replacement must be adjusted
  by the +3-character offset. The
  `offsetMap_afterReplacement_correct` test pins the contract.
  Without this, re-rendering after a rule change would scatter
  highlights across the wrong text.
- **CJK is explicitly tested.** Replacement on CJK text uses
  Swift's String semantics (grapheme-cluster aware), but regex on
  CJK is where engines often misbehave. The `cjkCharacters_correct`
  test pins the case where a Chinese pattern matches a Chinese
  span and the replacement preserves UTF-8 alignment.
- **Bug #128 mitigation lives in `ReplacementRulesView`.** Users
  configuring rules see "Rules apply only when reading in Unified
  mode" so the silent-no-op in Native mode doesn't surprise them.
  This is honest UX â€” the mitigation says what it cannot do.
- **The native mode path (UITextView, PDFKit, custom WKWebView
  bridges) is intentionally NOT in scope here.** Wiring those
  engines through `ReplacementTransform` is a future work item.
  The Foliate/Unified path is what's wired today.
