## Observations

**Root cause of the EPUB-body fail**: `ReaderUnifiedDispatch.swift:73–76`. When an EPUB is "complex" (has stylesheets/images/JavaScript needing native rendering), `unifiedCoordinator.textContent` ends up nil and the dispatch falls back to `nativeReaderView(fingerprint:)` (WKWebView). The native WKWebView path does not consume `unifiedCoordinator.activeTransforms`, so the SimpTradTransform never reaches rendered HTML. "道诡异仙" is complex enough to take this fallback. Realistically, every published Chinese EPUB hits this fallback.

**Adjacent code smell at `ReaderUnifiedCoordinator.swift:128–130`**: when `activeTransforms` is non-empty and the EPUB IS simple, `attributedText` is deliberately set to `nil` with a comment "Note: attributedText is not transform-aware yet (text-only transform). For display, textContent takes precedence when transforms are active." That path can in theory render transformed text via `textContent`, but I could not exercise it without a small simple Chinese EPUB fixture.

**Native mode (default) wiring**: I confirmed by code-read that `chineseConversion` is only piped into `unifiedCoordinator.activeTransforms`, which is never read by the native UITextView/PDFKit/WKWebView paths. That means the setting is a no-op for the default reading mode — the most common user state. Whether that's intentional ("conversion is a Unified-mode V2 feature" matches the panel's blurb "Native uses format-specific readers. Unified reflow engine is coming in V2.") or a regression isn't documented in the row.

**Picker affordance**: the picker is only reachable by dragging the `.medium` detent sheet upward — a `.large` detent would make it discoverable without a gesture trick. Minor UX paper-cut.

**Defaults reverted**: `readerChineseConversion` and `readerReadingMode` deleted at end of run so the next iteration sees the unmodified default (none / native).

## Recommendation

File a bug: "Simp/Trad conversion has no effect in the realistic case (default Native mode + complex EPUBs)". Two scope options:

1. **Narrow fix**: route `activeTransforms` into the native EPUB WKWebView via a postMessage that swaps text nodes through `SimpTradDictionary`, so the existing setting starts working for the path users actually hit.
2. **Honest fix**: surface the limitation in the picker — disable the picker (or show a "Unified mode only" footer) when the current book/format is using native rendering, so the setting doesn't lie about doing something.

Either way, status #28 should NOT move to `VERIFIED` based on this run. Tracker row updated to reflect the partial gap. The 25 unit tests in `SimpTradTransformTests` (audit-gate) still pass — they verify the pure transform, not the wiring to displayed text.
