## Summary

Slice verification for feature #3 (Manual text highlighting —
High priority). Status was already `DONE` with the note "Resolved
by bug #44 — Highlight action added to UITextView edit menu" and
no evidence file. This iteration runs **84 tests across 14 suites**
covering the cross-format highlight pipeline core: ViewModel state
machine + edit + overlap + out-of-bounds + TXT UTF-16 edge cases,
the highlight coordinator (the boundary between selection events
and persistence), the highlightable text view bridge, dedupe,
record-anchor storage + migration, snippet formatting (used in the
highlight list), and `PersistenceActor` highlight CRUD against an
in-memory container.

Format-specific renderers (PDF, EPUB, Foliate AZW3/MOBI, TXT
chunked) are verified in their own per-feature evidence files
(#17, #11) — this slice covers the cross-format core that all
renderers consume. UI gestures (long-press text → edit menu →
Highlight → highlight applied + visible) deferred — display
unavailable.

## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 84 passing tests give **strong Gate-4-class evidence** for the
cross-format highlight core:

- **`HighlightListViewModel` end-to-end** — the ViewModel is the
  boundary between SwiftUI's HighlightListView and the
  SwiftData-backed `PersistenceActor`. 6 sub-suites cover load /
  add / remove / edit / out-of-bounds / overlap / TXT UTF-16 — the
  full state machine + edge cases.
- **`HighlightCoordinator`** — the cross-format dispatch that
  takes a selection event and routes it through the ViewModel.
  Coordinator + integration tests pin both the unit boundary and
  the wiring.
- **`HighlightableTextView` (bug #44 fix)** — the UITextView
  subclass that exposes the Highlight + Add Note edit-menu items.
  This is the entry point for feature #3 (manual highlight) and
  feature #4 (notes/annotations). Tests exercise the menu-item
  builder + action dispatch directly.
- **Dedupe + anchor + snippet** — supporting pieces. Dedupe makes
  rapid double-taps idempotent; anchors round-trip through
  Codable so persistence survives migrations; snippets format
  highlights for the list view with multi-word query bolding.
- **Anchor storage migration** — `HighlightAnchorStorage` lives
  under `vreader/Models/Migration/` and tests the SwiftData
  schema migration that moved anchors into a structured field.
  Important load-bearing piece — if migration breaks, every
  existing user's highlights silently disappear.
- **Bug #44 regression covered** — the Highlight + Add Note
  edit-menu actions are pinned by `HighlightableTextViewTests`.
  This was the bug that originally landed both feature #3 and
  feature #4.

What's deferred: the actual UI gesture path (select text →
long-press → edit menu opens → tap Highlight → see visual
overlay appear in the rendered view). Per-format render layer is
verified in each format's own evidence file (#17 PDF, #11 EPUB);
this slice is the cross-format core.

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Manual highlighting is cross-format.** The feature lives at
  the ViewModel + coordinator + edit-menu level so the entry
  point is uniform; the rendering of the highlight overlay is
  per-format (PDFAnnotation for PDF, CSS Highlight API for EPUB,
  Foliate-js for AZW3/MOBI, NSAttributedString for TXT/MD). This
  slice covers the "uniform" half; the per-format halves are in
  their own evidence files.
- **84 tests is more than I expected for a "Resolved by bug #44"
  feature.** The original feature was minimal (add the menu
  item) but the surrounding pipeline grew substantially —
  overlap detection, dedupe, anchor migration, multi-word snippet
  bolding, etc. The breadth reflects how much load-bearing
  infrastructure highlights touch.
- **`HighlightableTextView` is the bug #44 entry.** Before
  bug #44, UITextView's default edit menu had Cut/Copy/Paste/
  Look Up/Translate/Share but no Highlight or Add Note. The
  custom subclass adds those.
- **TXT UTF-16 alignment is a known hazard class** — same family
  as bug #99 (search highlights) and bug #92 (encoding). The
  dedicated `HighlightListViewModelTXTTests` suite pins the
  invariant.
- **Anchor storage migration** is what allowed feature #46
  (materializing restore) to round-trip highlights through a
  WebDAV backup; without structured anchor storage, restore
  wouldn't have a stable handle to reattach highlights to the
  recovered text.
- **Format-specific renderer tests are NOT counted here** —
  PDFHighlightRenderer (verified in #17), EPUBHighlightRenderer
  + bug #77 regression (verified separately), FoliateHighlightRenderer
  (cross-feature with #44/#21 AZW3), TXTChunkedHighlightDeferredTimer
  (covered by #99/#26 bug evidence). All in their own files.
