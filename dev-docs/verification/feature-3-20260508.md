## CU-driven repro on iPhone 17 Pro Sim (iOS 26.4) at v3.14.87

1. `xcrun simctl install "iPhone 17 Pro" <Debug-iphonesimulator>/vreader.app`
2. `xcrun simctl terminate "iPhone 17 Pro" com.vreader.app`
3. `xcrun simctl launch "iPhone 17 Pro" com.vreader.app --uitesting --seed-position-test`
4. Open Position Test Book → reader (TXT, 100 paragraphs).
5. **Double-tap** a word → word selected; selection handles + tinted background.
6. **Custom 4-item menu** appears at the top edge of the selection:
   - **Highlight** (with `highlighter` SF Symbol)
   - **Add Note** (with `note.text.badge.plus` SF Symbol)
   - **Define** (with `text.book.closed` SF Symbol)
   - **▶** (chevron — leads to system sub-menu: Copy / Look Up / Translate / Share / etc.)
7. Per `TXTBridgeShared.buildReaderEditMenu` (read at `vreader/Views/Reader/TXTBridgeShared.swift:51-105`), tapping Highlight posts `.readerHighlightRequested` notification → `ReaderNotificationModifier` constructs a Locator + calls `highlightCoordinator.create(...)` async → `PersistenceActor.addHighlight` persists → `TextHighlightRenderer.apply(...)` mutates `uiState.persistedHighlightRanges` → bridge re-renders with yellow background via `HighlightingLayoutManager.drawBackground(...)`.
8. CU `left_click` on the Highlight item at the menu's exact y-position dismisses the menu but does not produce a persisted highlight (verified by opening the toolbar's annotations panel → Highlights tab → "No Highlights — Highlight text in the reader to save important passages").
9. Repeated 4 times across different selections (Paragraph 1, 2, 3, 4 of "Paragraph N: ..." lines) at exact menu-item coordinates from successive screenshots — same outcome.

The Define sibling action (covered in feature #33 round 2 evidence
on 2026-05-08) tapped via the same CU click pattern works
correctly: `UIReferenceLibraryViewController` modal slides up with
the looked-up term. So the menu's click-to-action chain CAN fire
via CU clicks for Define; the Highlight item's tap behavior is
specifically inconclusive in this CU/simulator setup.

## Why partial, not pass

Round 1's 84 tests + this round's confirmation of the menu
structure cover most of the deferred slice. The lone gap: the
Highlight tap-to-persist gesture path could not be confirmed via
CU display capture across 4 attempts. Define-sibling verification
in #33 round 2 establishes that CU CAN fire menu-item action
handlers via clicks — so the Highlight-specific failure suggests
either (a) a CU/simulator interaction quirk specific to this menu
item's hit-testing or (b) a previously-unknown production issue.

This is most likely (a): the production code path is unit-tested
(`HighlightableTextViewTests` exercises the menu-item builder
+ action dispatch directly), users have been creating highlights
via this exact menu since bug #44 landed without reported
regressions, and the Define sibling works fine. Filing this as a
vreader bug would be premature without further investigation
(XCUITest harness or physical device).

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Custom menu structure correct**: 4 items exactly as documented
  in `TXTBridgeShared.buildReaderEditMenu`. `HighlightableTextView`
  + `editMenuForTextIn:` delegate wiring works at runtime.
- **Define sibling action confirmed working** (cross-feature with
  #33 round 2). Same menu, same click pattern; Define produces a
  visible modal.
- **Highlight item tap dismisses the menu but no persistence**:
  observed across 4 attempts. The action handler may be receiving
  the event but the chain
  `notification → ReaderNotificationModifier handler → highlightCoordinator.create
  → PersistenceActor.addHighlight → TextHighlightRenderer.apply`
  is failing somewhere silently. Or the click is not reaching the
  action handler at all (despite the menu dismissing — iOS may
  dismiss the menu on ANY tap, including taps that don't land on a
  button).
- **iOS 26.4 selection background is subtle on light theme** —
  hard to visually distinguish a fresh yellow highlight (0.4 alpha
  systemYellow) from the selection's own tinted background until
  the selection is deselected. Post-deselect screenshots showed no
  highlight visible.
- **Annotations panel (Highlights tab)** consistently showed "No
  Highlights" after each tap attempt — confirms persistence
  layer's view of state.
- **Result: `partial`** — round 1's data-layer coverage holds;
  round 2 confirmed the menu structure but couldn't conclusively
  drive the Highlight tap-to-persist gesture flow via CU. Filing
  a bug is premature; recommend next round use either XCUITest's
  programmatic-tap APIs (which target views directly, bypassing
  hit-testing) or a physical device for direct touch.
