# Feature #31 round-10 — last gate lifted: auto-page-turn advances the MD paged view end-to-end (VERIFIED)

## Context

Rounds 4-9 all returned `partial`. Round-9 (2026-05-21,
`feature-31-20260521-round9.md`) was the breakthrough: it confirmed the
round-8 blockers (Bug #215 MD paged layout, Bug #239 paged side-tap producer)
were both FIXED and MD paged mode now engages cleanly, then root-caused the
**sole remaining** criterion-5 failure as a distinct defect — **Bug #258 /
GH #1125**: `AutoPageTurner.scheduleTimer()` advanced the navigator's internal
page on its timer but never synced the observable `uiState.pagedCurrentPage`
that drives both the `NativeTextPagedView` render param and `snapshot.position`,
so the page never turned (position stuck at `0`).

Bug #258 was FIXED in `fix/issue-1125-autopageturn-paged-sync`, merged to main
via **PR #1127** (`4bd39177`, fix commit `bf19575f`). The fix added an
`AutoPageTurner.onAdvance: (@MainActor () -> Void)?` callback fired after each
timer-driven `nextPage()`; `TextReaderUIState.updateAutoPageTurner` installs it
to run `syncPagedState()` + the container's `onAutoAdvancePersist` — the same
sync the `.readerNextPage` observer does, MINUS the `pause()` (routing the timer
through `.readerNextPage` directly would have been fatal — that observer calls
`autoPageTurner?.pause()` per Bug #131, halting auto-advance after one tick).

Round-10 re-runs the round-9 recipe on merged main (now v3.39.4, build 625)
**with Bug #258's fix present** to determine whether criterion 5 — the only
gate left — now passes and the `DONE → VERIFIED` flip is unblocked.

Auto page turning is timer-driven, so this round verifies it **CU-free** via
`snapshot.position` deltas over time + a `simctl io` screenshot. CU was DOWN
(virtual-display host) and was not used or needed.

## Observations

- **Same CU-free method, opposite verdict from round-9.** Round-9's position
  sequence was `0 / 0 / 0` (stuck); round-10's is `0 → 3523 → 4758 → 5970 →
  7258 → 8440` (monotonic, then auto-stop). The position-delta harness cleanly
  distinguishes "advances" from "doesn't" with no computer-use — exactly the
  signal Bug #258's fix targeted.
- **The cadence matters for reading the climb.** At a 3s cadence (coarser than
  the 2s interval) the first post-open snapshot already shows `8440` because
  multiple ticks elapsed between snapshots; the 1.5s fine cadence (run b) is
  what exposes the intermediate pages (3523, 4758, 5970, 7258). Both runs agree
  the page advances and stops at `8440`.
- **Auto-stop at last page works.** Position plateaus at `8440` (document end,
  last page) — `AutoPageTurner`'s last-page guard (`nav.currentPage >=
  nav.totalPages - 1`) returns before `nextPage()`, so `onAdvance` correctly
  stops firing there. `renderPhase: idle`, `lastError: null` at the end — no
  wrap, no crash.
- **Pagination count varies (8 vs round-9's 6 "Page X of 6").** The
  `--seed-md-multi-page` fixture paginates to a slightly different page count
  depending on viewport/content rendering, but the document-end position is the
  same `8440` and the behavior is identical: full auto-advance page-1 → last,
  then stop. The criterion is "advances over interval + stops at end," which
  holds regardless of the exact page count.
- **Bug #259 workaround held.** The clean build into a fresh `build/verify-31b`
  produced an Info.plist with the `vreader-debug` scheme present; every
  `simctl openurl` succeeded (no error 115). The "Inject DebugBridge URL types"
  no-output build-phase ran (build warning confirms).
- **Disposition**: all five criteria PASS → `result: pass`. Feature #31 flips
  `DONE → VERIFIED`. No code changed in this verification round; no new bug
  found.
