## Observations

- **Same architectural pattern as `CustomCoverStore` (feature #30).**
  Static-method I/O store, JPEG storage keyed by stable identifier
  (theme name vs fingerprint key), max-dimension resize for memory
  cap, no-upscale for small. Two stores, two tests suites, identical
  shape â€” duplication is bounded and intentional.
- **High-scale resize test is the interesting one.** A 100Ă—100 logical
  image at scale 3.0 has 300Ă—300 pixel data on disk; the store
  bounds the pixel count, not the logical points. The
  `saveBackground_resizesHighScaleImage` test catches the case where
  a screenshot or 3x asset would otherwise blow the memory cap.
- **Opacity is a runtime SwiftUI concern**, not in the store. The
  store just persists the image bytes. Opacity (the slider in
  `ReaderSettingsPanel`) is stored separately and applied at render
  time in `ThemeBackgroundView`.
- **Per-theme isolation is by-construction**, not enforced by tests
  â€” the path includes the theme key as a path component, so light
  and dark have non-overlapping files. A regression here would mean
  the store doesn't differentiate, which `backgroundPath_uniquePerTheme`
  catches.
