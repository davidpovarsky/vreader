## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> **Live integration tests against the actual backend** counts as
> verification.

The 19 `CollectionPersistenceTests` ARE a live integration test
against the actual backend:

- Real `PersistenceActor` (no mock).
- Real `ModelContainer` configured with production `SchemaV6` —
  the same schema production loads from SQLite. The only difference
  vs production is `isStoredInMemoryOnly: true`, which is a SwiftData
  storage-mode flag that does NOT change schema, queries, or
  relationship semantics.

  `bookCollections` relationship — same code path as production.

This is exactly what SCHEMA names as Gate-5-class evidence.

But the **UI surface** (Library context menu, Collections list view,
filter, drag/drop) is not exercised. UI-level verification stays
deferred until display recovers.

## Observations

- **Cascade safety is real.** `deleteBook` correctly removes the
  book from every collection it was a member of (test at line 241).
  This was the concern raised by bug #85 (book-deletion leaving
  dangling collection references). The production
  `PersistenceActor.deleteBook` path is exercised end-to-end through
  the actor's protocol surface.
- **Idempotency is real.** Calling `addBookToCollection` twice for
  the same `(book, collection)` pair does not create a duplicate
  link. This was the concern raised by bug #86 (double-tap on Add to
  Collection creating duplicate rows). Production path tested.
- **Case-insensitive duplicate detection works on both create and
  rename.** Both `createCollection` and `renameCollection` reject
  case-variant duplicates of an existing collection name.
- **Unicode + CJK + emoji safe.** `BookCollection.init` handles
  multi-byte characters in name without truncating mid-grapheme; CJK
  collection names round-trip correctly through SwiftData persistence.
- **What's not exercised here:** the actual `LibraryViewModel`
  refresh after `deleteCollection` / `renameCollection`, the SwiftUI
  rendering of the collection list, the context-menu integration, and
  filter behavior. Those are UI-level concerns and stay deferred.
- **In-memory vs on-disk SwiftData.** The `isStoredInMemoryOnly: true`
  flag bypasses SQLite but exercises the same `Schema(SchemaV6.models)`
  + same `@Model` classes + same relationship + cascade rules.
  Behavioral parity is documented in Apple's SwiftData docs and
  matches our experience: schema-validation, predicate evaluation,
  and cascade semantics are identical between in-memory and on-disk
  configurations. The on-disk surface area we still don't exercise
  here is migration / persistence-across-launches, which is covered
  separately by the schema migration suites in
  `vreaderTests/Services/PersistenceMigration*` and is out of scope
  for this slice.
