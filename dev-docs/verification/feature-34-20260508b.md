## Summary

Round-3 slice for feature #34 (Collections / tags / series organization
— Medium). Extends rounds 1 (`feature-34-20260507.md`, data-layer 19
unit tests) and 2 (`feature-34-20260508.md`, XCUITest empty-state
shape, 4 methods) by closing the **create → swipe-delete UI
round-trip** that was deferred under round 2 ("per-book Add-to-
Collection assignment, tap-to-filter, rename, swipe-to-delete, and
populated Tags/Series sections still deferred").

This round used **computer-use display capture** to drive the
gesture flow on iPhone 17 Pro Sim end-to-end:

1. Library empty (`--seed-empty`) — folder toolbar button visible.
2. Tap folder (Collections) → sidebar presents under `Filter` nav title.
3. Empty state: All Books row, Collections section header, `+ New Collection` row.
4. Tap `+ New Collection` → row flips to inline TextField + greyed Add button (Add gated on non-empty input).
5. Type "My list" via `key` action one-character-at-a-time (CU's `type` action doesn't reach iOS Simulator text fields — see Observations).
6. Add button turns blue (enabled) once text is entered.
7. Tap Add → collection row "My list" with count "0" appears in Collections section; inline form returns to `+ New Collection` entry-point.
8. Swipe-left on "My list" row → red Delete button revealed via `.swipeActions(edge: .trailing)`.
9. Tap Delete → row removed; section returns to empty-state shape.

All 9 transitions captured visually via `mcp__computer-use__screenshot`
plus `xcrun simctl io booted screenshot` for high-resolution artifacts
(3 PNG files saved).

## Observations

- **No regressions surfaced**: every transition the production code
  promises rendered correctly. Validation (Add-disabled-on-empty),
  optimistic UI update (count "0" appearing immediately), and
  swipe-delete all work as expected.
- **CU `type` action doesn't reach iOS Simulator text fields**:
  hardware-keyboard is connected (`defaults read com.apple.iphonesimulator
  | grep -i hardware` → `ConnectHardwareKeyboard = 1`), but the bulk
  `type` action's keystrokes don't propagate into the SwiftUI
  TextField. Single-key `key` actions work fine (verified by
  typing "M y space l i s t" character-by-character). This is a
  CU/Simulator interaction quirk, not a vreader bug — but it's
  worth flagging for future test infrastructure: prefer `key`
  bursts over `type` for iOS form input via CU. (Not filing as a
  vreader bug — it's external infrastructure.)
- **Auto-capitalization observed**: typing lowercase `m` produced
  "M" in the field (iOS default first-letter auto-capitalization on
  text fields). Subsequent letters retained their case. Production
  behavior — no concern for this verification round.
- **Sim binary version mismatch**: the running binary on the sim
  was 3.14.80 (build 189), behind 3.14.82 (build 191) which is
  HEAD. This round's observations cover the sidebar UI which
  hasn't changed across these patches — round still valid.
- **Result: pass** for the in-scope criteria. The remaining
  deferred legs (per-book context-menu Add to Collection,
  tap-to-filter, rename, populated Tags/Series) require either
  pre-seeded data or harness extensions not in scope for this
  iteration.
