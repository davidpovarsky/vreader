## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> **Live integration tests against the actual backend** counts as
> verification. Unit + protocol tests alone are NOT verification —
> they cover the audit gate (Gate 4), not the integration gate
> (Gate 5).

The 47 passing unit tests give Gate-4-class evidence. They DO cover
the production round-trip:

- Encoder side: tests use a JSONEncoder with the **identical settings
  string** as `JSONExportFormatter` (`.iso8601` dates, `.prettyPrinted
  + .sortedKeys`). Identical settings produce identical bytes.
- Decoder side: tests use the **production**
  `VReaderAnnotationParser.parse` directly, no test seam.
- Importer side: tests use the **production**
  `AnnotationImporter.importJSON`, no test seam.
- Store side: mocks substitute `HighlightPersisting`/
  `BookmarkPersisting`/`AnnotationPersisting`. The real
  `PersistenceActor` extension implements those same protocols, so
  the substitution is mechanical.

But — the **mock-store seam is the gap**. A "live integration" test
would substitute the real `PersistenceActor` (in-memory
`ModelContainer`) for the mocks and assert that the imported
annotations land in the real SwiftData stores correctly. Because the
seam is at the protocol boundary and the protocol is well-typed,
substitution is mechanical, but until that test exists, the
integration claim isn't independently verified.

UI gesture flows (export menu, share sheet, file picker, panel
refresh including bug #88's fix) are also deferred to a UI-driven
iteration.

`result: partial` per SCHEMA. Tracker status stays `DONE` (already
there); no flip to `VERIFIED`.

## Observations

- **Round-trip is structurally guaranteed.** `JSONExportFormatter` uses
  Swift `JSONEncoder` with `.iso8601 + [.prettyPrinted, .sortedKeys]`.
  `VReaderAnnotationParser` uses `JSONDecoder` with `.iso8601`.
  Both serialize/deserialize the same `AnnotationExportPayload`
  Codable type (`vreader/Models/ExportedAnnotation.swift`), so by
  Codable's contract any payload encoded by the formatter decodes
  cleanly via the parser. The 13 importer tests exercise this round
  trip with realistic data shapes.
- **Schema drift is the realistic risk, not implementation drift.**
  Because both encoder and decoder go through the same Codable type,
  any field rename/addition at the type level breaks both sides
  symmetrically — there's no asymmetric drift between
  `JSONExportFormatter` and `VReaderAnnotationParser` short of one
  adding a custom `init(from:)` and the other not. Future change to
  `ExportedAnnotation` should run both `JSONExportTests` and
  `AnnotationImporterTests` together.
- **Bug #88 (panel refresh after import) is NOT verified here.** That
  fix is at the `@Observable` propagation boundary in
  `AnnotationsPanelView` — outside the importer's scope. A UI-driven
  slice (open panel → import → see new entries appear without close-
  and-reopen) would close that gap. Filed as deferred, not
  re-opened, because the bug is on `main` as FIXED.
- **`importer.existingAnnotationIds` is captured at importer-init time
  and never refreshed.** If a user imports two files in quick
  succession, the second import won't see annotations created by the
  first unless a fresh `AnnotationImporter` is constructed in
  between. This is documented behavior (the property is `let`), but
  worth keeping in mind for the eventual UI flow — the panel's
  "Import" button needs to construct a fresh importer per tap, not
  cache one. Not filed as a bug because it's the documented contract,
  not broken behavior.
