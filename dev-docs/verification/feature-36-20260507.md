## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`, "Unit + protocol tests alone
are NOT verification — they cover the audit gate (Gate 4), not the
integration gate (Gate 5)." This iteration:

- The 24 parser unit tests (Gate 4-class evidence on the parser
  schema).
- A **real-backend wire-format probe** against live Project Gutenberg
  — the actual XML that `OPDSClient.fetchFeed` would receive on the
  wire. This IS Gate-5-class evidence for the parse path, since
  Gutenberg is a third-party server outside the test fixtures and
  conforms (or doesn't) to the parser independently.

But the **download → import slice is not exercised** — that path
requires UI driving (tap an acquisition link) and the
`BookImporter.importFile(url:source:)` integration that follows.
That slice stays deferred until computer-use display recovers OR a
DebugBridge command exposes "import-from-OPDS-URL" so it can be
driven via `vreader-debug://`.

Because of the deferred slices, `result: partial` and the row is
flipped TODO → DONE (not VERIFIED).

## Observations

- **Project Gutenberg is a high-fidelity OPDS test backend.** The wire
  format conforms to OPDS 1.x cleanly (Atom namespace + opds + dcterms
  + opensearch + relevance namespaces), and exposes the
  three-tier hierarchy vreader's parser is built for: root navigation
  → search/listing (with `subsection` links) → per-book detail (with
  `http://opds-spec.org/acquisition` links across multiple formats).
- **`OPDSEntry.coverURL` matches reality.** Gutenberg embeds covers as
  `data:image/png;base64,…` payloads on `rel="http://opds-spec.org/image/thumbnail"`,
  exactly the rel the model checks for. (Size: ~1.4 KB per cover —
  cheap.)
- **`OPDSEntry.navigationURL` rel-matching is correct for Gutenberg.**
  The popular-books feed exposes book entries via `rel="subsection"`
  on a per-book OPDS detail link, which `OPDSEntry.navigationURL`
  explicitly handles. Without that, vreader would treat the
  acquisition feed as a dead end.
- **Standard Ebooks rejected the probe (HTTP 401).** Their OPDS feed
  requires a registered patron account; this is documented on their
  site. Gutenberg is the better choice for a public verification
  target. Future verification of basic-auth could use a local test
  Calibre OPDS server (Calibre's `calibre-server --auth-mode basic`),
  which would also exercise the `OPDSCredentials` path that's untested
  this iteration.
- **What's deferred is not just "tap-driven", it's the
  `OPDSClient.downloadBook` → `BookImporter.importFile` round-trip.**
  Wire-format conformance is necessary but not sufficient — a download
  could still fail due to MIME-type confusion, EPUB signature
  validation, or schema-mismatch on import. A future DebugBridge
  command (`vreader-debug://import-from-url?url=…&format=epub`) would
  let this be driven without UI.
