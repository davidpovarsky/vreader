## Summary

Round-2 slice for feature #36 (OPDS catalog support — Medium).
Extends the 2026-05-07 evidence (`feature-36-20260507.md`) which
verified the parser layer (24 unit tests covering Atom / OPDS 1.x
schema, dedup, navigation vs acquisition, cover-URL, Unicode/CJK)
plus three live curl probes against Project Gutenberg confirming
real-backend wire-format compatibility — with `result: partial — UI
flow deferred`.

This round closes the **OPDS catalog list sheet presentation leg**
via a new XCUITest suite of 4 methods (31.9 s wall-clock):

- `vreaderUITests/Library/OPDSCatalogListTests` (4 methods) —
  verifies the catalog list sheet opens from the
  `opdsCatalogsToolbarButton` (globe icon) under the `OPDS Catalogs`
  navigation title, surfaces the toolbar `+` (`opdsAddCatalog`)
  entry point, dismisses cleanly via Done
  (`opdsCatalogsDoneButton`), and passes the iOS 26 accessibility
  audit. Assertions are **state-invariant** so the suite is robust
  whether the user has zero catalogs (empty-state shape) or any
  catalogs already saved.

The state-invariant framing is a deliberate response to a test
infrastructure gap discovered during the round (filed as bug #152 /
GH #426): \`--uitesting\` does NOT clear UserDefaults state, so the
empty-state shape is non-deterministic across simulator sessions.
Per-catalog navigation (tap row → OPDSBrowserView → entry detail →
download) and Add Catalog flow (tap `+` → form → save → row appears)
are still deferred — they'd need either a UserDefaults-clearing
harness command or a fixture catalog server stub.

All 4 synthesized real taps via XCUIApplication, no Computer Use,
no display capture. Target simulator was iPhone 17 Pro (iOS 26.4
Simulator). Same harness pattern as PRs #419 / #420 / #422 / #425.

## Observations

- **Bug #152 / GH #426 filed**: an early draft of the suite asserted
  the empty-state shape (`opdsEmptyState` container + the empty-state
  Add Catalog button). 2/6 tests failed because OPDS catalog state
  persisted in UserDefaults at `opds.savedCatalogs` from a prior
  simulator session — `--uitesting` only swaps the SwiftData store
  for in-memory; UserDefaults is unaffected. The suite was refactored
  to state-invariant assertions; the infrastructure gap was filed for
  separate work. Affects #15, #28, #32, #36, #37 future verification
  rounds — anyone needing UserDefaults initial-state isolation will
  hit the same flake.
- The toolbar `+` (`opdsAddCatalog`) and the empty-state prominent
  Add Catalog button (`opdsAddCatalogEmpty`) are TWO separate entry
  points to the same Add flow — confirmed by code-read at
  `OPDSCatalogListView.swift:41-49` (toolbar) and lines 78-89
  (empty-state). The toolbar version is always present; the
  empty-state version only when `catalogs.isEmpty`.
- No production code touched in this verification round — pure
  XCUITest addition leveraging accessibility identifiers already
  present in `OPDSCatalogListView.swift` and `LibraryView.swift`.
