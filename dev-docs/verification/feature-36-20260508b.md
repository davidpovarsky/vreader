## Summary

Round-3 slice for feature #36 (OPDS catalog support â€” Medium).
Completes the visual / display-based verification leg that was
deferred under round 2 (`feature-36-20260508.md`) because XCUITest
couldn't reach into the Add Catalog modal â€” iOS 26.4 SwiftUI
propagates the outer VStack's `opdsEmptyState` accessibility ID
down to descendant elements, shadowing `opdsAddCatalogEmpty` on the
inner Button (same pattern as bug #150's `txtReaderContainer` on

This round used **computer-use display capture** to drive the
gesture flow end-to-end and screenshot every shape that was
previously dark:

3. Tap globe (OPDS toolbar button at ~(760, 161)) â†’ OPDS Catalogs sheet presents.
4. Empty state visible: globe icon, "No OPDS Catalogs" heading, "Add an OPDS catalog server to browse and download books." description, and prominent "Add Catalog" button.
5. Tap toolbar `+` (~(820, 169)) â†’ Add Catalog modal pushes onto the navigation stack.
6. Modal renders with: Cancel / Save toolbar (Save **correctly disabled** on empty input â€” gate against whitespace-only catalogs), "Catalog Info" section (Name + URL fields with placeholders), "Authentication (Optional)" section (Username + Password fields).
7. Tap Cancel â†’ returns to OPDS Catalogs empty state.
8. Tap Done â†’ returns to Library.

All 8 transitions captured visually via `mcp__computer-use__screenshot` plus `xcrun simctl io booted screenshot` for high-resolution artifacts.

## Observations

- **No regressions surfaced**: every shape that XCUITest round 2 verified
  programmatically renders exactly as expected on screen. The
  visual layout matches the production code in
  `OPDSCatalogListView.swift:62-91` (empty state) and the
  `addEditSheet` modal (Form-style with two sections).
- **Save button gating** is visually clear: the button is rendered
  in a muted grey when fields are empty, and the toolbar layout
  matches the screenshot exactly. This was the missed gate in
  XCUITest round 2 (Picker/Form rendering nuance) â€” display
  capture closes the gap.
- **iOS 26.4 SwiftUI parent-id propagation** confirmed visually:
  the empty-state's prominent button has the correct visual
  identity (filled blue Capsule-style "Add Catalog") even though
  XCUITest can't query it by `opdsAddCatalogEmpty` because the
  outer VStack's `opdsEmptyState` shadows the child. This is the
  same pattern documented in bug #150's fix and now in
  `OPDSCatalogListTests.testOPDSEmptyStateWithResetPreferences`'s
  comment.
- **Result: pass** for the in-scope criteria. The remaining
  deferred legs (per-catalog navigation, download â†’ import, edit
  / delete) all require a real OPDS endpoint and are tracked as
  follow-up â€” they can't be closed without backend infrastructure
  separate from the UI surface.
- **Sim binary version mismatch**: the running binary on the sim
  was 3.14.80 (build 189), one revision behind 3.14.81 (build 190)
  which shipped the `--reset-preferences` flag. The CU verification
  uses `--seed-empty` instead of `--reset-preferences` to wipe
  state, so the older binary is sufficient for this round's
  observations. Future verification rounds touching the new flag
  should `simctl install` v3.14.81 first.
