## Observations

- **The level field is the entire feature.** `TOCEntry.level: Int`
  encodes nesting depth (h1 = 0, h2 = 1, ...). Everything else is
  ergonomics: the view uses level for padding + font weight, the
  builders set it per format. No tree data structure, no
  parent-child references — flat array with depth annotation. The
  spec explicitly states "Not a disclosure tree but visual nesting
  works", and the implementation honors that.
- **MD builder is the cleanest test surface.** Markdown has explicit
  `# / ## / ###` level markers, so tests can assert exact level
  values (`h1 → 0`, `h2 → 1`, etc.). The MD test suite includes a
  parametric h1-through-h6 walk that covers all 6 nesting levels.
- **TXT chapter detection is fuzzier.** TXT files don't have
  explicit headings, so the builder uses 14/25 Legado-style regex
  rules (per bug #83 fix) to identify chapter starts. The TXT tests
  verify level assignment for the patterns those rules detect, but
  the level signal is less rich than MD's (typically just 0 or 1).
- **TOCListView design choice — flat with indent.** A SwiftUI
  `DisclosureGroup` tree would let users collapse subsections, but
  the spec's choice (flat list with leading padding by level)
  trades collapsibility for performance and simplicity. With 1860
  chapters in real-world fixtures (per feature #23 evidence), a
  flat list is the right architecture.
- **No regression with PDF / EPUB.** PDFOutline-derived levels and
  EPUB NCX-nesting levels feed into the same `TOCEntry.level` field
  the view consumes. The view doesn't care about format — it just
  reads `entry.level`.
