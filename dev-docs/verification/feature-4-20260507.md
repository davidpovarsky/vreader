## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 130 passing tests give **strong Gate-4-class evidence** for the
notes/annotations cross-format pipeline:

- **`AnnotationListViewModel` end-to-end** — load / add / remove /
  edit / edge cases. The ViewModel is the boundary between
  AnnotationsPanelView and the SwiftData-backed `PersistenceActor`.
  4 sub-suites cover the full state machine including content
  validation (empty / whitespace rejected) and Unicode/CJK
  preservation.
- **Unified annotation model** — `Highlight`, `Bookmark`, and
  `AnnotationNote` are siblings sharing a `profileKey`. The model
  tests pin the invariant that all three types serialize their
  profile key consistently. Important load-bearing piece — the
  panel's "All" tab depends on this consistency.
- **Per-format AnnotationAnchor** — EPUB (xpath + CFI offsets), PDF
  (page index + rect set), TXT (UTF-16 char range). 24 tests cover
  Codable round-trip, SHA-256 hash stability across calls, and
  hash-differs across anchor types. SHA stability is what makes
  re-attachment work after a backup/restore round-trip.
- **`PDFAnnotationBridge` (PDFKit rendering)** — 24 tests cover
  highlight creation on a real `PDFDocument` page, color mapping
  (yellow / blue / green / pink string → UIColor), normalize ↔
  denormalize rect round-trip (page-space ↔ 0-1 normalized),
  removal, and `restoreHighlights` reattaching from
  `HighlightRecord`. Edge cases: empty rects, nil anchor, page
  index out-of-bounds, zero-size page, negative page index.
- **Export pipeline** — `AnnotationExporter` builds a payload from
  mixed types (highlight / bookmark / note) with chapter mapping,
  dispatches to either JSON or Markdown formatter. Unicode + CJK
  + long notes preserved without truncation. This is what the
  user-visible "Export annotations" feature emits.
- **Import pipeline** — `VReaderAnnotationParser` + `AnnotationImporter`
  end-to-end. Parser handles valid payloads, rejects malformed /
  empty / truncated JSON, ignores future fields (forward compat),
  preserves Unicode / CJK / nil optional fields, handles ISO8601
  dates. Importer creates highlights / bookmarks / notes,
  deduplicates by ID (idempotent re-import), reports progress.
- **Bug #44 regression covered cross-reference** — the Add Note
  edit-menu action is pinned by `HighlightableTextViewTests`,
  already counted in feature #3's evidence file (this is the
  shared edit-menu surface; bug #44 added both Highlight + Add
  Note items).

What's deferred: the UI gesture path itself (long-press text → edit
menu → Add Note → modal → type → save → note appears in panel;
long-press existing note → context menu → Edit / Delete; export
share sheet; import file picker). All four require a real display.

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Notes are the third leg of a unified annotation system.** The
  same `profileKey`-keyed pipeline serves highlights, bookmarks,
  AND notes. The model tests pin this invariant — if Note diverged
  from Highlight on profileKey scheme, the panel's "All" tab would
  fragment.
- **Bug #44 was the entry-point fix for both feature #3 and #4.**
  Before the fix, UITextView's edit menu had Cut/Copy/Paste/Look
  Up/Translate/Share but no Highlight or Add Note. The
  `HighlightableTextView` subclass added both. This means feature
  #4 "Resolved by bug #44" is accurate but understates the
  pipeline weight — the surrounding annotation infrastructure
  (anchor types, exporter/importer, panel) grew substantially.
- **PDF annotations are PDFKit-native, not custom-rendered.** The
  bridge creates real `PDFAnnotation(.highlight)` instances on the
  `PDFPage`. This means PDF highlights round-trip through any
  PDF reader that respects the highlight annotation type — if the
  user exports their PDF, the highlights survive in third-party
  apps. EPUB / TXT use vreader's own anchor scheme.
- **Anchor SHA-256 hash stability is load-bearing for restore.**
  Feature #46 (materializing restore) relies on anchor hashes to
  reattach annotations after a backup/restore round-trip. The 5
  hash-stability tests in `AnnotationAnchor` would fail if the
  hash function ever drifted, which is exactly the regression
  signal restore needs.
- **Export pipeline supports JSON + Markdown today.** JSON is the
  round-trippable format (used by import); Markdown is for
  reading/sharing in a non-vreader context. Both formatters
  preserve Unicode + CJK + long-note content — the test suite
  pins all three explicitly.
- **`AnnotationsPanelViewTests` did not run via -only-testing.**
  The file is named `AnnotationsPanelViewTests.swift` but the
  suite is `@Suite("AnnotationsPanelTab")`. Swift Testing
  resolves -only-testing by suite/struct name; the struct
  declaration uses a different identifier. The 4 tests in that
  suite cover only the enum case set + system images + raw
  values — minor compared to the rest of the slice and not
  load-bearing for the verification. Filed as a low-priority
  test-naming inconsistency; not a feature-#4 bug.
- **Cross-format anchor hashing happens in one place.** Per
  `AnnotationAnchor`, the SHA-256 includes the case discriminator
  so different-type anchors with the same payload produce
  different hashes. Important — without this, an EPUB anchor with
  the same xpath as a TXT anchor's sourceUnitId would collide.
- **The xcodebuild summary line agrees with the actual count this
  time.** 130 reported / 130 actual. (Compare feature #3's evidence
  where the runner-bucket split made the summary line under-count.
  Annotations slice happens to fit cleanly in one Swift Testing
  bucket.)
