merged-main `41067e5` (v3.14.123). Closes the **long-press → Add Note modal**
slice that round-1 (2026-05-07) had deferred. Round-1 already pinned 130
unit/integration tests across 10 suites at the data layer.

## Observations

- **Edit menu surfaces correctly**: long-press over a word in the TXT
  reader's `HighlightableTextView` brings up the custom-action menu
  (vreader's `UITextViewDelegate.editMenuFor:rangeIn:suggestedActions:`
  hook) with the four expected actions: **Highlight | Add Note | Define
  | ▶**. The `▶` chevron indicates additional actions in a submenu (likely
  Translate + Look Up + Copy + Share — out of scope for #4 verify).
- **Modal gating correct**: empty text input keeps the Save button
  greyed/disabled — confirms vreader's empty-string gate (cannot create a
  note with no body).
- **Selection attribution rendered**: the modal correctly shows the
  long-pressed word `"how"` (in italic, above the text input) so the user
  knows which text the note is anchored to. This matches the feature
  spec's "annotations attached to selected text" requirement.
- **Cancel works**: dismissing via Cancel returns to the reader with the
  original text selection still rendered (blue highlight + selection
  handles). Selection isn't committed unless the user confirms via Save.
- **Save round-trip not driveable from this harness**: the same CU/iOS
  Simulator text-field quirk previously documented in feature #34 round-3
  (`dev-docs/verification/feature-34-20260508b.md`) blocks synthetic
  text input. Tried four input methods — all observable as no-op:
  1. `osascript "tell application System Events to keystroke <text>"`
  2. `pbcopy <text> && osascript "keystroke v using command down"`
  3. AX-API attempts to find/focus the text field returned nothing — the
     SwiftUI TextEditor (or whatever vreader uses for note input) doesn't
     surface as a `text area` / `text field` in the macOS-side Simulator
     AX tree.
  4. Simulator menu bar `Edit → Paste` click via AX (no effect on input).
  Hardware Keyboard is enabled in Simulator's I/O menu (✓ checkmark),
  Simulator is verified frontmost during the keystroke (status bar clock
  is bright, not dimmed). The input reaches the Simulator process but
  not the UIKit text-input layer — same external infra quirk, not a
  vreader defect.
- **No bugs filed** — all observable user-facing behavior is correct.
  The save-round-trip slice is structurally deferred, identical reason to
  features #34 (collection name input) and other text-input-driven
  features.
- **Annotations-panel inspection not pursued** — without a saved note,
  the panel would only show pre-existing annotations (if any). Reset cleared
  the library, so the panel would be empty on this fresh seed. Not
  informative for this round; would require a fixture pre-seeded with
  annotations OR a working save-round-trip.
