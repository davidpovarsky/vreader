## Summary

Round-3 verification of feature #4 (Add notes/annotations to text) on
HEAD (commit 6615c8b7, v3.21.8 build 285). Closes the **save round-trip
half** of round-2's deferred criterion 8 by demonstrating that
clipboard-paste via `mcp__computer-use__write_clipboard` + `key("cmd+v")`
reaches the iOS Simulator's UITextView input layer — bypassing the
synthetic-keystroke quirk round-2 hit. The note is persisted end-to-end
and visible in the Annotations panel's Notes tab. The remaining gap is
the visual-highlight half: the saved annotation does NOT render a
yellow background on the annotated text. This is the same root-cause
class as bug #160's chapter-mode visual paint gap (and bug #154's
search-tap highlight render gap); no new bug is filed — bug #160 is
already FIXED status with chapter-mode visual-paint device-verify
pending.

**Net change**: criterion 8(b) panel-entry path is now PASS; criterion
8(a) visual-highlight-paint stays deferred per bug #160.

## Observations

- **Clipboard-paste bypass works.** Round-2 (and feature #34 round-3 + feature #4
  round-2 evidence) reported that `osascript` synthetic keystrokes and Simulator
  menu Edit → Paste do not reach the iOS Simulator's UITextView input layer. This
  round used the MCP `computer-use` server's `write_clipboard` + the `Cmd+V` key
  combo dispatched through CU. The paste landed in the note input field
  byte-for-byte, the Save button gating flipped correctly, and the save round-trip
  completed. Same technique was also used earlier in today's verify-cron iteration
  for feature #5's search query input. Documenting it here for future iterations.
- **Save round-trip is sound at the data layer.** The Notes tab shows the saved
  note with the correct attribution ("Genoa"), body (verbatim paste), and
  timestamp. `AnnotationListViewModel` → `PersistenceActor.addAnnotation` →
  Notes tab render path is fully functional.
- **Visual-paint gap reproduces the bug #160 symptom.** After Save, the reader
  view shows no yellow background on "Genoa". The text remains plainly rendered
  (verified via post-selection-clear screenshot). war-and-peace.txt's CHAPTER I/II/III
  markers trigger chapter mode, exactly the surface bug #160's "Runtime symptom
  on war-and-peace.txt NOT yet gone" note covers. No new bug filed — cross-ref
  bug #160 instead.
- **Two annotation paths now both have device evidence for the chapter-mode
  visual-paint gap.** Bug #154 (search-tap path) and bug #160 (gesture-create
  path) both fail to render yellow background in chapter mode despite their
  respective code-complete fixes (PR #436 for #153 scroll + bug #154's wiring
  + feature #48 WI-1/2/3 chapter translation). The next bugfix-cron iteration
  with proper OSLog-based runtime diagnostics should pinpoint where in the
  render path the highlight is dropped.
