## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> **Live integration tests against the actual backend** counts as
> verification. Unit + protocol tests alone are NOT verification —
> they cover the audit gate (Gate 4), not the integration gate
> (Gate 5).

The 9 passing tests give Gate-4-class evidence on the
**tokenizer + range lookup + UI state update** chain. The
production code path exercised is:

- `TTSHighlightCoordinator.tokenizeSentences(in:)` —
  **production NLTokenizer**, not stubbed.
- `TTSHighlightCoordinator.sentenceRange(containing:)` — production
  binary search.
- `TTSHighlightCoordinator.updateHighlight(offset:)` — production
  state-update path; touches the **production `TextReaderUIState`**.
- `TTSHighlightCoordinator.clearHighlight()` — production cleanup
  path that respects `highlightIsTemporary` (so user-added
  highlights from feature #11/#17 don't get clobbered by TTS).

The test seam is at `TTSService` (mocked via
`MockSpeechSynthesizer`), so the AVSpeechSynthesizer →
`willSpeakRangeOfSpeechString` delegate → `currentOffsetUTF16`
chain is NOT exercised. That's the deferred slice — it requires a
live audio session, which can't be driven without a UI / device
context.

`result: partial` per SCHEMA. Tracker status stays `DONE` (already
there); no flip to `VERIFIED`.

## Observations

- **NLTokenizer + CJK is solid.** The full-width 。 (U+3002) is
  recognized as a sentence terminator out of the box; no custom
  tokenizer logic needed.
- **Binary search correctness** depends on tokenizer ranges being
  monotonic + non-overlapping, which is `NLTokenizer`'s contract for
  `.sentence` unit. Out-of-bounds offset returns nil rather than
  crashing or returning the wrong sentence.
- **`highlightIsTemporary` discipline is critical** — the
  coordinator's `clearHighlight` only nukes the range if it's
  flagged as temporary. This is what prevents a TTS playback from
  clobbering a user's permanent highlight (feature #11 EPUB / #17
  PDF / TXT/MD highlights). Two tests cover both branches.
- **`updateHighlightSetRange` covers feature #41 incidentally.** The
  same code path that sets `uiState.highlightRange` (#40 sentence
  highlight) also sets `uiState.scrollToOffset` (#41 auto-scroll).
  Both are asserted in one test. So the auto-scroll feature gets
  the same Gate-4 unit-level coverage from this iteration.
- **What we don't have unit-level coverage for:** the SwiftUI host
  wiring (`onChange(ttsService.currentOffsetUTF16)` → coordinator
  call), the AVSpeechSynthesizer delegate's offset reporting (which
  is platform-managed), and the actual rendering in
  `HighlightableTextView`. Those need the UI path.
