## Why partial, not pass

Same reasoning as `feature-40-20260507.md`: the coordinator path is
covered Gate-4-class through real `NLTokenizer` + production
coordinator + real `TextReaderUIState`, but the **scroll-actually-
happens-on-screen** slice requires a SwiftUI runtime + scroll
observation, which needs UI driving.

The 9 tests do cover the **scrollToOffset is set** half of the
auto-scroll contract. The **scroll actually moves the view** half
is the SwiftUI host's job (`TXTTextViewBridge` /
`MDReaderHost`), which observes `uiState.scrollToOffset` and calls
`UITableView.scrollToRow` / equivalent. Those host implementations
are not unit-tested for the scroll-on-change behavior.

`result: partial` per SCHEMA. Tracker status stays `DONE` (already
there); no flip to `VERIFIED`.

## Observations

- **Sentence-anchored scrolling is the right semantic.** TTS
  narrates word-by-word, but auto-scroll happens once per sentence
  boundary. Production line 66 (`scrollToOffset = range.location`)
  ensures the scroll target is stable while TTS reads through a
  sentence, only advancing when the next sentence starts. Without
  this, the reader would jitter horizontally as each word's
  position is reported.
- **The coordinator is a single source of truth for both #40 and
  #41.** Future changes to the auto-scroll semantic (e.g., scroll
  to sentence END instead of START) would update one line in one
  file. Tests for both features run from one suite. Architecture
  is clean.
- **What's intentionally out of scope:** EPUB / AZW3 / PDF auto-
  scroll. Those use web views (EPUB / Foliate) or PDFKit (PDF) for
  rendering; their scroll model isn't TextKit's `contentOffset`,
  so the same `scrollToOffset`-driven approach doesn't transfer.
  Future work would need format-specific auto-scroll bridges.
