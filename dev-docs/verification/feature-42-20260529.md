# Feature #42 WI-13 — Phase-1 acceptance pass (Readium EPUB engine, flag FORCED ON)

WI-13 is the **acceptance verification** that produces the evidence the human
reviews at the WI-14 default-ON gate (G2). The Readium engine is FORCED ON for
this run via the launch arg `-com.vreader.featureFlags.readiumEPUBEngine YES`
(lands in NSArgumentDomain, read by `FeatureFlags.init` from
`UserDefaults.standard`), so the EPUB dispatcher routes to `ReadiumEPUBHost`.

`commit_sha` is the branch base (`feat/feature-42-wi13-acceptance` = main,
v3.40.30 build 711); the verification ran against the working-tree Debug build of
this branch (the WI-13 corpus fixtures + acceptance suite are test/fixture-only —
no production reader code changed).

## How the verification ran (host-vs-runner)

`xcrun simctl openurl` is **unreachable from inside the XCUITest runner sandbox**
(NSPOSIX 61 — bug #242 / #1054; confirmed live this run: all bridge-dependent
XCUITest cases `XCTSkipUnless(bridgeReachable())`-skip). The authoritative CU-free
evidence is therefore the **HOST driver** (`xcrun simctl openurl` from the host,
where the bridge works), recorded in full below. The XCUITest acceptance suite
(`Feature42ReadiumAcceptance{Render,Features}VerificationTests`) carries the
runner-reachable regression net: the unconditional render-chrome gate runs always
(top-chrome back button — the Readium host renders NO `ReaderBottomChrome`, so the
legacy Display-button gate does not apply under this engine), and the
eval/snapshot assertions execute when the channel is reachable.

CU (computer-use) was not used — the iOS 26.4 virtual-display environment is
CU-free; all signals are read via the DebugBridge `eval` (JS on the active Readium
spine via `ReadiumNavigatorEvaluating`) and `snapshot` (semantic state JSON) seams.

## Observations

- **In-runner bridge is genuinely unreachable** in this environment (NSPOSIX 61),
  confirming bug #242/#1054 is structural — the host-driven layer is mandatory,
  not optional. The XCUITest suite is built so its one unconditional assertion
  (render chrome) carries the runner-side regression net and every bridge-dependent
  case skips cleanly with a reason pointing back here.
- **The Readium host renders no `ReaderBottomChrome`** — only the legacy per-format
  containers compose the bottom toolbar. The reader-loaded XCUITest signal under
  Readium is the shared `ReaderTopChrome` back button. (Recorded so a future agent
  doesn't mistake the absent Display button for a regression.)
- **`navigate?spine=N` is a legacy-engine-only verification path** — it posts
  `.debugBridgeNavigateCommand`, observed only by `EPUBReaderContainerView`. The
  Readium host observes the downstream `.readerNavigateToLocator`, so navigation
  (criteria 2/5/9) is driven through search-nav, which exercises the same
  `go(to:)` path a TOC jump uses. A follow-up could add a DEBUG-gated
  `.debugBridgeNavigateCommand` observer to the Readium host for direct
  spine-jump verification (out of WI-13 scope — that touches production reader code).
- Position restore is real (chapter2 round-trips) but the snapshot `position`
  STRING field reads `null` under Readium; the rendered `location.href` is the
  authoritative restore signal. A snapshot `position` projection for the Readium
  host would tighten future CU-free position assertions.
- CJK vertical-rl took effect (computed `writing-mode: vertical-rl` on ch2) — a
  good sign Readium honors per-document inline `<style>` writing-mode.
