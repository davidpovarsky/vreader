# Feature #42 WI-10b — Readium TTS speaking-position follow (slice verification)

This is a **slice verification** for WI-10b (behavioral WI), recorded per
`.claude/rules/47-feature-workflow.md` Gate 5a. It is NOT the final-WI
acceptance pass that flips feature #42 to `VERIFIED` (that is a later WI). The
`commit_sha` is the branch base (`feat/feature-42-wi10b-tts-follow`); the
verification ran against the working-tree build of this branch's changes.

## What was verified
As TTS read-aloud speaks an EPUB under the Readium engine, the navigator
auto-advances so the spoken position is followed. Verified CU-free via the
DebugBridge eval seam + OSLog `log stream` (the CU virtual display is
unavailable in this environment — `mcp__computer-use__screenshot` returns "CU
display unavailable" — so the eval-seam + diagnostic-log path is the CU-free
evidence, per the rAF/virtual-display precedent in MEMORY).

Cross-spine follow (offset crossing into chapter2 at ~12141) uses the identical
code path with a different `href`; the offset→href+fraction mapping for all 4
spines is unit-tested (`ReadiumTTSFollowMapperTests`) and the intra-spine +
boundary cases (separator gap, past-end clamp) are covered there. The natural
TTS rate (~80 UTF-16 / 2.5s) makes the live chapter-1→2 crossing ~10 min of
speech; the live evidence captured the same navigate path firing within
chapter 1 at the throttle threshold, and the parser-walk builder test confirms
the 4-spine table that drives the cross-spine resolution.

## Why scroll mode `window.scrollY` is NOT the signal
Readium scroll mode scrolls an OUTER UIScrollView (UIKit), not the spine
WebView's `window` — confirmed by the eval probe (`bodyH=8146 >> innerH=750`
yet `window.scrollY=0`). The follow-nav OSLog event (offset → href+fraction) is
therefore the authoritative CU-free signal that the navigate fired; spine-`href`
change is the observable boundary signal.
