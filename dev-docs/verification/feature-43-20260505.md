## Observations

### EPUB extractor failure (root cause confirmed)

The EPUB declares its cover via:

```xml
<meta name="cover" content="cover"/>
...
<item href="OEBPS/cover.jpg" id="cover" media-type="image/jpeg"/>
```

But the OPF lives at `OEBPS/content.opf`, so `MetadataExtractor.resolveArchivePath(coverHref: "OEBPS/cover.jpg", opfDirPath: "OEBPS")` joins to `"OEBPS/OEBPS/cover.jpg"` — which does not exist in the archive. The actual cover is at `OEBPS/Images/cover.jpg`.

This is a malformed OPF (the publisher set the `href` as if it were absolute from the archive root, when EPUB spec requires it to be relative to the OPF directory). Common publisher mistake.

Robust EPUB readers handle this via fallbacks. Suggested order:
1. Try the spec-compliant resolved path first (current behavior).
2. If 404, try the bare filename (search any archive entry whose basename matches).
3. If still 404, scan the manifest for any `image/*` item whose `id` contains `cover` and whose `href` resolves to an actual archive entry.
4. Finally, scan the archive root for `cover.{jpg,jpeg,png}` patterns.

`MetadataExtractor.swift:92–108` currently only does step 1 — returns nil on first miss.

### AZW3 case (inconclusive)

The bytes are exactly where `MOBICoverExtractor.swift` says they should be (record 135 = `firstImageRecord` + `EXTH 201 value 0`, with valid JPEG signature). I could not reproduce the failure deterministically without re-importing the book — and the import path is gated through WebDAV restore (`source: "restore"` in the SwiftData column), which I'd need DebugBridge URL access to drive cleanly (currently blocked by bug #121).

Three possibilities, listed by my prior probability:
1. Extraction did succeed at restore time, but the saved JPEG was later deleted. The CustomCovers directory mtime is 2026-05-04 19:07 local — between the EPUB import (17:46) and the AZW3 import (21:49). Something modified the directory in between. Could be: extractor saved EPUB cover (likely failed, see above), saved AZW3 cover, then a subsequent app op cleared one or both. Improbable without a code path that deletes covers.
2. Extraction silently failed for a reason not visible from static analysis — e.g., DRM detection in the EXTH header, an exception in `UIImage(data:)`, a sandbox-write permission issue. The current code uses `try?` on `saveCover` which swallows everything.
3. The AZW3 was imported by an older app version that lacked AZW3 cover extraction, and `if !CustomCoverStore.hasCover(for: fingerprintKey)` short-circuited it on this newer build because... wait, the local store is empty so `hasCover` would be false, so the extraction would run. So this hypothesis doesn't hold.

The honest call: file the EPUB bug now (root cause clear); flag the AZW3 case in the same ticket as a related-but-unproven failure pending a deterministic re-import.

### Vestigial `Book.coverImagePath` column

The SwiftData `Book.coverImagePath` column is always NULL because `EPUBMetadataExtractor.extractMetadata` and `AZW3MetadataExtractor.extractMetadata` both return `coverImagePath: nil` unconditionally. The actual cover storage is `CustomCoverStore` keyed by fingerprint. Either remove the column in a future SchemaV5 migration, or wire it (mildly redundant since the file path is `CustomCoverStore.coverPath(for:)`).

## Recommendation

File one bug — call it #122: "EPUB cover extraction fails on books with redundant-prefix `href` in `<meta name=\"cover\">` manifest entry (extractor needs fallback to bare-filename / manifest-id scan)."

Re-promote to DONE only after bug #122 ships and a re-run of this verification observes the cover from "道诡异仙" landing in `Library/Application Support/CustomCovers/`.
