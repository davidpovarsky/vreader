# Feature #43 — Cover image extraction — verification round 2 (post bug #122 fix)

This is the second verification of feature #43. Round 1 (`feature-43-20260505.md`) was `fail` — EPUB cover extraction returned nil for the "道诡异仙" repro shape (redundant-prefix `href`), and AZW3 cover for "被讨厌的勇气-1" was missing with inconclusive root cause. Both gaps were tracked: EPUB → bug #122 (filed, FIXED in PR #248 v3.13.12), AZW3 → root cause unknown, blocked by bug #121 at the time (since fixed).

This round focuses on the EPUB half: bug #122's fallback chain (spec → OPF-tree-ranked basename → archive-root) is now in `EPUBMetadataExtractor.coverPathCandidates`. The synthetic-EPUB high-fidelity integration test reproduces the "道诡异仙" OPF shape and exercises `extractCoverImage` through the real subsystem boundaries (real `ZIPReader`, real `EPUBParser`, real `UIImage(data:)` decoder, real candidate-probe loop) with no stubs.

The AZW3 half remains blocked: deterministic re-import of "被讨厌的勇气-1" still requires a working `vreader-debug://seed?fixture=<azw3>` flow, which fails today because of bug #124 (`Bundle.url` doesn't search the `DebugFixtures/` subdirectory). Bug #124 is in progress on `fix/issue-252-debugbridge-seed-bundle-subdirectory`. Once #124 lands, run round 3 to address the AZW3 path.

## Observations

- **Bug #122's fallback chain works exactly as designed**: spec-compliant resolved path tried first (preserves no-regression for normal EPUBs), then OPF-tree-ranked basename match (the "道诡异仙" path), then archive-root canonical cover.
- **Codex's earlier audit fix (basename ranking) is exercised**: `coverCandidates_basenameRankingPrefersOPFDirTree` confirms inside-OPF entries beat outside-OPF entries when basename collides. Without this ranking, a stray `backup/cover.jpg` could outrank the real cover.
- **The synthetic-EPUB test drives real subsystem boundaries**: the test builds a real ZIP via `buildZIP()` helper, opens it through real `ZIPReader` (memory-mapped), parses it through real `EPUBParser.parseOPF`, and decodes JPEG bytes through real `UIImage(data:)`. This is verification-exception qualifying.
- **The AZW3 path remains blocked** transitively. Until DebugBridge `seed` works (bug #124 fix), there's no deterministic way to drive an AZW3 import-with-cover flow on a clean simulator. The "被讨厌的勇气-1" book itself is not in the repo's debug fixtures (the same Debug-bundle weight + copyright reasons that apply to "道诡异仙" apply here).
- **No regressions in other cover paths**: spec-compliant EPUB, percent-encoded paths, relative `..` paths all still resolve correctly. The basename and archive-root fallbacks only fire when the spec path misses.
- Pre-existing test failures (TXTChapterContentLoader / FoliateView / etc., 19 total) are unchanged from main. None touch cover-extraction code.

## Why partial, not pass

- **EPUB acceptance criteria 1–5, 7**: PASS via deterministic high-fidelity tests at real subsystem boundaries.
- **AZW3 acceptance criterion 6**: BLOCKED — not failing, just untested deterministically. The earlier round 1 evidence file (`feature-43-20260505.md`) describes the AZW3 case as "root cause inconclusive without deterministic re-import." That's still true. Until bug #124 lands and a working `seed` lets the verification harness re-import the AZW3 fixture deterministically, this criterion can't be either pass or fail.

Per `dev-docs/verification/SCHEMA.md`, `partial` is the right result-class when "the feature works for some criteria but not others, and the gaps are tracked as separate work." The AZW3 gap is now linked to bug #124's resolution.
