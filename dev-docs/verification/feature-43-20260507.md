## Observations

- **AZW3 cover extraction is robust on Project Gutenberg's
  `kindle.noimages` Mobipocket variant** — the EXTH 201 cover index
  resolution and target-record JPEG extraction both work cleanly.
- **The cover bytes are honest.** 341 × 512 px is a typical Kindle
  thumbnail aspect ratio; the JFIF + Exif preamble is standard JPEG
  (decodable by Core Graphics and any image viewer).
- **`coverImagePath` is empty in the SwiftData record.** Initially
  this looked like a missing-write bug, but per
  `CustomCoverStore.coverURL(for: fingerprintKey)`, the path is
  derived from the fingerprint key implicitly — every Library card
  looks up `CustomCovers/<key>.jpg` directly. So an empty
  `coverImagePath` field is by-design when the cover is in the
  CustomCovers directory. (`coverImagePath` is reserved for
  externally-pathed covers, e.g. for future reference-only schemes.)
- **What this round did NOT verify, and isn't in scope for #43:**
  metadata title / author extraction. The mini-azw3 import shows
  title = "mini-azw3" (filename) and author = "" because
  `AZW3MetadataExtractor.extractMetadata` returns
  `.fromFilename(fileURL)` — the comment at line 267 is explicit:
  "EXTH title parsing is out of scope". This is a separate concern
  from feature #43 (which is about *covers*); the metadata gap is
  filed as **bug GH #340** for the bug-fix cron to pick up.
- **Round-1 mystery now plausibly resolved.** Round 1 reported "AZW3
  cover for '被讨厌的勇气-1' missing" without root cause. With the
  cover-extraction path now confirmed correct on a different
  Mobipocket dialect, the most likely explanation for the round-1
  failure is an EXTH-201-record format quirk specific to that book
  (different cover encoding, missing EXTH record, or DRM-stripped
  variant). Bug #122's cover-fallback chain was added precisely to
  handle that class of issue. If the original failing book becomes
  available, a follow-up round-4 verification can confirm.
