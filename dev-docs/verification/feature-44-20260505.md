## Observations

The `DebugBridge.plist` file's own header comment lays out the intended design:

> Wired via per-configuration INFOPLIST_FILE: vreader target Debug config references this file; Release config leaves INFOPLIST_FILE unset so the vreader-debug URL scheme is absent from shipped builds.

But that wiring was never landed. `project.yml` at line 60–87 sets one global `info: path: vreader/SupportingFiles/Info.plist` with hard-coded Debug-and-Release-shared properties. There is no per-configuration `INFOPLIST_FILE`, and no build phase merges `DebugBridge.plist` into the main Info.plist.

The result is the worst-of-both-worlds: the DebugBridge router/snapshot/fixture machinery is fully implemented in Swift (93 unit tests pass) and the URL scheme partial-plist is checked in, **but no installed build actually accepts `vreader-debug://` URLs**. Every "device verification" iteration of /loop has been forced to drive the UI manually, and the verification harness (`vreaderUITests/Verification/`) planned in feature #45 cannot actually run against this build.

This blocks feature #45 outright, and silently undermines the tooling premise of every "Needs device verification" backlog item that names DebugBridge as the harness. It also explains why iteration 1 of /loop hit a wall on /loop's first attempt to use `simctl openurl` for Foliate diagnostics — the URL never reached the app.

**Two adjacent observations**:
1. `DebugBridge.plist` is a Resources-phase entry (pbxproj line 3123), so it ships in BOTH Debug and Release installed bundles as a sibling file. iOS never reads `CFBundleURLTypes` from sibling plists, so this is a no-op leak in Release — but `verify-release-no-debugbridge.sh` would still flag it via the `*DebugBridge*` filename check (line 90). If anyone runs the Release gate on a fresh Release build, it would fail. Worth confirming as a follow-up.
2. The 93 unit tests for `DebugBridgeRouter`, `DebugStateSnapshotter`, etc. test the *parsing and snapshotting logic in isolation*. None of them test that the URL scheme is registered in the bundle (which is a build-system concern, not a Swift concern). So this gap was always going to slip past the test suite.

## Recommendation

Filed as bug #121: "DebugBridge URL scheme not registered in installed builds — `DebugBridge.plist` is orphaned."

Status of feature #44 is **demoted DONE → IN PROGRESS** per SCHEMA `result: fail` semantics. The Swift implementation, 93 unit tests, and subsystem doc are all in place — the blocking gap is a 5–10 line `project.yml` change to merge `DebugBridge.plist` for the Debug configuration. Until bug #121 lands, **no `simctl openurl vreader-debug://` invocation will work** and feature #45's verification harness has no foundation to build on. Re-promote to DONE only after bug #121 ships and a re-run of this verification observes a successful `simctl openurl ... snapshot` writing a valid `state.json` to the app container.
