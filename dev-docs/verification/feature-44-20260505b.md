## Observations

The URL scheme registration fix in bug #121 is solid and verified:
- `verify-debug-has-debugbridge.sh` PASS
- `verify-release-no-debugbridge.sh` PASS
- `simctl openurl` returns exit 0 (was: `LSApplicationWorkspaceErrorDomain code=115`)
- Debug Info.plist is correctly populated; Release Info.plist is correctly empty

But the **dispatch from `.onOpenURL` to `debugBridge.handle(url)` isn't firing**. The URL is accepted by iOS (registered scheme, no LaunchServices error) but the SwiftUI modifier doesn't observably fire its block. Filed as bug #123 / GH #243.

Suspected root cause (UNVERIFIED hypothesis — needs investigation during the bug #123 fix): the `.onOpenURL` modifier in `vreader/App/VReaderApp.swift:225` is attached inside the conditional `if let modelContainer, let contentView` at line 216. URLs delivered before those let-bindings resolve might be silently dropped if SwiftUI doesn't queue across modifier-attachment transitions. Alternative possibilities to verify:
- `debugBridge` is unexpectedly nil at handler-time (despite synchronous init in `init()` per the comment at line 47).
- `URL.scheme == DebugCommand.scheme` guard incorrectly rejects.
- A different Scene/WindowGroup root is the URL recipient and the handler block isn't reached.

This means feature #44 stays at `IN PROGRESS`. The implementation work (router parser, snapshotter, fixture catalog, error codes, release-gate script, 93 unit tests, doc) is all in place and shipped, but the entry-point routing is broken. Re-promote to DONE only after bug #123 fix lands and a re-run observes:
- `simctl openurl vreader-debug://reset` actually wipes the library
- `simctl openurl vreader-debug://snapshot?dest=state.json` actually writes the JSON file
- Log lines fire on `com.vreader.app` subsystem during URL invocation
