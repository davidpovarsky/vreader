## Observations

- **Bug #121 + bug #123 fixes are correctly resolved.** Both gates green via `scripts/verify-debug-has-debugbridge.sh`. The infrastructure that lets DebugBridge URLs reach the app is sound.
- **Bug #124 (seed) is a one-line oversight in `RealDebugBridgeContext.swift:92-97`.** The `subdirectory: "DebugFixtures"` argument is missing from the `Bundle.url(...)` call. The fixture file IS in the bundle (verified via direct `ls`); it's just not at the path the lookup expects. The fix is mechanical and could land in one PR.
- **Bug #125 (settle) is a doc/code mismatch.** The header doc claims "a hung probe still produces the sentinel" but the implementation only writes a sentinel when the probe path is actually reached — `noActiveReader` short-circuits before that. Mirroring eval's noActiveReader pattern (sentinel + log) is the correct fix.
- **The four working commands (reset, theme, snapshot, eval-error-path) are cleanly observable.** Each produces a log line, a file in `Caches/DebugBridge/`, or a state field that can be programmatically inspected. This validates the bridge's *design*; the remaining bugs are implementation gaps in two specific handlers.
- **Once #124 and #125 land**, this verification can be re-run to exercise:
  - `seed` → log line + ZBOOK increment + ImportedBooks/ file
  - `open` → snapshot reports `currentBookId` + `format`
  - `settle` (no-active-reader path) → `ready-<token>.json` with `error: "no active reader"`
  - `settle` (with-reader path) → `ready-<token>.json` with empty error after relocate fires
  - `eval` (with EPUB open) → `eval-foliate.json` with `result` field carrying JS output
- The `partial` items in snapshot's `partial` array (`["selection", "currentBookId", "format", "position"]`) are expected — they're documented as "not yet wired" fields that the snapshot reports as null until a probe registers.
- Pre-existing test failures (TXTChapterContentLoader / FoliateView / etc., 19 total) are unchanged from main. None touch DebugBridge code.

## Why partial, not fail

- 4 of 7 commands work end-to-end (reset, theme, snapshot, eval-error-path).
- 1 command works for its error path but the happy path is blocked by bug #124 (eval).
- 2 commands fail outright with their own newly-filed bugs (seed → bug #124, settle no-active-reader → bug #125).
- `open` was not exercised end-to-end because `seed` is broken — it is transitively blocked by bug #124 rather than carrying its own newly-filed bug. A future round 4 (after #124 lands) will validate `open` directly.

Per `dev-docs/verification/SCHEMA.md`, `partial` is the right result-class when "the feature works for some criteria but not others, and the gaps are tracked as separate work." Both gaps are filed as bugs with clear fix scopes.
