## Summary

Round 10 of feature #44 (DebugBridge). Adds another operational slice to acceptance criterion (a) â€” eval coverage now includes EPUB theme-CSS injection state. **Status unchanged** (still gated on bug #141 settle for VERIFIED); this is additive evidence that the eval surface continues to expose more reader internals than originally specified.

**What this round verifies**:

EPUB theme rendering via `getComputedStyle` round-trips correctly for both dark and light themes:

- DARK (default): body `color: rgb(234, 234, 237)` (light text), `backgroundColor: rgb(28, 28, 30)` (dark gray bg).
- LIGHT (after plist write of `readerTheme = "light"` + relaunch): body `color: rgb(25, 25, 25)` (near-black), `backgroundColor: rgb(255, 255, 255)` (pure white).
- Both modes inject the `vreader-theme` style element (`document.getElementById('vreader-theme') !== null` returns true).

The darkâ†’light contrast confirms vreader's theme CSS is real-render-correlated, not just a stored preference. Bug #126's eval surface is the verification primitive enabling this slice.

**Known gap exposed by this round**: `vreader-debug://theme?mode=light` writes to `UserDefaults` but does NOT propagate into the live `ReaderSettingsStore` of an open EPUB. `RealDebugBridgeContext.theme(mode:fontSize:)` instantiates a fresh `ReaderSettingsStore(defaults:)` rather than addressing the active reader's `@State`-owned store. Result: a running EPUB doesn't re-theme via the bridge command. Workaround for verification: kill the app, edit the plist directly, relaunch.

The bridge-driven re-theming is a separate gap â€” could be filed as a follow-up bug but is narrow (verification-flow only; the user-facing settings panel UI does work because it mutates the live store directly).
