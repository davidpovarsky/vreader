## Observations

- **DebugReaderRegistry binding works for AZW3.** The keyed
  `setActiveFoliateWebView(_:for:)` + `foliateWebView(for:)` pair from PR
  #311 successfully resolves the active Foliate `WKWebView` for the open
  AZW3 book. JS round-trips return real DOM-state JSON (not the
  stub-throw-`unsupportedBridge` of the pre-fix path).
- **Foliate bridge surface is named `FoliateHost` + `readerAPI`** in
  AZW3 path — these are the named WKScriptMessageHandler / IIFE
  bindings used by `FoliateSpikeView` / `FoliateViewCoordinator`. The
  presence of both keys in `window` confirms the JS context is the live
  app's WebView, not a generic loadHTMLString stub.
- **Outer document has no `<p>` and no custom elements** because
  Foliate-js paginated readers render chapter content inside per-chapter
  iframes (`<foliate-paginator>` proxies); the outer document is the
  host shell. `docHTMLLen: 285987` (~286KB) shows substantial layout
  scaffolding even in the host doc — fixture loaded successfully.
- **`__foliate` is undefined on the global** in AZW3 mode. This is
  expected: the EPUB native-mode bridge mounts `window.__foliate` from
  `foliate-bridge.js` because it's a custom WKWebView injection; the
  AZW3 path uses Foliate's own runtime which exposes `FoliateHost`/
  `readerAPI` instead. Both surfaces work; they're distinct namespaces
  for distinct WKWebView injection paths.
- **Snapshot post-open is not stubbed.** `currentBookId` exactly matches
  the opened key; `format=azw3` matches the seed format; `partial`
  excludes both `bookId` and `format` (only `selection` and `position`
  remain partial — those are sub-states of an active reader, not stubs).
- **No regressions across the 25-PR session.** Mini-azw3 fixture seeds
  cleanly, opens cleanly, eval bridge round-trips cleanly. App version
  is `v3.14.121` (the `bug #160 partial fix` ship), proving the
  Foliate-eval surface survives all session-day churn.
