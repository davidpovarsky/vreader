## Why we're flipping to VERIFIED rather than re-running every criterion live

- AC #1, #2, #3, #4, #6 were already pass on the initial run — the regression risk is unchanged.
- AC #5 — the picker → preplant → row insert chain — was the partial-pass on the first run because of bug #114. The fix is unit-tested and the integration with persistence + library refresh is covered by bug #116's verification (which exercised the same code path).
- AC #7 — the gating criterion that blocked the original VERIFIED — is the entirety of bug #115's verification (10/10 acceptance criteria observed end-to-end with diagnostic logging from a real iPhone 17 Pro Sim against rclone over Tailscale).
- AC #8 — Wi-Fi vs cellular gating — sim doesn't model cellular state. The policy gate is unit-tested against `WebDAVNetworkPolicy` directly.
- AC #9 was the bug #113 fix — regression test pins it.

A full live re-run of every criterion would re-test paths that haven't changed since the initial pass and add no signal. The bug-by-bug verification chain is what closes the gap.

## Observations

- **The verification framework worked.** The PreToolUse hook (`check_terminal_status_evidence.sh`) blocked the first VERIFIED flip until evidence existed. The Stop hook (`check_unfinished_verification.sh`) surfaced the gap. The 5 bugs caught here would have shipped to users if the original "tests pass = ready to mark VERIFIED" mental model had stayed in place.
- **Bug #115's diagnostic logging is the model for future bug-fix branches.** The first commit on the bug #115 branch (99facee) added per-guard diagnostic logging in LibraryView.onReceive, which immediately proved the observer chain was working and pointed at LazyDownloadCoordinator.didFinishDownload as the actual gap. Without those logs, the bug would have stayed mis-diagnosed as "tap doesn't fire."
- **The picker-side and lazy-download-side paths now share the same library-refresh notification (`bookFileStateDidChange`).** That convergence is good — one observer covers both the preplant flow and the lazy-promote flow.
