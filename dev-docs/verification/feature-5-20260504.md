## Observations

The dismiss feature is comprehensively covered by `vreaderTests/Views/Reader/SearchHighlightDismissTests.swift` (25 tests) — these wire the real `HighlightableTextView` and real `HighlightingLayoutManager` and exercise every dismiss trigger (tap, scroll, new search) against real subsystem boundaries, plus persistence invariants and idempotency. Per `SCHEMA.md` line 78–80, those tests cover the audit gate (Gate 4), NOT the integration gate (Gate 5).

The blocker for visual end-to-end confirmation is **search-tap navigation viewport positioning**: when a user taps a search result, the reader navigates to the chapter and sets `searchHighlightRanges`, but the resulting content offset places the matched text past the visible viewport. This is a separate UX concern from the dismiss feature itself — the dismiss logic still fires correctly per the unit tests; the visibility issue is about the navigation step, not the dismiss step.

This was likely OK during initial WI-003 review because contemporary tests used short documents where any chapter offset put the match in the viewport. With a longer real book ("被讨厌的勇气" or similar long-chapter TXTs), the navigation lands lower than the match.

**Follow-up**: file a separate bug for "search-result-tap should center the matched text in the viewport, not just navigate to the chapter" so feature #5's verification can complete cleanly in a future pass. Once the navigation lands the match in view, all 8 acceptance criteria become straightforwardly verifiable end-to-end.
