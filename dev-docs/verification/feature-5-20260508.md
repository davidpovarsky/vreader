## CU-driven repro on iPhone 17 Pro Sim (iOS 26.4) at v3.14.83

1. `xcrun simctl launch "iPhone 17 Pro" com.vreader.app --uitesting --seed-position-test`
2. Library shows "Position Test Book" (TXT, 100 paragraphs).
3. Tap book → TXT reader opens at paragraph 1.
4. Tap reader-chrome search button → search panel slides up.
5. Type `100` (single digits via per-char `key` action).
6. Search returns 1 result: snippet "Paragraph 100: This is test content for verifying rea... / Section 101".
7. Tap the result row.
8. **Observed**: reader scrolls to ~95% of document; visible region shows "reading and restore the scroll position when reopened." (the END of paragraph 100). The line "Paragraph 100: ..." is OFF-SCREEN above the viewport. No yellow highlight visible anywhere on screen.
9. Wait 3 seconds — the documented `HighlightingLayoutManager` auto-clear timer fires.
10. Scroll up to find "Paragraph 100" → it's now visible (paragraphs 97–100 + progress bar at 95%) but no highlight.

## Acceptance criteria

| Criterion | Observed | Verdict |
|---|---|---|
| Search-result tap navigates to matched paragraph | reader scrolled to ~95% (correct paragraph area) | pass |
| Yellow highlight (systemYellow @ 0.4 alpha) renders on the matched text | rendering pipeline works at the data layer (25 `SearchHighlightDismissTests`) but the highlight is OFF-SCREEN at the moment of navigation; user never sees it | **fail** |
| Highlight auto-clears after 3 seconds | timer fires correctly per the existing test suite | pass (timing) |
| Highlight visible BEFORE auto-clear | **NO** — by the time user scrolls up to the matched text, the 3s timer has already fired | **fail** |
| Search-result-tap dismiss highlight on next user action | not exercised this round (the highlight was never visible to begin with) | n/a (blocked) |

**Two failing criteria** — the symptom is the navigation strategy, not the highlight rendering itself. The highlight code is correct; the scroll anchor is wrong.

## Bug filed

**Bug #153 / GH #433** — "Search-result-tap navigation places matched text above viewport; highlight invisible before 3s auto-clear" (Reader/*, severity:medium).

Suspect surface: `TXTTextViewBridge.scrollToGlobalOffset` /
`scrollRangeToVisible` may use the START of the matched range as
the scroll anchor but TextKit rounds to BOTTOM of visible region,
pushing the highlight position above viewport. Files to investigate:

- `vreader/Views/Reader/TXTTextViewBridge.swift`
- `vreader/Views/Reader/TXTTextViewBridgeCoordinator.swift`
- `vreader/Views/Reader/HighlightingLayoutManager.swift`
- `vreader/Services/Reader/TextHighlightRenderer.swift`

## Observations

- **Bug #153 is a separate root cause from bug #99** (GH #26 — search highlight missing in some TXT). Bug #99 lists three suspect causes including "if scroll hasn't settled, range may be outside visible area" — bug #153 is essentially that hypothesis confirmed for the UTF-8 / non-chunked-reader case. They could share a fix path (centering the scroll anchor) but should remain tracked separately because #99 also covers non-UTF-8 / chunked-reader rendering failures that bug #153's fix won't touch.
- **CU `type` action vs `key` action** — confirmed again that bulk `type` doesn't reach iOS Simulator text fields; per-character `key` actions do. (Also documented in bug #152 / GH #426 round-3 evidence.) Auto-correct on the search field also bit me first attempt: typing "Paragraph 50" produced "Pragraph 50" (autocorrect dropped the 'a'), which returned no results. Switched to digit-only "100" to bypass autocorrect.
- **Result: `fail`** — feature #5's deferred slice was always a known gap, and this round upgrades it from "could not confirm" to "confirmed broken." Status of feature #5 row in `docs/features.md` should NOT be flipped — bug #153 needs to be FIXED before VERIFIED is reached. Row stays `DONE` with a round-2 reference + bug #153 link.
- **No regressions to existing pass criteria** — search panel UX itself works fine, search index returns correct results, highlight rendering pipeline is sound (per data-layer tests). The bug is narrowly in scroll-anchor positioning.
