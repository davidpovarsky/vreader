## Observations

Bug #154 (FIXED status, fix tied to feature #48 WI-3 / commit a472a6d shipped in
v3.20.0 build 276) was supposed to close the chapter-mode highlight gap that
round-2 (2026-05-08) deferred. Round-3 on v3.21.7 confirms the highlight still
does not render after a search-result tap, in chapter mode, on the Position Test
Book fixture — the exact same surface bug #154 covered.

Notably the scroll behavior is correct: matched paragraph 100 lands a clean ~25%
from the viewport top with paragraph 99's tail above it (vs. the round-2 symptom
where the match was OFF-screen). The bug #153 scroll-anchor fix held. The
regression is isolated to the temporary highlight render path itself.

Two possible failure modes (next bugfix iteration to diagnose):

1. The chapter-mode display path (`chapterReaderContent` ~line 496) still
   hardcodes `highlightRange: nil` — feature #48 WI-3 added
   `LocatorFactory.txtChapterRange` + `TXTReaderContainerView.makeLocatorForTXT`
   but the search-tap event may still feed an unmapped global offset to the
   chapter renderer.
2. The new `LocatorFactory.txtChapterRange` produces a chapter-local
   `NSRange`, but `chapterReaderContent` (or the chunked TXT view used at this
   chapter size) reads from a different `uiState` field than the one the
   search-tap handler writes to.

The artifacts capture the immediately-after-tap moment (well within the 3s
auto-clear window), so this is not a timing artifact. The result is `fail` —
Feature #5 must stay at `DONE` and bug #154 must be reopened.

No fix attempted this iteration — verify-cron scope is verification only.
