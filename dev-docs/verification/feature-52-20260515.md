# Feature #52 — Multiple WebDAV server profiles with active-server switching

WI-6 round-2 final acceptance verification. Round-1 (2026-05-14) verified
UI shape but deferred the backup roundtrip slice because CU `type` couldn't
reach iOS Simulator text fields. Round-2 unblock path (clipboard fast-path:
`clipboardWrite` grant + cmd+v) works perfectly — text input now reaches
all SwiftUI TextFields, enabling end-to-end acceptance verification.

## Observations

### What worked

**Round-2 unblock — clipboard fast-path.** CU's `type` action doesn't
reach iOS Simulator text fields, but `write_clipboard` + `cmd+v` does.
Confirmed across all 4 fields in both Add and Edit sheets (Name, Server
URL, Username, Password). The grant requires `clipboardWrite: true` in
`request_access`.

**Test Connection against real rclone WebDAV.** Tapping Test Connection
after entering Server A's creds returned **"Connected — the WebDAV server
responded successfully."** in green. This validates the end-to-end network
path: `WebDAVTransport` → URLSession → BasicAuth → rclone PROPFIND
handler. Screenshot `feature-52-wi6-r2-01-test-connection-success-server-a-20260515.png`.

**Backup roundtrip — file inspection on the host filesystem.** Because
the test rclone servers write to mounted directories (`/tmp/vreader-test-webdav-{a,b}`),
the host can directly inspect what landed where. Each backup produced
both a `*.vreader.zip` AND a per-book blob (`books/epub/<sha>_<size>.epub`),
confirming `BackupDataCollector` is correctly walking the library and
uploading both the backup metadata zip and the actual book files.

**Active-only behavior.** Distinct timestamp + UUID-suffix in each
zip name (`2026-05-14T20-10-23Z_d6c6c5ce` vs `2026-05-14T20-12-21Z_d0348cab`)
proves these are genuinely separate uploads to separate servers, not a
single zip that both servers received.

**Confirmation alert for active-delete.** Swiping the active row →
red Delete chip → tap → alert with title "Delete active server?" and
body `"Server B (test)" is currently the active backup server. Deleting it leaves no active server until you switch to another or add one.`
Matches the WI-4a row narrative exactly.

**Backup-disabled state.** With no active profile, the WebDAV Backup
screen renders only the Servers nav-link + Network policy + footnote
"Add a WebDAV server above to enable backup." — Back Up Now button +
Available Backups section both correctly hidden.

### Bug discovered (per scope guardrail: filed, NOT fixed this iteration)

**UI refresh bug** — the `WebDAVServerProfileListView` shows the empty-state
("No WebDAV Servers" + Add Server CTA) AFTER any successful mutation
(Save, Add, setActive, delete) until the view is dismissed and re-presented.
Underlying UserDefaults persistence is correct (verified by reading the
plist directly), but the SwiftUI list view doesn't re-render on
`.webdavProfilesDidChange` notifications. Reproduction observed across 4
distinct mutation paths this iteration. Filed as separate bug row (see
docs/bugs.md addition + GH issue).
