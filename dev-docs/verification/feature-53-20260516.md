## Observations

1. **Bug #199 fix is correct conceptually but its device verification is currently blocked.** The Bug #199 fix (just shipped at v3.23.1) wires `UIKitHighlightActionPresenter` into `FoliateSpikeView` so tap-on-existing-highlight produces the inline menu in AZW3. But device verification requires an existing highlight, which the AZW3 UI cannot create — the selection event isn't bridged in `FoliateSpikeView.Coordinator.handleMessage` (only `bridge-ready`, `book-ready`, `layout-ready`, `tap`, `annotation-show`, `error`; no `case "selection":`). Filed as Bug #201 / GH #739.

2. **TXT WI-2/WI-2b appears to be regressed.** Tap on a successfully-painted, persistence-confirmed highlight does NOT surface the inline edit/delete menu — chrome-toggle wins the tap. Two attempts, identical behavior. The features.md row for Feature #53 claims TXT/MD/EPUB/PDF are "feature complete"; this round disproves that for TXT. Suspect gesture priority issue: the chrome-toggle gesture on the text view container is winning over the highlight-tap hit-test. Filed as Bug #202 / GH #740.

3. **Highlight CREATION works correctly on TXT.** Long-press → custom Highlight/Add Note/Define menu → tap Highlight → yellow paint applied → `snapshot.highlightCount: 1`. This is the WI-2 create-path, which is healthy.

4. **AZW3 highlight CREATION is broken too.** The infrastructure (FormatCapabilities.azw3 advertises `.highlights`; the Foliate-side `readerAPI.addAnnotation` exists; the AnnotationsPanelView Highlights tab supports AZW3) is all wired EXCEPT the missing `case "selection":` handler in `FoliateSpikeView.Coordinator.handleMessage`. Same as Bug #201.

5. **Verify-cron stayed strictly in scope.** Both bugs filed, neither fixed. Bug #199's `awaiting-device-verification` label stays in place; device verification will need to wait until Bug #201 (AZW3 selection wiring) lands so a highlight can be created in the first place.
