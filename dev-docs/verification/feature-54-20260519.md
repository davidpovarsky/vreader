## Observations

- The DebugBridge `open` command consistently returned `lastError: null` for
  the seeded MD book — the bridge's `fetchAllLibraryBooks()` lookup found the
  book and posted `.debugBridgeOpenBook`. But the `settle` ready file showed
  `"no active reader"` and the snapshot kept `currentBookId: null`.
- A **control test** opening a TXT book that demonstrably opened during the
  WI-5 verification (`txt:bd82...:1705`) ALSO returned `no active reader` in
  this session — so the reader-mount failure is **not WI-7-specific** and not
  a WI-7 regression. It is a headless-`simctl openurl` navigation limitation:
  the `.debugBridgeOpenBook` → `LibraryView` `NavigationStack` push does not
  reliably commit without an actively-rendered/driven UI, and the
  computer-use display was unavailable this session ("CU display
  unavailable") so the UI could not be driven manually.
- The simulator's per-app data container UUID changed across
  install/relaunch cycles, which split early seed/open attempts across
  containers — re-resolved with `simctl get_app_container` each time.
- WI-7's behavioral correctness is covered by 20 tests that drive the real
  subsystem boundaries: `MDReplacementRuleFetcherTests` (10 tests) runs the
  exact scoped `FetchDescriptor` against a real in-memory `SchemaV6`
  `ModelContainer`; `MDReaderReplacementRulesTests` (10 tests) drives the
  real `MDFileLoader.load` → real `ReplacementTransform` → real `TextMapper`
  pipeline. The SwiftData-backed suite caught a genuine production bug during
  development — `#Predicate { rule.scopeKey.isEmpty }` silently matched zero
  rows, so global replacement rules would never have applied; fixed to
  `scopeKey == ""`.
- App launch was clean on every attempt — `ReadingModeMigration.run(...)`
  executes synchronously at launch with no crash/hang.
