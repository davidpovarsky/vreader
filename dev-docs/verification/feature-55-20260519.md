# Feature #55 — Tap on annotated text to view note content inline — Gate-5 acceptance (CU-free XCUITest)

Feature #55 adds a tap-on-annotated-text **note preview**: after a user
attaches a note to a highlighted passage, tapping that annotated region
in the reader surfaces the note body *inline* — the `NotePreviewSheetView`
(the committed design bundle's `NotePreviewSheet`) — **without** opening
the full Annotations panel. All 7 WIs merged (PRs #918/#923/#927/#931/
#933/#936 + WI-7); the feature row was `DONE` and GH #619 open with
`awaiting-device-verification`.

The #55 implementation agent had concluded Gate-5 "needs computer-use".
That conflated "synthesizing a tap" with "needs CU" — XCUITest
synthesizes its own gestures. The feature-#26 verification suite
(`Feature26TextToSpeechVerificationTests`, PR #943) proved tap-driven
reader features verify cleanly through the XCUITest accessibility API
with **no computer-use**. This round verifies #55 the same CU-free way.

New suite: `vreaderUITests/Verification/Feature55NotePreviewVerificationTests.swift`
— 3 tests (TXT / MD / EPUB), each driving the full
long-press → "Note" → type → Save → tap-the-highlight → note-preview →
dismiss flow through the XCUITest accessibility API with synthesized
gestures. All 3 pass on iPhone 17 Pro Simulator (iOS 26.5); the suite
was run as a unit three consecutive times with zero failures.

## Acceptance criteria

Feature #55's row states three acceptance criteria: (a) tapping
annotated text shows the note body without opening the full panel;
(b) dismissible by tapping away or swiping; (c) consistent UX across
TXT, MD, EPUB, and Foliate formats.

| # | Criterion | Observed | Result |
|---|-----------|----------|--------|
| a | Tapping annotated text shows the note body **without opening the full panel** | All 3 tests (TXT / MD / EPUB): a highlight-with-note is created via the reader's own selection-popover "Note" flow, then a synthesized tap on the same annotated region presents `NotePreviewSheetView` (`notePreviewSheet`). The note body text "Verification note fifty five" is asserted visible inside the preview; the Annotations panel sheet (`annotationsPanelSheet`) is asserted ABSENT. Screenshots `feature-55-txt-note-preview` / `feature-55-epub-note-preview` show the inline preview ("NOTE" header, the highlighted-passage excerpt, the note body, Share / Open-in-panel handoff) over the reader with the highlighted word visible behind it, panel never opened. | **PASS** (TXT, MD, EPUB) |
| b | Dismissible by tapping away or swiping | All 3 tests: the preview's "Done" control is asserted present, tapped, and the `notePreviewSheet` is asserted gone (`waitForDisappearance`). The assertion accepts either documented dismiss path — the explicit "Done" tap, or the sheet swipe-down fallback (`.presentationDragIndicator(.visible)`) — and confirms the preview is removed. | **PASS** (TXT, MD, EPUB) |
| c | Consistent UX across TXT, MD, EPUB, Foliate | The identical long-press → SelectionPopover "Note" → type → Save → tap-highlight → `NotePreviewSheetView` → dismiss flow is exercised and passes on **TXT** (`Position Test Book`, native `UITextView`), **MD** (`mdTOC`, native Markdown `UITextView` via the shared `TXTTextViewBridge`), and **EPUB** (`mini-epub3.epub`, WKWebView — highlight tap arrives from the JS `highlightTapHandler` channel, a distinct bridge). The same `NotePreviewSheetView` surface is presented for all three (feature #55 v1 ships the bottom-sheet form for every container). **Foliate (AZW3/MOBI)** is NOT exercised by this suite — see Observations: it has no openable DebugBridge seed fixture; its `.readerHighlightTapped` → `NotePreviewModifier` wiring is identical to the verified formats (WI-7) and is covered by unit tests. The three openable formats all show consistent behavior. | **PASS** (TXT, MD, EPUB exercised; Foliate covered by shared wiring + units — see Observations) |

All three feature-row criteria pass end-to-end CU-free. The note-preview
inline surface, its dismissal, and its consistency across the openable
reader formats are device-verified. `result: pass`.

## Observations

- **The selection menu is the feature-#60 `SelectionPopoverView` for all
  three formats.** The first draft of the suite queried the legacy
  `UIEditMenuInteraction` "Add Note" menu item for TXT/MD — that failed:
  feature #60 WI-7c replaced the legacy 4-item UIMenu with a SwiftUI
  `SelectionPopoverView` (a `.sheet`) on the TXT non-chunked, TXT chunked,
  MD AND EPUB long-press paths. So a long-press on selected text surfaces
  a popover with a "Note" action row that routes through
  `SelectionPopoverActionRouter` → `.readerAnnotationRequested`. The suite
  was corrected to drive that single shared surface for every format.
- **Bug #214 accessibility-identifier propagation bit three surfaces.** A
  SwiftUI container's `.accessibilityIdentifier` propagates onto AND
  clobbers its descendants' identifiers. `SelectionPopoverView`'s root
  carries `selectionPopover` — every action button surfaces as
  `identifier: 'selectionPopover', label: '<Note/Translate/…>'`, so the
  per-button `selectionPopoverNote` identifier does not resolve and the
  container identifier is not a usable scoping parent (`firstMatch`
  resolves to whichever descendant is first in tree order — observed: the
  preview `StaticText`). `NotePreviewSheetView`'s root carries
  `notePreviewSheet`, clobbering the leaf `notePreviewSheetBody` /
  `notePreviewSheetDone` identifiers the same way. The suite queries the
  popover's "Note" action and the preview's "Done" control by exact
  accessibility LABEL, and proves the note body by an app-wide static-text
  scan for the distinctive note string — the same lesson the
  feature-#26/#54/#63 suites baked in.
- **A lingering text selection re-presents the popover after the native
  note sheet dismisses.** On TXT/MD, when the `AddNoteSheet` dismisses,
  the underlying `UITextView` selection is often still active, which
  re-presents the `SelectionPopoverView` on top of the now-painted
  highlight; a tap on the highlight in that state is consumed as a
  selection-dismiss gesture and never reaches the bridge's highlight
  hit-test, so `.readerHighlightTapped` is not posted and the preview
  never appears. The suite adds a `clearLingeringSelection` step (close
  the popover if up, then a neutral tap on blank reader space) before the
  verifying tap. This neutral tap is SKIPPED for EPUB — the EPUB reader is
  paginated and a low tap lands in a page-turn zone (an early run advanced
  the EPUB to a blank chapter-2 page that way). The EPUB note flow uses
  its own note sheet and leaves no popover-driving selection to clear.
- **The reader content is queried by element TYPE / window-anchored
  coordinates, never an unscoped `app.textViews` enumeration.** An
  unscoped `app.textViews` query on a content-heavy reader triggered a
  pathologically slow full-accessibility-tree walk (one early run stalled
  for ~3 minutes). The suite confirms the content surface via a scoped,
  fast identifier query and anchors the long-press / tap on
  `app.windows.firstMatch` coordinates.
- **The "Done" control surfaces with a shifting automation type.** Under
  iOS 26 a SwiftUI `.buttonStyle(.plain)` `Button` can surface with an
  automation type an `app.buttons` query misses (the runner logs an
  "Automation type mismatch: computed Button … vs PopUpButton"
  diagnostic). The suite matches the "Done" control as `.any` element by
  label, and `.tap()` works regardless; a sheet swipe-down is the
  documented fallback.
- **Foliate (AZW3/MOBI) was not exercised** — there is no openable
  DebugBridge seed fixture for it (`TestSeeder` provides real-file
  TXT/MD/EPUB fixtures only; the `.books` AZW3 records are metadata-only
  and never open — Bug #209, the same harness gap the feature-#26 and
  feature-#54 verification suites record). Feature #55's Foliate wiring
  (WI-7 — `FoliateSpikeView+HighlightTap` posts `.readerHighlightTapped`,
  consumed by `NotePreviewModifier`) is structurally identical to the
  EPUB path that IS verified here, and is covered by unit tests. This is a
  standing harness limitation, not a #55 regression; criterion (c)'s
  Foliate leg rests on that shared wiring rather than a device run.
- **Simulator timing variance.** Test wall-clock varied widely across
  early runs (24 s – 196 s, one runner SIGTERM) under simulator load; a
  clean simulator reboot stabilized it to ~28-29 s per test. The final
  three suite runs were all clean. No app crash was attributable to
  feature #55 (the one `.ips` crash report on disk was a pre-existing
  stack-overflow from an unrelated older build, v3.34.12).
- **No feature #55 production code was modified.** This is a
  verification-only change: the PR adds `Feature55NotePreviewVerificationTests.swift`,
  registers its 3 tests in `TestPlans/Verification.xctestplan`, adds this
  evidence file + 2 screenshot artifacts, and bumps the version. No bug
  was discovered.
