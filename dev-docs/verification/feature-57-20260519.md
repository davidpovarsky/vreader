## Observations

- **The whole-book extraction seam works on the live MOBI render.** WI-1's
  device slice already proved `readerAPI.extractPlainText()` returns a
  32,884-char whole-book string for the `mini-azw3` MOBI fixture (head
  = the Gutenberg header, tail = the license — spanning all 7 sections).
  WI-2/WI-4 confirm the production `callAsyncJavaScript` Swift path
  delivers that text to `TTSService.startSpeaking` — `ttsState` flips
  to `speaking` and `ttsOffsetUTF16` advances. The `evaluateJavaScript`
  Promise-marshalling concern from the plan's Gate-2 audit was resolved
  by switching the production path to `callAsyncJavaScript` (WI-1
  Gate-4 audit) — that API IS Promise-aware, and the device evidence
  here is the end-to-end confirmation.
- **Pause/Resume is the one criterion-4 gap.** The `vreader-debug://tts`
  URL grammar exposes only `start`/`stop` — there is no `pause`/`resume`
  action. The `TTSControlBar` play/pause button is the sole UI path,
  and computer-use (the only way to tap it headlessly) is down this
  session. Pause/resume IS exercised by `TTSServiceStateTransitionTests`
  at the unit level, and the playback state machine is format-agnostic
  shared `TTSService` code — feature #57 changed only the AZW3 *text
  source*, not the playback path — so the risk that AZW3 pause/resume
  specifically is broken is low. But per the verification SCHEMA,
  "not exercised end-to-end on device" is not a `pass`; this run is
  recorded `partial` and the row stays `DONE`.
  **To reach `VERIFIED`**: either (a) a CU-driven tap of the
  `TTSControlBar` pause/resume button on AZW3, or (b) extend
  `DebugCommand.tts` with `pause`/`resume` actions (a small DebugBridge
  enhancement) and re-run. Recommend (b) — it also helps every future
  TTS verification.
- **Rapid-repeat idempotency held** across all WI slices — re-firing
  `tts?action=start` while speaking is a clean no-op (the debug
  observer's `state == .idle` guard); WI-2's slice additionally proved
  the in-flight `azw3ExtractionTask` gate prevents a duplicate
  whole-book walk when two taps land before the first extraction
  completes.
- **One pre-existing bug surfaced and was filed, not fixed**: bug #225
  / GH #910 — `TTSService.rate`'s `didSet` infinite recursion (same
  `@Observable` self-assignment defect class as the FIXED bug #222),
  crashes `TTSServiceSpeedControlTests`. Confirmed pre-existing on
  pristine `origin/main`; out of feature #57's scope (#57 does not
  touch `TTSService`).
