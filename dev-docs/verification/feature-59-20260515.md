# Feature #59 — Register vreader as a system document handler — round 1 device-verify

## Summary

Feature #59 ships in v3.22.6 with `CFBundleDocumentTypes` + `UTImportedTypeDeclarations` + `LSSupportsOpeningDocumentsInPlace: true` in `Info.plist` and a production `.onOpenURL` handler routing through `FileURLImportRouter`. This round-1 verification drove the import pipeline via `xcrun simctl openurl booted "file://..."` for the bundled fixture set, and confirmed:

- ✅ The `.onOpenURL` handler fires for `.epub`, `.txt`, `.md`, `.azw3` file URLs.
- ✅ `BookImporter` is invoked through `FileURLImportRouter`; logs confirm imports.
- ✅ SwiftData store correctly persists imported books (6/6 inserts observed).
- ✅ Duplicate handling works — second import of the same `.epub` logs `(duplicate=true)`.
- ✅ `Info.plist` declares all 5 document type families with `LSHandlerRank: Alternate` (criterion f — confirmed via plutil inspection).
- ❌ **Library view does not live-refresh after import** — Bug #197 / GH #708 filed.
- ❌ `.markdown` extension not dispatched to vreader (Apple `net.daringfireball.markdown` UTI tag-spec gap; not a vreader bug, documented as out-of-scope).
- ⏸ `.pdf`, `.mobi`, `.prc`, `.azw` not exercised this round (no bundled fixture for PDF; AZW3 family aliases assumed-equivalent based on UTImportedTypeDeclarations declaring them under the same UTI).
- ⏸ Real-world Share Sheet UX (criterion a + e) not yet driven via Files-app + Share menu CU sequence; deferred to round 2 if Bug #197 is fixed first.

**Result: partial.** Persistence + import-dispatch pipeline works correctly. Live-refresh issue (Bug #197) blocks marking the feature `VERIFIED` because it directly impacts acceptance criterion (b). Cold-restart workaround exists but isn't acceptable UX.

## Observations

- The 4 initial imports completed in sequence within ~30s; all logged success with `duplicate=false`.
- The duplicate import logged `duplicate=true` and SQLite count remained at 4 after the second `test.epub` import → fingerprint-based dedupe contract holds.
- After the initial 4 imports the app was still in foreground, but the library view showed "Your Library is Empty" — first surfaced live-refresh bug.
- Cold-restart of the app (terminate + launch) brought all 4 books into the visible library.
- A separate live-refresh test post-cold-start (with library showing 4 books) imported a 5th — SQLite count went to 5, but library UI remained at 4. Same bug.
- Cold-launch-via-openurl test (terminate vreader, then openurl) similarly imported successfully (SQLite count 6) but library UI showed only 5 — the just-imported book was missing.
- Bug #197 was filed as GH #708 with three screenshot artifacts as evidence.
- `.markdown` extension: simctl openurl handed the URL to iOS Files app's "Save" dialog rather than vreader. Investigated and concluded this is an Apple system UTI limitation (`net.daringfireball.markdown` UTI's `public.filename-extension` tag-spec on iOS 26.5 does not include `.markdown`). Not a vreader bug; if real-world coverage of `.markdown` is required, would need to declare a vreader-specific imported type OR file an Apple feedback. Documented in the bug row + this evidence file.
