## Summary

Slice verification for feature #6 (Persist library view preferences
across app restarts — Medium priority). Status was already `DONE`
with the note "WI-001. PreferenceStore + UserDefaults. 10 tests"
and no evidence file. This iteration runs the
**36 tests across 2 suites** that cover library-view-preference
behavior end-to-end through `LibraryViewModelTests` +
`LibraryViewModelImportTests` — sort-order persistence (title /
addedAt / lastReadAt / totalReadingTime), view-mode toggle (grid ↔

import flows that interact with the preference state.

**excluded from the test target** in `project.yml`; the
preference-persistence behavior that file covers is duplicated /
exercised through `LibraryViewModelTests` which IS compiled and
runs. UI gestures (open Library → sort picker → tap option →
re-launch app → preference persists) deferred — display
unavailable.

## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 36 passing tests give **strong Gate-4-class evidence** for the
preference behavior visible at the ViewModel boundary:

- **Sort-order mutators** — all four cases (title / addedAt /
  lastReadAt / totalReadingTime) exercised through the ViewModel,
  which writes through to `PreferenceStore` on every set.
- **View-mode toggle** — both directions (grid → list → grid)
  covered.
- **Default values** — initial sort-order and view-mode defaults
  are pinned by tests.
- **Import flow integration** — `LibraryViewModelImportTests`
  proves importing books doesn't reset preferences.
- **Cross-feature reset** — feature #20 (sort-order reset/revert
  to default) is bundled with this feature; the reset path is
  covered through the same suite.

What's **excluded** from the slice: `LibraryViewModelPersistenceTests.swift`
exists in the source tree but is excluded from compilation by
`project.yml` (`vreaderTests:sources:excludes:`). The
preference-persistence behavior that file would cover is
exercised end-to-end through the ViewModel's tests instead, so the
exclusion is shape-equivalent for slice-verification purposes.

What's deferred: the UI gesture path (open sort picker → tap
option → close app → relaunch → observe preference persisted) and
the explicit cross-launch persistence test. These need display + a
real `UserDefaults` lifecycle (the in-test setup uses an
`UserDefaults` suite for isolation; a true relaunch test would need
the app to be killed and reopened).

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Row claimed 10 tests; current LibraryViewModel surface has 36.**
  The 10-test count was the WI-001-original `PreferenceStore`
  test scope; the broader Library ViewModel grew alongside
  features that depend on preferences (#9 context menu, #20 sort
  reset, #24 book sources) — the 36 tests cover the union of
  those surfaces.
- **`LibraryViewModelPersistenceTests.swift` is excluded.** The
  exclusion is in `project.yml`:
  ```yaml
  vreaderTests:
    sources:
      - path: vreaderTests
        excludes:
          - "**/.gitkeep"
          - "ViewModels/LibraryViewModelPersistenceTests.swift"
  ```
  This is a known limitation; the file exists for reference but
  isn't compiled. Behavior it would cover is exercised through
  `LibraryViewModelTests` instead.
- **Preferences are bidirectional** — read on init, written on
  every mutator. The ViewModel tests exercise both directions
  through the public API.
- **Default sort order is `addedAt`** — newest book first. Most
  apps default to alphabetical; vreader chose recency because
  users tend to read recently-imported books first.
- **Feature #19 is marked DUPLICATE of #6** — display mode
  persistence was originally a separate feature row but was
  merged in here. The view-mode toggle is the manifestation.
- **No backwards-compat concerns** — preferences live in
  UserDefaults under stable keys; no schema migration needed
  across versions.
