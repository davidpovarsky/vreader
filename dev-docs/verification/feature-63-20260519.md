# Feature #63 — Search results panel v2 re-skin — verification

Device verification of feature #63 (Gate 5b — post-merge final acceptance
for the `DONE` feature). The feature re-skins the in-reader `SearchView`
onto visual-identity-v2 (PR #854, merge commit `abfeda3`). Verified against
merged `main` `fd96f93` (v3.31.11, build 471) on iPhone 17 Pro Simulator
(iOS 26.4.1), Debug.

**Result: `partial`.** All five user-facing acceptance criteria PASS via
manual device verification. However, the re-skin shipped a regression in
the search-sheet automated UITests — all 6 `vreaderUITests` search-sheet
tests fail because the `searchSheet` accessibility identifier no longer
resolves to a queryable `otherElements` container. Filed as **bug #223**
(GH #TBD). #63 stays `DONE` (not flipped to `VERIFIED`) pending #223's fix.

## Observations

- **The re-skin itself is correct and works on-device.** Every user-facing
  criterion (C1–C5) was confirmed visually: the custom search bar, the idle
  prompt, the grouped results list, the no-results state, the clear button,
  and result-tap navigation all render and behave per the design bundle
  (`vreader-search.jsx`) and the plan's scope reconciliation (toggle / Recent
  / hint-chips correctly omitted; format-neutral "sections" copy).

- **C6 — the UITest regression — is the reason for `partial`.** PR #854
  replaced `SearchView`'s `NavigationStack` root with a `VStack`. The
  `searchSheet` accessibility identifier set on the sheet wrapper
  (`ReaderContainerView+Sheets.swift`) previously sat over a `NavigationStack`
  that XCUITest surfaced as an `Other` element, so `app.otherElements
  ["searchSheet"]` matched. With a `VStack` root, SwiftUI propagates that
  identifier onto every leaf descendant instead — confirmed by the failure
  `.xcresult` AX hierarchy, where the `Image`, `TextField`, `Button`, and
  `StaticText` all carry `identifier: 'searchSheet'`. No element is a
  queryable container, so all 6 search-sheet UITests fail at their first
  assertion. PR #854's migration updated the dismiss-button assertion
  ("Done" → "Cancel") but did not catch this container-identifier change,
  and the PR's pre-merge gate ran `-only-testing:vreaderTests` (unit only),
  so the breakage shipped. Filed as **bug #223**.

- **TXT search returns "No matches" for the war-and-peace fixture — NOT a
  #63 defect, and NOT what C4 verifies.** While probing C3 with the
  `war-and-peace.txt` fixture, in-app search for `Prince` returned zero
  results even though the FTS5 index has the hits (verified directly:
  `search_index MATCH '"prince"'` scoped to the book returns 3 segments).
  Root cause: `search_metadata.segment_base_offsets` for the fixture is
  **empty** even after a forced clean reindex, so `SearchHitToLocatorResolver
  .resolveTXT` returns `nil` for every hit (`guard let segBase =
  bases[segIndex]`), and `SearchService.search`'s `compactMap` drops them
  all. This is a pre-existing search-pipeline behavior in the TXT
  segment-offset persistence path — entirely outside feature #63's scope
  (#63 is a view-layer re-skin; it does not touch `SearchService`,
  `SearchHitToLocatorResolver`, or the indexing pipeline). It is noted here
  for the record but is **not** filed against #63. C3 was instead verified
  with `mini-epub3.epub`, whose EPUB hits resolve via href (no segment
  offsets needed) — and the grouped list rendered correctly. If the
  TXT-segment-offset gap warrants a tracker row it should be triaged
  independently against `Search/*`; it did not block this verification.

- **Tooling note.** Computer-use is virtual-display-only on this host; the
  whole run used DebugBridge + the `sim-drive-fallback` toolkit
  (`clickat.swift` / `pastekey.swift`) + `simctl io screenshot`. The
  SwiftUI grouped-list `Button` rows do not surface in the macOS
  `System Events` AX tree, so result-row taps were placed by
  window-geometry coordinate mapping rather than AX lookup — the
  navigation snapshot (`vreader-debug://snapshot`) confirmed the tap landed.
