## Summary

Gate-5b device verification of feature #65 — the AI sheet tab-body
re-skin (Summarize / Chat / Translate). Run against a Debug build of
`main` (`e30f769`, v3.31.7 build 467) on the iPhone 17 Pro Simulator
(iOS 26.4.1), driven via the `sim-drive-fallback` toolkit
(`clickat.swift` / `pastekey.swift` + `xcrun simctl io screenshot`)
because computer-use MCP is down (virtual-display-only host —
`/cu-diagnose` verdict from a prior iteration).

**Result: `partial`.** Every AI-sheet tab-body surface that is
reachable WITHOUT a live AI provider response was verified end-to-end
on device and matches the `vreader-fidelity-v1` design — the headline
re-skins of all three tabs (Translate language pill rail, Chat v2 pill
input bar + accent user bubble, Summarize idle/error states). Three
**response-rendering** surfaces — the Summarize summary card
(`.complete` state), the Chat assistant sparkle-avatar bubble, and the
Translate translation result card (`.complete` state) — could NOT be
exercised on device: the AI provider configured on the simulator
returns **HTTP 401 (Authentication failed)**, there is no valid AI
credential available in this environment (`.secrets/.env`
`OPENROUTER_API_KEY` is empty), and the DebugBridge exposes no AI-stub
command to inject a `.complete` response. Those three surfaces are
covered by the passing unit suites (`AISummaryCardTests`,
`AIChatMessageRowTests`, `TranslationResultCardTests`) but their
on-device visual rendering is deferred.

Because the result is `partial`, the `docs/features.md` #65 row stays
at `DONE` (NOT `VERIFIED`) and GH #823 stays open, per
`dev-docs/verification/SCHEMA.md` result semantics. No code defect was
observed — every surface that rendered matched the design; the gap is
purely a tooling/credential limitation, so no bug was filed.

## Observations

- **Computer-use is down** and the host is virtual-display-only (the
  `/cu-diagnose` verdict from the prior iteration). Verification used
  the `sim-drive-fallback` toolkit (CGEventPost taps + `simctl io`
  screenshots) as documented.
- **Simulator windowing constraint.** The intended isolated simulator
  (iPhone 17 Pro, iOS 26.5, UDID `1FAB9493`) is booted but the
  Simulator app can only present it as a passive "External Display"
  window on this virtual-display host — that window exposes no
  accessibility tree and does not translate CGEventPost clicks into
  device touches. The only simulator with a real interactive device
  window is the iPhone 17 Pro / iOS 26.4 (`61149F0E`). After the
  unrelated `xcodebuild` unit-test process that had been running on it
  finished, that sim was uncontended, so verification ran there. The
  build, install, and unit gate ran against the iOS-26.5 sim (no
  window needed for `xcodebuild`); the interactive device-drive ran
  against the iOS-26.4.1 sim.
- **AI provider returns HTTP 401.** The simulator had an "OpenAI"
  provider configured (a prior session enabled the `aiAssistant`
  feature flag + saved a key), but the key is invalid/expired — every
  Summarize and Translate request returns `Authentication failed
  (HTTP 401)`. This is actually useful: it exercised the re-skinned
  **error** states of both tabs end-to-end. But it means the
  `.complete` states (summary card, translation result card) and the
  Chat assistant bubble cannot be reached on device. There is no valid
  credential in this environment (`.secrets/.env` `OPENROUTER_API_KEY`
  is empty) and no DebugBridge command to stub an AI response.
- **What this confirms about the #60 gap.** Feature #65 was filed
  because feature #60's evidence overstated coverage of its criterion
  (e) — #60 only observed the AI-sheet *chrome* + the Summarize idle
  CTA, not the tab bodies. This run goes materially further: it
  verified the Translate language pill rail, the Chat v2 pill input
  bar, and the Chat accent user bubble — the headline re-skins of
  WI-2 and WI-3 — none of which #60 ever exercised. The remaining gap
  (the three AI-response cards) is narrower and is a tooling/credential
  limitation, not a re-skin defect.
- No regression observed: the AI sheet opened, all three tabs
  switched, the Chat input accepted text and rendered the user bubble,
  the language rail scrolled and re-highlighted on selection — all
  functional, no crash.

## Follow-up required

To flip #65 → `VERIFIED`, a follow-up Gate-5 run must exercise the
three deferred surfaces (rows 3, 6, 11) end-to-end. That needs ONE of:

1. A valid AI provider credential on the verification simulator (e.g.
   a working OpenRouter free-tier key pasted into `.secrets/.env`),
   so a real Summarize / Chat / Translate request reaches `.complete`.
2. A new DebugBridge command to inject a stubbed AI `.complete`
   response (the plan's own Gate-5 UITest description assumed a
   "stubbed summarize" — `vreader-panels.jsx` line ~363-368 — but no
   such stub seam was built; `Feature65AISheetTabBodyVerificationTests`
   was not created either).
