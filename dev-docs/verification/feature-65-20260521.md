## Summary

Gate-5b verification of feature #65 — **attempt 2**, re-running on
v3.38.29 after Bug #237 (`--enable-ai` consumed via
`AITestOverride.forceAvailable`) AND Bug #243 (DebugBridge `provider`
URL family) both shipped. Goal: unblock the 3 deferred slices from
`feature-65-20260519.md` (rows 3 / 6 / 11 — the AI-response cards).

**Result: `partial` (unchanged from prior round).** The two
*should-have-unblocked* prerequisites both shipped — the
`--enable-ai` short-circuit reaches the AI surfaces in XCUITest (per
Bug #93's new XCUITest in this same iteration), and the DebugBridge
provider URL configures a real OpenRouter profile from the host.
However a third structural blocker emerged on inspection: the standard
XCUITest harness has no way to drive a host-side
`xcrun simctl openurl` mid-test (Bug #1054 — the XCUITest sandbox
refuses the XPC connection to CoreSimulatorService with NSPOSIX 61).
Provider configuration via `vreader-debug://provider?action=add` MUST
happen between `XCUIApplication.launch()` and the AI-sheet interaction
phase — neither side of that interleave is reachable from a pure
XCUITest test method, and the only currently-working CU-free UI driver
on this host IS XCUITest (because the macOS-side AX bridge for the
Simulator window is broken — see "Observations" below).

The 9 pre-verified rows from the 2026-05-19 evidence
(`feature-65-20260519.md`) remain PASS on v3.38.29 by inference from
the same merged code path; no regression suspected. The 3 deferred
rows (3 / 6 / 11) still need ONE of: (a) CU restored on this host so
the existing 2026-05-19 pixel-driver flow can run, OR (b) a hybrid
host-driver / XCUITest harness that wires DebugBridge provider URL
calls in setUp before the assertion phase. (b) is a real harness
feature — filing as a follow-up.

## Observations

- **CU is structurally down for the 3rd consecutive verify-cron
  session** (this round: 2026-05-21). The host is virtual-display
  only. `mcp__computer-use__screenshot` returns
  `CU display unavailable`. The `sim-drive-fallback` toolkit's
  CGEventPost taps also do not translate into iOS touches:
  `osascript` queries the Simulator window's AX tree and gets only
  window-chrome elements (close / minimize / Home / Save Screen /
  Rotate / status text); the iPhone display group has **zero**
  child accessibility elements — the iOS-side AX tree is not being
  exported through the macOS AX layer. Empirically confirmed by
  tapping (75, 175) (the Settings gear) and observing **no** state
  change in the captured screenshot.
- **XCUITest IS the working CU-free path.** Bug #93's new XCUITest
  drove the Library chat sheet end-to-end in 50.4s, including
  finding the toolbar AI Chat button, opening the sheet, typing into
  `chatInputField`, tapping `chatSendButton`, observing the user
  bubble, dismissing, and re-opening. XCUITest synthesizes touches
  via the iOS accessibility API through simctl, NOT through the
  macOS-side AX layer that's broken on this host.
- **The provider URL (Bug #243 / GH #1057, v3.38.12) works from
  the host.** Confirmed by sending the URL and inspecting the
  resulting `ProviderProfileStore` state.
- **The hybrid harness gap.** The remaining deferred slices need
  three things in sequence: (1) the app launched with
  `--enable-ai` (so AI surfaces show), (2) provider configured via
  host-side `simctl openurl`, (3) UI-driven AI request via XCUITest.
  Today (1) and (3) live inside an XCUITest method; (2) MUST run
  from a host shell (Bug #1054 confirms host-shell `simctl openurl`
  fails from inside the XCUITest sandbox). There is no current
  harness that interleaves (1)+(3) with a host-side (2) at the right
  moment. **This is the same wall the 2026-05-19 round hit, with
  one extra layer named explicitly: the provider URL is now
  available, but the harness to invoke it mid-XCUITest is not.**
- **No regression observed** in the previously-PASS rows — the unit
  suites for #65's re-skin (51 tests) all green on v3.38.29; Bug #93's
  XCUITest drives the same `AIChatView` component as the AI sheet's
  Chat tab and confirms the pill input + send button + user bubble
  re-skin renders.

## Follow-up required

To flip #65 → `VERIFIED` definitively, rows 3 / 6 / 11 need end-to-end
exercise. Options:

1. **CU restoration**: re-run the 2026-05-19 pixel-driver flow once
   CU display is back. Lowest harness lift.
2. **Hybrid harness (filed as follow-up)**: a host-side wrapper
   script that launches the app with `--enable-ai`, sends the
   provider URL, then runs an XCUITest that attaches to the
   already-launched process (or coordinates via a fixture file the
   XCUITest reads). This is a real harness feature; filing as
   `Bug #DevTools/Verification` so it's tracked.
3. **DebugBridge AI-stub command**: instead of driving a real AI
   provider, ship a `vreader-debug://ai?stub=...` URL that injects a
   `.complete` response directly into `AIService`. Lowest credential
   dependency; highest invariance.

The follow-up is recorded in the `docs/features.md` #65 row Notes.
