## Summary

Gate-5 verification of feature #69 ‚Äî the AI Summarize scope selector
(the Section / Chapter / Book-so-far chips made functional). Run
against a Debug build of the feature-#69 WI-5 branch
(`feat/feature-69-wi-5-scope-chip-strip`, HEAD `039beb8`, v3.34.0
build 487) on the iPhone 17 Pro Simulator (iOS 26.4).

**Result: `partial`.** The complete #69 logic layer ‚Äî the scope
enum/value types, the TOC‚Üí`ChapterBounds` resolver, the scope-aware
`AIContextExtractor` (Section / Chapter / Book-so-far slicing), the
`AIAssistantViewModel` scope wiring, and the `AISummaryTabView`
chip-strip selection + `runSummarize` scope forwarding ‚Äî is verified
deterministically by **163 unit/integration tests across 12
AI/summarize suites**, all green. The TXT reader was confirmed to open
on device (the AI surface ‚Äî the bottom-chrome "AI" button ‚Äî is present;
`feature-69-09-reader-20260519.png`).

The **end-to-end on-device acceptance** of the scope chips ‚Äî opening
the AI sheet, tapping a chip, observing a scoped summary render ‚Äî could
**not** be completed, for two independent, named environmental reasons:

1. **The AI provider returns HTTP 401.** A scoped summary's *output*
   (the summary text rendered in the card) requires a live AI provider
   response. The provider configured on the verification simulator
   returns **HTTP 401 (Authentication failed)** ‚Äî `.secrets/.env`
   `OPENROUTER_API_KEY` is empty and no valid AI credential is
   available in this environment. The DebugBridge exposes no AI-stub
   command to inject a `.complete` response. This is the identical wall
   feature #65 (`feature-65-20260519.md`) and feature #68 hit.

2. **The CoreSimulator service is unstable on this host.** The
   `sim-drive-fallback` UI-tap path (computer-use MCP is down ‚Äî this is
   a virtual-display-only host) repeatedly failed: `xcrun simctl`
   returned `No devices are booted` mid-sequence, the app would not
   stay foregrounded after `simctl openurl`, and a `simctl openurl`
   eventually failed with `Mach error -308 - (ipc/mig) server died`
   (the CoreSimulator backing service crashed). The simulator state did
   not survive between commands, so the AI sheet could not be reliably
   opened and the chip strip could not be screenshotted in situ.

No code defect was observed ‚Äî the reader opens, the AI surface is
present, and every #69 logic surface that a deterministic test can
reach passes. The gap is purely a tooling/credential limitation. No
bug was filed.

Because the result is `partial`, the `docs/features.md` #69 row stays
at `DONE` (NOT `VERIFIED`) and GH #897 stays open, per
`dev-docs/verification/SCHEMA.md` result semantics. A follow-up
verification on an AI-credentialed, stable-simulator environment is
required to flip #69 to `VERIFIED`.

## Observations

- The **full `xcodebuild test -only-testing:vreaderTests` run** hits a
  SIGSEGV-at-launch flake that taints the run (`5633x "Test crashed
  with signal segv"`, zero real assertion failures). This was confirmed
  **identical on the clean WI-4 baseline** (`42908c4`, no #69 WI-5
  changes) ‚Äî it is a pre-existing test-runner/environment flake, not a
  feature-#69 regression. Every *targeted* suite run passes
  deterministically; the 163-test targeted run above is the reliable
  evidence path.
- The CoreSimulator service on this host is genuinely unstable for
  sustained UI verification: `xcrun simctl` intermittently reports
  `No devices are booted`, the app will not stay foregrounded after a
  `simctl openurl`, and a `simctl openurl` failed outright with
  `Mach error -308 - (ipc/mig) server died`. The `sim-drive-fallback`
  pixel-tap path additionally suffers because the Simulator window
  relocates between launches, defeating screenshot‚Üímac-coord
  calibration. This is a host-environment limitation, not a #69 defect.
- The HTTP-401 AI-provider blocker is structural for this repo's
  verification environment and has now blocked the *output* acceptance
  of three consecutive AI features (#65, #68, #69). A standing
  follow-up ‚Äî an AI-credentialed verification environment, or a
  DebugBridge AI-stub command that injects a deterministic `.complete`
  response ‚Äî would unblock all three.
