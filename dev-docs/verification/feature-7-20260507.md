## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 5 passing tests give **strong Gate-4-class evidence** for the
haptic-fire contract:

- **Fire-on-add path** — `feedbackGeneratorFiredOnAdd` and
  `feedbackGeneratorFiredOnAddWithPreExistingBookmark` cover the
  positive cases.
- **Don't-fire-on-error path** — `noFeedbackOnError` ensures the
  user doesn't feel a confirming tap when the add silently fails;
  `noFeedbackWhenProviderIsNil` covers the unconfigured case.
- **No debounce** — `rapidRepeatedBookmarksFeedbackFiresEachTime`
  pins that the user gets per-action confirmation, not a smoothed
  burst. The bug class to watch is "iOS coalesces rapid haptic
  fires"; the test asserts the call-count, not the felt intensity.
- **Intensity pinned** — the `BookmarkFeedback` initialiser uses
  `.light`; the tests assert this explicitly. Misclassifying as
  `.medium` would feel wrong in the user's hand but pass a
  "function got called" check.

What's deferred: the actual physical haptic on a real device.
Simulator does not vibrate. This is the only path to a true `pass`
for a haptic feature.

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Haptic intensity is a UX detail that's easy to regress.** A
  pull-request that bumped `.light` to `.medium` "for stronger
  confirmation" would feel wrong but pass a generic
  "feedback fired" assertion. The dedicated 5-test suite pins the
  intensity contract.
- **No-feedback paths are equally important.** Firing the haptic on
  failure would tell the user "saved!" when nothing was saved.
  Two of the five tests (`noFeedbackOnError`,
  `noFeedbackWhenProviderIsNil`) pin the negative invariants.
- **This feature has the same `BookmarkFeedback` surface as
  feature #1.** Feature #1 (bookmark CRUD) cited it as one of its 6
  suites; this evidence file gives the haptic its own slot in the
  inventory because the feature is *about* the haptic, not just
  the data surface.
- **Real-device verification is the canonical `pass` path here.**
  Simulator doesn't fire `UIImpactFeedbackGenerator`. Even if the
  display were available for UI gesture verification, the haptic
  itself wouldn't be felt — only a real iPhone or iPad confirms
  the user-felt outcome.
