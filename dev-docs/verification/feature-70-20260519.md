# Feature #70 — Cross-format font-size perceptual calibration — Gate-5 final acceptance

Final-WI (WI-4) acceptance pass. All four text-reflow formats (TXT, MD, EPUB,
AZW3/MOBI) opened at one fixed font-size slider value and the rendered text
confirmed perceptually consistent through the calibrator. AZW3/MOBI — whose
font-size slider was a complete no-op before this feature — confirmed to now
visibly resize the reader text.

Run on a **dedicated** iPhone 17 Pro Max Simulator (`6C32EE30…`): the default
iPhone 17 Pro sims were both occupied by concurrent agents' `xcodebuild test`
runs, which collided with the verification session (`LSApplicationWorkspace`
error 115). Per rule 48, a dedicated free simulator was used. For a font-size
calibration verification (eval-based, not safe-area layout), iPhone 17 Pro Max
is equivalent to iPhone 17 Pro.

## Observations

- **AZW3/MOBI is genuinely first-time wiring.** Before WI-4 the live
  `FoliateSpikeView` host never read `typography.fontSize` and never called
  `FoliateStyleMapper.themeCSS` — the slider did nothing. WI-4 routes a
  calibrated `setStyles` CSS through it. The before/after screenshots at
  unified 28 vs 12 are the clean proof: the same MOBI book, visibly different
  text size.
- **Probing the AZW3 rendered font size programmatically is awkward** —
  Foliate-js's `<foliate-view>` keeps the rendered book in an internal
  renderer (`fv.renderer.sections`), not in a `document.querySelector`-reachable
  iframe or shadow DOM (`hasShadow: false`, `totalIframes: 0`). So
  `getComputedStyle(document.body)` reads the *host* page's body (16px,
  irrelevant), not the book. The verification therefore confirms AZW3 two
  ways that ARE reachable: (a) the stashed `window.__vreaderTargetThemeCSS`
  global carries the exact calibrated CSS (`31px` / `13px`), and (b) the
  visual before/after screenshots show the text resizing. EPUB, by contrast,
  injects CSS into the same document the eval reads, so EPUB's `31.4px` is
  directly eval-confirmed.
- **Simulator contention is real.** Two concurrent agents' `xcodebuild test`
  runs were occupying both iPhone 17 Pro simulators; their XCUITest
  reinstall/relaunch cycles repeatedly killed the verification session with
  `LSApplicationWorkspace` error 115. Moving to a dedicated, freshly-booted
  iPhone 17 Pro Max resolved it cleanly. Lesson for next time: pin
  verification to a simulator no test gate is using.
- **The `TEST FAILED` on the full suite is not feature #70.** Across every
  WI's full-`vreaderTests` gate the run ended `TEST FAILED` with 0 assertion
  failures and exactly 1 test-host crash landing at a random unrelated suite
  (`TTSServiceSpeedControl`, `CardBody`, `UpdateChecker`,
  `PDFReaderViewModelPositionService`, `DebugReaderRegistry`, …). Verified on
  a pristine `origin/main` checkout: the suite crashes identically with zero
  feature-#70 code present. Tracked as GH #849 (`fix/issue-849-vreadertests-suite-crash`,
  fix in progress by another agent). Two instances — Bug #225 (TTS recursion)
  and Bug #228 (DebugReaderRegistry isolation) — were fixed on `main` during
  this feature's implementation; a residual flake remains under GH #849.
