# Feature #74 — Locate bloom on highlight/note landing (TXT/MD) — Gate-5b acceptance

The ~1.5s locate bloom (wash value-lift 0.4→0.86 + focus ring + soft glow) when a
Notes/Highlights row tap jumps to a highlight. The sub-second visual cannot be
screenshot/video-captured on the Screen-Sharing virtual display (`recordVideo` has
a ~3.5s capture-startup delay; TXT renders blank intermittently in `simctl io`), so
it is verified through the DEBUG CU-free harness (PR #1541): `vreader-debug://locate?highlight=N`
fires the bloom through the REAL `.readerNavigateToLocator` → reader → `playLandingBloom`
CADisplayLink path, and `DebugSnapshot.landingBloomCount` / `landingBloomPeakIntensity`
read it back. The harness surfaced (and Bug #322 fixed) that the bloom was missing
from the chunked reader; BOTH the non-chunked and chunked TXT readers now bloom.

## Observations

The bloom's sub-second visual genuinely defeats `simctl io` capture on the virtual
display (the original Gate-5b screenshot-timing attempt + a 60fps `recordVideo`
attempt both failed). The DEBUG `landingBloomCount`/`landingBloomPeakIntensity`
readback is the durable verification: it asserts the bloom FIRED + reached a peak
intensity > the resting wash, through the real CADisplayLink on the real reader,
triggered by the real `.readerNavigateToLocator` locate channel — the pixel paint
itself is unit-proven. The harness also caught that chaptered TXT (chunked) never
bloomed → Bug #322, now fixed, so the feature is complete across both TXT bridges.
