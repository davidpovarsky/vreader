# Feature #75 — RTL / vertical-writing EPUB reading — acceptance (via the Readium default engine)

## Context — engine change

Feature #75's original implementation (WI-1..WI-5a) targeted the legacy
`EPUBWebViewBridge` paged path. As of **v3.42.0 (#42 WI-14, 2026-06-01)** the
default reflowable EPUB engine is the **Readium Swift Toolkit navigator**
(`ReadiumEPUBHost`), which consumes per-publication reading direction +
writing-mode natively. So a default-engine user reads RTL / vertical-writing
EPUBs through Readium. This acceptance verifies #75's user-facing goal on the
**default** engine. The legacy bridge (reachable via the persisted
`readiumEPUBEngine` override OFF) is no longer the default path; its known
vertical-rl paged-nav imperfection (the WI-5b native-`contentOffset` snap-back)
therefore does not affect default users.

## Observations

- The Readium engine renders `writing-mode: vertical-rl` correctly out of the
  box — proper vertical columns, right-to-left column progression, upright CJK
  glyphs. This is the rendering the legacy `EPUBWebViewBridge` paged path could
  not produce reliably (see the 2026-05-31 evidence's correction history).
- Navigation in vertical-writing advances the reading position (swipe moved past
  the title into later paragraphs). Readium owns paged/scroll pagination per the
  publication's reading direction natively.
- The legacy bridge's WI-5b vertical-rl paged-nav bug (native UIScrollView clamps
  the negative `contentOffset.x`) is now **non-default** — it only affects a user
  who overrides `readiumEPUBEngine` OFF. Left as optional polish on the legacy
  engine, not a default-path blocker.
