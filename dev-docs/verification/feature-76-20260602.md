## Observations

- **WI-6 (memory gate) PASSES decisively** on the real large CJK AZW3 — the exact
  scenario the Feature-#73 Gate-2 H7 finding flagged as "never run" (the feature
  had only been verified on a 7-section English mini-azw3). RSS stays flat
  (~389–403 MB) across 60 swipes through an 85-chapter / 6.3 MB book, proving the
  K=3 windowing's `#evictOutsideWindow` keeps memory bounded; it does NOT grow
  with scroll distance. delta +1 MB, ratio 1.0 — vastly under the gate.

- **WI-5 (vertical-writing behavior) is BLOCKED on a real fixture.** The repo's
  only AZW3 (`被讨厌的勇气` / "The Courage to Be Disliked") renders **horizontal-tb**
  (confirmed visually — the body text flows left-to-right; a rendered-iframe
  writing-mode eval found the content nested below the host webview's reachable
  DOM). Modern Chinese ebooks are horizontal; vertical-rl is traditional/Japanese.
  No real vertical-rl AZW3 exists in `test-books/`, and the `mini-cjk-vlong`
  vertical-rl fixture is an **EPUB** (Readium/legacy engine), not AZW3 (Foliate),
  so it does not exercise the Foliate windowed paginator. This is the limitation
  the WI-6 plan explicitly anticipated. The vertical-rl/-lr windowing path
  (WI-2 layout + WI-3 gates/axis-math) is therefore validated by **unit tests**
  (`FoliateVerticalWindowBundleTests`, `FoliateScrollModelTests`,
  `FoliateScrolledWindowMathTests`) + a **2-round Codex audit** that reasoned
  through the vertical-rl coordinate math and CAUGHT a real bug (the right-edge
  logical-start `rectStartProp`), with horizontal-tb proven byte-identical.

## Status decision

continuity (WI-4) and the K=3 memory ceiling (WI-6) are device-verified on real

DebugBridge fixture — tracked as the WI-5 follow-up.
