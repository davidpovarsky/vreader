## Observations

- **WI-6 (memory gate) PASSES decisively** on the real large CJK AZW3 — the exact
  scenario the Feature-#73 Gate-2 H7 finding flagged as "never run" (the feature
  had only been verified on a 7-section English mini-azw3). RSS stays flat
  (~389–403 MB) across 60 swipes through an 85-chapter / 6.3 MB book, proving the
  K=3 windowing's `#evictOutsideWindow` keeps memory bounded; it does NOT grow
  with scroll distance. delta +1 MB, ratio 1.0 — vastly under the gate.

- **WI-5 (vertical-writing behavior) PASSES via a synthetic-vertical harness.**
  The repo has no real vertical-rl AZW3 (the only AZW3, `被讨厌的勇气`, renders
  horizontal-tb — modern Chinese ebooks are horizontal; the `mini-cjk-vlong`
  vertical fixture is an EPUB, not AZW3). So a DEBUG-only harness
  (`--force-foliate-vertical-rl` → a locked `window.__vreaderForceVerticalRL`
  global → the paginator's section `afterLoad` injects `writing-mode: vertical-rl`
  BEFORE `getDirection` reads it) forces the REAL AZW3 to vertical-rl. On-device,
  the book then renders genuine vertical-rl columns (top-to-bottom, RTL), and the
  windowed continuous scroll crosses sections smoothly on the horizontal axis in
  BOTH directions with RSS bounded — exercising the full WI-3 vertical path
  (`scrollModelFor('vertical-rl')` → horizontal axis + negative sign →
  `#applyScrolledContainerAxis` → `#ensureWindow` → the axis-aware windowed
  primitives). The harness is Release-inert (the global is hard-`false` +
  non-writable in Release; the flag is `#if DEBUG`-only) and is itself
  Codex-audited (2 rounds — closed a global-poisoning bypass). This complements
  the unit tests (`FoliateVerticalWindowBundleTests`/`FoliateScrollModelTests`/
  `FoliateScrolledWindowMathTests`) + the WI-3 audit (which caught the right-edge
  `rectStartProp` bug). Caveat: this exercises the windowed-scroll MATH on real
  book content, not a real vertical-authored AZW3's publisher metadata/CSS.
