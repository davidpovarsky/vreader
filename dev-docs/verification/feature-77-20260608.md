# Feature #77 — Inline bilingual translation-progress shimmer (Gate-5b attempt)

First verification attempt (no prior evidence file). **Result: partial /
CU-free-blocked** — the live in-flight loading shimmer could not be observed.
Structural correctness is covered by 23 unit tests
(`BilingualLoadingShimmerWI1Tests`); the live end-to-end shimmer is not reliably
CU-free verifiable in this harness.

## What was attempted

Reached the bilingual setup sheet (multi-chapter EPUB, `--mock-ai --enable-ai`):
menu → "Bilingual mode · Translate inline" → setup sheet (Chinese, Paragraph
granularity). Turned bilingual ON — the "EN | 中" pill appeared (mode enabled) —
but **no interlinear translations and no loading decorations ever rendered.**

## Blockers (why the shimmer couldn't be observed)

1. **Provider-readiness gate.** The setup sheet showed *"No AI provider
   configured"*. `BilingualAIReadiness.resolve` (the exact live gate) requires an
   **active provider profile with a non-empty per-profile keychain key** —
   `--mock-ai` only sets `AITestOverride.mockProvider`, which is honored by
   `AIService.resolveProvider`/`sendRequest` (lines 162/320 — confirmed) but does
   NOT populate a profile, so the readiness gate returns false.
2. **Seeded a mock-backed profile** via `vreader-debug://provider?action=add`
   (Bug #243 harness; dummy key `mock-test-key`, `https://mock.test` endpoint —
   no real auth, the mock intercepts the actual request). Still no translations.
   There is **no readiness/state readout**, so whether the seeded profile
   satisfied `activeProfileSnapshot()` + the keychain read could not be confirmed.
3. **Prefetch trigger.** The inline prefetch fires on a Swift-side reader
   locator-change (`prefetchTrigger(forLocator:)`); a DOM `scrollTop +=` eval and
   an off/on re-toggle did not induce a fresh prefetch that produced output.
4. **Engine.** Under `--uitesting` the EPUB rendered via the **legacy #71
   continuous stitch** (`scrollRoot:true, iframes:0, readiumish:false`), not the
   feature's primary Readium surface (WI-2). A definitive top-level eval showed
   `anyBilingualClass:0` / `[data-vreader-decoration]:0` — a true zero, not a
   sub-frame false-negative.

## What IS verified (structural)

- `BilingualLoadingShimmerWI1Tests` (23 tests): `bilingualLoadingCSSRule()`
  light/dark/oled gradients + `@keyframes vreaderBilingualShim`;
  `EPUBBilingualJS.bilingualInjectLoadingJS` (loading decoration + 2 shimmer bars
  per bid, skip-already-decorated, FoliateJSEscaper hardening, land→translate
  class-clear, global vs section-scoped clear); `EPUBBilingualOrchestrator.buildLoadingJS`;
  the `setInFlight` funnel posting the full in-flight set on
  `.readerBilingualPrefetchDidChange`.
- `FoliateBilingualJSTests` / `FoliateBilingualOrchestratorTests` (Foliate builders).

## Harness gap (recommended unblock)

A DebugBridge command to **enable bilingual + force a prefetch** on the active
reader (bypassing the setup-sheet UI + the Swift-side locator trigger), plus a
**readiness/state readout** (is a provider configured? is a unit in-flight?).
With those, the structural eval (`.vreader-bilingual-loading[data-vreader-decoration]`
+ 2 `.vreader-shimmer-bar` + resolved `vreaderBilingualShim` animation, during the
in-flight window) becomes a reliable CU-free gate. Without them, the inline
bilingual surface needs a real API-key provider (prohibited to enter) to drive a
genuine prefetch.

## Disposition

Stays `DONE` (not VERIFIED). Structural path unit-verified; live in-flight shimmer
CU-free-blocked. Recommend the harness command above, or a human/CU on-device pass
with a real provider configured.
