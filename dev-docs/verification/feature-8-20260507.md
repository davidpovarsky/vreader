## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 83 passing tests give **strong Gate-4-class evidence** for the
data + view layer:

- **`ReadingProgressBar` SwiftUI view** — its tests exercise the
  binding shape, percent → thumb-position mapping, and scrub-drag
  behavior. Pure SwiftUI, no UIKit gesture path.
- **Per-format conversions** — `TXTMDProgress` (continuous, char-
  offset based) and `PDFProgress` (page-based) are tested as pure
  functions across all boundary conditions. Same input always
  produces same output, including the 0-char and 0-page edges.
- **EPUB chapter-based progress** is exercised through the EPUB
  bridge's locator round-trip tests rather than a separate
  `EPUBProgress` suite. The chapter index + intra-chapter
  progression flows through `Locator.totalProgression` which is
  covered in the EPUB locator suites.

What's deferred: the UI gesture path (drag scrubber → release →
reader scrolls in real time across all four formats) needs the
simulator with a display. Pure data-and-view-layer behavior is
fully exercised below.

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Row claimed 108 tests; the isolated progress-suite count is 83.**
  The "108" appears to have included adjacent locator and
  position-service tests that were part of WI-004's broader
  surface but live in their own files (per-format ViewModel
  position-service suites). Those are verified separately under
  the per-format feature evidence (e.g. PDF position service
  tested in `feature-17-20260507.md`).
- **Three progress modes, one bar.** The `ReadingProgressBar` view
  is format-agnostic — it takes a 0.0-1.0 binding and renders. The
  per-format helpers convert format-native positions (char offset
  / page index / chapter+progression) into that 0-1 percent.
  Adding a new format means writing one converter, not modifying
  the bar view.
- **EPUB doesn't have its own `*Progress` test suite.** The EPUB
  bridge generates `Locator` objects with `totalProgression` set
  by Foliate-js; that field is the percent the bar consumes.
  Tested through the EPUB locator suites already verified for
  feature #11 (EPUB rendering) on a prior iteration.
- **Continuous vs page-based vs chapter-based** — three
  fundamentally different reading-position models, all surfaced
  through the same scrubber UI. The per-format converter is the
  abstraction boundary; once the converter is right, the bar
  works without further wiring.
- **No fancy edge cases beyond zero-length docs.** The scrubber
  doesn't do non-uniform-density tricks (e.g. weighting by
  paragraph count or word density); it's pure linear by character
  / page / chapter index. Tests reflect that simplicity.
