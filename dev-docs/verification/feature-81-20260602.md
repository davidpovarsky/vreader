# Feature #81 — In-reader AI Providers entry — Gate-5 acceptance

Device verification of the in-reader AI Providers routing on iPhone 17 Pro
Simulator (UDID `61149F0E-DC18-4BE2-BB37-52659F1F4F62`), app v3.43.0 build 808
(== the WI-1 branch / PR #1395). Driven via DebugBridge (state) + idb
`scripts/sim-tap.sh` (gestures); provider state seeded/cleared via the
`provider?action=add|clear` bridge so no API key was ever typed (entering
credentials is a prohibited agent action — see criterion (c)).

`--enable-ai` set the `aiAssistant` flag + AI consent; an active provider was
present from the bridge-seeded set, so `BilingualAIReadiness` resolved true for
the "Change…" path and false (after `provider?action=clear`) for the "Set up"
path — exercising BOTH engine-strip states truthfully.

## Observations

- The bilingual setup sheet only shows on FIRST enable per book (`needsSetupSheet`),
  and per-book bilingual config persists by deterministic fixture fingerprint
  across `reset` — so a fixture configured earlier in a session won't re-show the
  sheet. Use a not-yet-configured fixture to reach the first-enable sheet. (Harness
  note, not a defect: there is no DebugBridge to reset per-book bilingual config or
  force the setup sheet.)
- Engine-strip truthfulness held: "AI provider configured"/"Change…" with an active
  provider; "No AI provider configured"/"Set up" after `provider?action=clear`.
- No double chrome, no nav glitches on push/modal/pop.
