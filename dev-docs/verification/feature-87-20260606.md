# Feature #87 — AI stop/cancel control (interrupt an in-flight AI request) — Gate-5b acceptance

A user-facing Stop control that interrupts an in-flight AI request across all three
AI tabs (Chat / Translate / Summarize), mirroring the standard chat-app pattern where
the composer send disc morphs in place into a Stop while a reply streams. Shipped over
3 WIs (WI-1 Chat cancellation primitive + send-disc morph, WI-2 Translate stop, WI-3
Summary stop / VM-lifecycle refactor), all merged to main by v3.55.0.

The stop behavior is a pure, VM-level cooperative-cancel state machine: an op-token
(`opId`) race guard plus id-based writes guarantee that a cancelled stream's late chunks
/ late errors are discarded, the partial chat reply is kept (or dropped, per surface),
and the send-disc state (`ComposerSendState`) is a pure function of `(isLoading, hasInput,
isComposerDisabled)`. None of this requires a live AI provider to exercise — the shipped
Swift Testing suites drive every acceptance criterion through the real `AIChatViewModel`
/ `AITranslationViewModel` / `AIAssistantViewModel` cancellation paths against gated stub
providers, which is the appropriate end-to-end substitute for cancellation logic whose
observable contract is VM state, not network bytes. Verified CU-free by re-running those
suites on merged main (v3.59.0) as part of a targeted re-run of the shipped suites (35 via -only-testing).

> **Verification-exception note (live-observation criteria).** The literal *visual* abort
> — watching the streaming reply stop mid-flight against a real AI backend — is
> provider-gated and not deterministically reproducible on a device without a live API
> key + a long-enough running request to interrupt (same posture as the #314 / #311 AI
> bug verifications). This evidence closes under the verification-exception path: each
> criterion above is exercised by a shipped Swift Testing suite that drives the **same
> production cancellation code path at the real subsystem boundary** — the actual
> `AIChatViewModel` / `AITranslationViewModel` / `AIAssistantViewModel` cancel primitives,
> their op-token / id-based-write race guards, and the `ComposerSendState.resolve` pure
> state machine — against gated stub providers (`GatedChatAIProvider`,
> `WI3GatedTranslationProvider`, `GatedAIProvider` / `SecondCallGatedProvider`) that
> deterministically suspend a stream so cancel can be observed mid-flight. These are not
> casual stubs: they pause the real streaming loop the way a slow network would, so the
> cancel races (late chunk, late error, late normal return, regenerate data-loss) are hit
> exactly as production would hit them.

## Observations

The whole feature is a **cooperative-cancel + op-token** discipline applied uniformly
across the three AI tabs: each VM stores its in-flight `Task`, tags it with a
monotonically-advancing `opId`, and gates every post-`await` write on "is this still the
current op?" — so a cancelled stream's late chunk, late error, or even a late *normal*
return is dropped rather than clobbering fresh state. The most interesting (and most
regression-prone) cases are the ones the Gate-2 Codex audit caught before implementation
and which the suite now pins: a cancelled op erroring late (Chat 1c), a late normal
provider return after a Summarize cancel (3a), and Stop-during-regenerate that must keep
the prior summary instead of dropping to the idle prompt (3b). The send-disc morph itself
is the cheapest part to verify — `ComposerSendState.resolve` is a pure function, so its
parameterized table covers the full truth table including the "loading wins even when the
composer is disabled / has no input" rows.

No live API was used, and none is needed: the observable contract of a Stop is VM state
(`isLoading` cleared, partial kept/dropped, no stale error, prior summary preserved), not
the network transcript. The gated stub providers suspend the real streaming loop, so the
cancel races are exercised against the actual production code paths at the VM boundary.
`ComposerSendStateTests` lives alongside `AIChatViewModelTests` in
`vreaderTests/ViewModels/AIChatViewModelTests.swift` (same file, separate `@Suite`), which
is why both names map into the same source file.
