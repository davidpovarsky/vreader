# Feature #88 — AI conversation sessions (multiple switchable chats per book) — Gate-5b acceptance

Multiple persisted, switchable AI conversations per book: a new `@Model ChatSession`
(SchemaV8→V9 additive lightweight migration, `Book.chatSessions` cascade), a
`ChatSessionPayload` Codable persistence envelope (decoupled from non-Codable
`ChatMessage`/`ChatCitation`, encode→`Data?` never-wipe, version-gated decode),
`PersistenceActor+ChatSessions` CRUD, the `AIChatViewModel` session lifecycle on ONE
serialized async lane (`runSerializedSessionOp`), and the `ChatSessionBar` +
`ConversationsSheet` UI.

Verified CU-free by re-running the shipped Swift Testing suites on merged main (v3.59.0)
as part of a targeted re-run of the shipped suites (35 via -only-testing) that returned `** TEST SUCCEEDED ** (35 shipped suites, 0 failures)`. The
session-lifecycle and persistence behaviors are deterministic and exercised through the
REAL subsystem boundaries — the `AIChatViewModel` serialized session lane, the
`PersistenceActor+ChatSessions` actor methods, the SchemaV9 migration plan, and the
`ChatSessionPayload` Codable transport — against an in-memory `ModelContainer` on
`Schema(SchemaV9.models)`, the same SwiftData boundary production hits. The race/ordering
behaviors (rapid switch, cold-open send-before-load, mid-send transition) are timing
hazards that physically cannot be triggered deterministically on a device without a fault
injection harness; the cited `AIChatViewModelSessions` tests drive the same serialized-lane
code path through the real VM and actor boundaries (see the verification-exception note on
the criteria table).

> **Verification-exception note (race/ordering criteria in #3).** The "rapid switches don't apply a stale load", "cold-open send-before-load", and "in-flight send doesn't land in the wrong/deleted session" behaviors are timing-dependent interleaving hazards: on a device the serialized session lane resolves operations faster than a human can trigger the overlap, so the failure mode (a stale load clobbering a fresh thread, or a streamed reply landing in the wrong session) physically cannot be reproduced manually without a fault-injection harness that suspends the async lane mid-operation. The cited `AIChatViewModelSessions` tests drive the **same** `runSerializedSessionOp` lane through the real `AIChatViewModel` + `PersistenceActor+ChatSessions` actor boundary (not stubs) using `async let` overlap + abandonment assertions, exercising the production code path the device failure would hit. This is the narrow VERIFICATION-EXCEPTION close per AGENTS.md for failure modes that cannot be observed on a device without an injection harness.

## Observations

- **Session lifecycle (new / switch / rename / delete).** `AIChatViewModelSessionsTests`
  exercises the full lifecycle through the real VM: `newConversation_sealsCurrent_resetsToEmpty`
  + `firstTurn_lazyCreatesSession_persistsOnSettled` confirm a fresh thread is only persisted
  once it has a settled turn (empty sessions never hit the store); `switchToSession_*`,
  `renameActiveSession_*`, and `deleteActiveSession_fallsBackToMostRecentRemaining` /
  `…noneRemaining_resetsToEmpty` confirm graceful active-row fallback.
- **Serialized session lane.** All transition/race tests
  (`rapidSwitch_BthenC_olderLoadAbandoned`, `coldOpen_sendBeforeLoadResolves_sentTurnWins_loadAbandoned`,
  `switchMidSend_cancelsStream_replyDoesNotLandInNewSession`, `concurrentTransitions_serialize_toCleanSingleSession`)
  pass against the single `runSerializedSessionOp` lane — the WI-3 7-round Codex audit closed
  this whole interleaving-race class, and the tests assert the abandonment/seal invariants
  directly, deterministically (no `Task.sleep` timing). The non-active delete tests
  (`nonActiveDeleteDuringActiveStream_doesNotCancelStream_norLoseTurn`,
  `unrelatedDeleteDuringActiveSave_doesNotLoseActiveMessages`) confirm unrelated deletes
  never cancel the active stream or lose the active turn.
- **Stored-title persistence.** `AIChatViewModelTitleTests` confirms the displayed title is
  derived from the first user message, preserves CJK (`activeSessionTitle_preservesCJKFirstUserMessage`),
  survives a later settled turn (`storedTitle_survivesLaterSettledTurn`) and the
  new-conversation seal (`storedTitle_survivesNewConversationSeal`), and shows the stored
  title after switch/load — guarding the WI-4 preserve-title-on-update behavior.
- **Payload Codable round-trip.** `ChatSessionPayloadTests` guards the WI-1 envelope
  decision: `nilDataDecodesToEmpty` / `garbageDataDecodesToEmpty` / `wrongShapeJSONDecodesToEmpty`
  never throw, `futureVersionDecodesToEmpty_notSilentlyFlattened` enforces the version gate,
  and `encodeReturnsNonNilForValidMessages` guards the never-wipe `Data?` encode path. CJK
  content round-trips byte-for-byte (`cjkContentRoundTrips`).
- **SchemaV9 migration.** `SchemaV9MigrationTests` opens a populated V8 store under V9 with
  all rows preserved, confirms the plan is additive (length 9, V9 last, no explicit stages),
  and that deleting a `Book` cascades to its `ChatSession`s — the same cascade is independently
  re-asserted at the actor boundary (`PersistenceActorChatSessionsTests.deletingBookCascadesToSessions`).
- **Actor CRUD.** `PersistenceActorChatSessionsTests` drives create / fetch-summaries /
  fetch-full / update / rename / delete through the real `PersistenceActor` on an in-memory
  `Schema(SchemaV9.models)` container, including `updatePreservesFutureVersionBlob` (+ the
  incompatible-shape variant) — the never-clobber-a-future-blob invariant the Codex audit
  flagged — and `summariesSortedByUpdatedAtDescending` for the most-recent-first list order.
- All behaviors are deterministic and CU-free; no device tap-through was required because the
  session-switcher logic is fully exercised at the VM + actor + migration + payload boundaries.
