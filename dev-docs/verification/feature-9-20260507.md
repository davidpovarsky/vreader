## Why partial, not pass

Per `dev-docs/verification/SCHEMA.md`:

> Live integration tests against the actual backend counts as
> verification. Unit + protocol tests alone are NOT verification.

The 35 passing tests give **strong Gate-4-class evidence** for the
non-UI-gesture parts of the feature:

- **`BookInfoSheet` rendering** ŌĆö the SwiftUI view's metadata
  rendering tests cover what shows (title, author, format, file
  size, last opened) and how each field formats.
- **File-size formatter** ŌĆö round numbers, very small (under 1 KB),
  very large (GB scale), and exact-byte-boundary edges (1024 / 1
  KB, 1024┬▓ / 1 MB, etc.). Pure function, fully covered.
- **File-URL resolution** ŌĆö `LibraryBookItem.resolvedFileURL` is
  the bridge between the BookRecord (which lives in SwiftData) and
  the actual file on disk for sharing. Tests cover imported books,
  sandbox-relative paths, and the iCloud Drive case.
- **Share-sheet item construction** ŌĆö `ShareSheet.shareItems`
  packages the URL with correct UTI for the Activity sheet to
  render the right preview. Tested directly.

What's deferred: the UI gesture path itself ŌĆö long-press to open
the context menu, tap each entry, observe the sheet/share
activity/delete action. These are pure UIKit/SwiftUI plumbing on
top of the data layer this slice covers.

`result: partial` per SCHEMA. Tracker status stays `DONE`.

## Observations

- **Row claimed 24 tests; isolated count is 35.** Test count grew
  organically as edge cases for file-size formatting + URL
  resolution were added after the original WI-006 ship date.
- **Set Cover and Delete entries don't have isolated context-menu
  tests** ŌĆö they're verified at the feature surface they belong
  to: Set Cover via PhotosPicker (feature #6 library preferences),
  Delete via `LibraryViewModelTests` (the deletion path through
  `PersistenceActor`). The context menu is just the entry point;
  the per-action surface is covered elsewhere.
- **`BookInfoSheet` shows last-opened time + format + file size** ŌĆö
  classic info-sheet metadata. The format chip is computed by the
  `BookRecord`'s extension lookup; tested via the metadata
  rendering test.
- **File-URL resolution matters more than it looks** ŌĆö books can
  live in: app sandbox, the user's Documents/Inbox, iCloud Drive
  ubiquity container, or be lazy-downloaded blobs (feature #47).
  `resolvedFileURL` flattens those cases for the share sheet.
- **Pure-formatting features need fewer integration tests** ŌĆö once
  the formatter and URL resolver are correct, the SwiftUI view
  just renders them; gesture verification is the only deferred
  layer.
