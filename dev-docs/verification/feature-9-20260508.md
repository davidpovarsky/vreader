## Summary

Round-2 slice for feature #9 (Comprehensive book context menu in
library — Medium). Extends the 2026-05-07 evidence
(`feature-9-20260507.md`) which verified `BookInfoSheet`,
`FileSizeFormatter`, `LibraryBookItem.resolvedFileURL`, and
`ShareSheet.shareItems` at the unit-test level (35 tests across 4
suites) with `result: partial — UI gesture long-press deferred`.

This round closes the **long-press → context menu → Delete** UI
gesture leg via `vreaderUITests/Library/DeleteConfirmationTests`
(6 XCUITest methods, 160.2 s wall-clock; one method takes 96 s
because it deletes books one-by-one until the empty state appears).
All synthesized real long-press / swipe / tap events via
XCUIApplication, no Computer Use, no display capture. Target
simulator was iPhone 17 (iOS 26.4 Simulator).

The Info and Share legs of the context menu are NOT yet exercised
by XCUITest — DeleteConfirmationTests focuses on the Delete leg
because that's where the destructive-action confirmation gate
lives. The remaining Info / Share gestures are simpler UI
hand-offs (Info presents a sheet; Share presents the system share
sheet) and should be added as a follow-up to fully retire the
"long-press deferred" annotation.

## Observations

- `testDeleteLastBookShowsEmptyState` takes 96 s because it iterates
  through all 8 seeded books, deleting one at a time and waiting for
  the alert / dismiss / list-refresh cycle each iteration. That's
  legitimate end-to-end work; the slow run is the test, not a
  regression.
- The long-press gesture itself is synthesized by XCUITest's
  `press(forDuration:)` API on the book-card button — confirms that
  SwiftUI's `.contextMenu { ... }` modifier does land the menu in
  the accessibility tree, not just in the visual layer. That was a
  real concern when context menus were originally wired (some early
  iOS versions exposed them only to VoiceOver via different paths).
- Empty-state transition is verified via `AccessibilityID.emptyLibraryState`
  becoming visible after the last delete — same path that
  `LibraryEmptyStateTests` covers on a fresh launch. Two test files,
  one shared identifier; no drift.
