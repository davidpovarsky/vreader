# Feature #90 — Bilingual AI summary (Summarize-tab language + single/dual control) — Gate-5b acceptance

Brings the bilingual language choice into the **Summarize** tab: the summary renders
as produced (original-only), translated to a target language (target-only), or
interlinear (both stacked), driven by a control row beneath the #69 scope chips, with
a dual-skeleton loading state and a translation-failure recovery (Retry translation /
Keep original). Verified CU-free by re-running the shipped Swift Testing suites on
merged main (v3.59.0) as part of a targeted re-run of the shipped suites (35 via -only-testing). The acceptance criteria
split into three layers, each owned by a shipped suite that drives the real boundary:
the summarize→translate two-step in `AIAssistantViewModel` (16 tests), the card render
selector (13 tests), and the language/single-dual control derivations (langrow suite) —
exercised against an injectable `AIProvider` double through the real `AIService`, so no
live API key is needed.

## Observations

AI live-response is NOT needed for this feature's acceptance. The summarize→translate
two-step is driven entirely by an injectable `AIProvider` double in the VM suite — a
`GatedAIProvider` (whose `sendRequest` blocks until the test explicitly releases each
response, so tests can hold the summary done while the translate is still in flight) plus
a canned `StubAIProvider` for the summary half — wired through the real `AIService`. No
real API key, no network. That gating is what lets the suite assert the load-bearing
behaviors deterministically: that the second (translate) step does NOT clobber the
summary's `responseText`/`state`, that Retry re-runs only the translate half, and that a
language change / mode flip / reset / re-summarize each supersedes or cancels an in-flight
translation. The render selector (`AISummaryCardBody.resolve`) and the `AISummaryLangRow`
control mapping are pure functions, so they are exercised directly with no double at all —
`everyPairResolvesToABody` covers the full display-mode × translation-state matrix so no
composition is left undefined. The only thing not exercised here is the live model's actual
translation quality, which is out of scope for an acceptance gate and would require a real
key; every observable wiring/state/render contract is covered CU-free.
