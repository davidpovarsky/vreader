# Feature #91 — Agentic AI chat (tool/function calling) — Gate-5b acceptance

Adds **tool/function calling** to the AI chat: the model can invoke three
read-only tools (`search_current_book`, `search_other_books`,
`get_book_content`) inside a bounded, max-iteration-capped agentic loop with one
pre-resolved provider. Provider seam supports Anthropic `tool_use` + OpenAI
`tool_calls`, gated per-provider on `supportsToolUse`. The `agenticTools` flag is
default OFF — when off/unsupported the chat is byte-for-byte today's behavior.

Verified CU-free by re-running the shipped Swift Testing suites on merged main
(v3.59.0) as part of a targeted re-run of the shipped suites (35 via -only-testing). The suites exercise every
acceptance criterion through the REAL subsystem boundaries — the DTO codec, the
provider `sendToolRequest` seam (Anthropic + OpenAI body/response parsing), the
three tool executors over real in-memory FTS index / content adapters, the
`AgenticChatDriver` send→tool→result loop, and the `AIChatViewModel` routing +
flag — against an in-memory environment. The loop is driven by a **stub
tool-use provider** (no live API), which is the appropriate end-to-end substitute
for this backend, silent-loop feature with no shipped UI (the visible
tool-activity UI is the separately-designed needs-design #1483, held for a
separate go-ahead).

**Note (VERIFICATION-EXCEPTION close — `backend: n/a`).** The only path that
physically requires a device is the human-gated default-ON flip driving a *live*
Anthropic/OpenAI provider through a real network tool-using chat — that depends
on a paid external API key + network round-trips and cannot be reproduced CU-free
on a device without a live-credential harness that does not exist here (and would
leak the loop into real billing). Per the AGENTS.md close-gate exception, the
cited suites drive the **same code path** through the real subsystem boundaries:
the production `AgenticChatDriver` send→tool→result loop, the real
Anthropic/OpenAI body-build + response-parse seam (`sendToolRequest`,
`supportsToolUse`), the three executors over the real in-memory FTS index /
content adapters, and the real `AIChatViewModel` routing — exercised with a stub
tool-use provider standing in only for the network transport. The loop logic, the
provider serialization, the tool dispatch, and the flag gating are therefore
verified at the real boundary; only the live HTTP transport is substituted.

## Observations

The agentic stack verifies cleanly across every layer:

- **DTOs / provider seam** — `AIToolDTOTests` guards the `JSONValue` codec at the
  brittle edges (whole-number→Int bridging, non-finite coercion, Foundation
  bool-vs-number disambiguation), which is what keeps the Anthropic `tool_use`
  input and OpenAI `tool_calls` `arguments`-string from silently corrupting on
  the wire. `AnthropicProviderToolUseTests` and `OpenAICompatibleProviderToolUseTests`
  exercise BOTH the request body (`bodyCarriesTools` / `toolsAsFunctionObjects`,
  empty-tools omission) and the response parse incl. malformed-block skipping and
  the strict history-shape validation (`toolUseNotFollowedThrows`,
  `missingToolResultIDThrows`) — the half-message states a real provider can emit.

- **Tools over real adapters** — the three executors run against real in-memory
  FTS index / content adapters (`LibrarySearchBackendAdapterTests`,
  `BookContentProviderAdapterTests`), not bare stubs. `search_other_books`
  specifically asserts gating on the CANONICAL format parsed from the
  fingerprint key, not the parallel String `@Model` column
  (`gatesOnCanonicalFormatNotColumn`) — the Gate-2 audit's High finding — and
  reports excluded coverage rather than silently dropping un-indexed books.
  `get_book_content` proves locality beats format (`localityBeatsFormat`) so a
  remote-only book in a supported format is still an explicit error, not a fetch.

- **Bounded loop** — `AgenticChatDriverTests` is the load-bearing suite: it proves
  the loop terminates at the iteration cap and returns the last text rather than
  spinning (`capStopsAtMaxFallback`, `capReturnsLastText`), continues past a tool
  error, and surfaces provider throws. The loop is driven by a stub tool-use
  provider — there is no live API call in any test, which is exactly why this is a
  verification-exception close for the live-network portion.

- **Default-OFF safety** — the four `…_fallsBackToStreaming` /
  `nonToolProvider_fallsBackToStreaming` tests plus `agenticToolsDefaultOffEverywhere`
  together prove the headline safety property: with the flag off OR no registry OR
  a non-tool provider, `sendMessage` takes the exact legacy streaming path, so
  shipping this on main with the flag default-OFF is byte-for-byte today's chat.
  `toolDrivenReply_suppressesCitations` confirms the Gate-2 decision to suppress
  the "Drew on" snapshot on tool-driven replies (it can't reflect tool-read
  sources).

No regressions surfaced; the targeted re-run of the shipped suites (35 via
`-only-testing`) returned `** TEST SUCCEEDED **` (0 failures) with no concurrent
sim driving (rule 52 honored — UDID pinned, no verification traffic during the run).
