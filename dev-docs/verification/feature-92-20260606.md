# Feature #92 — Justified text alignment for the TXT reader

Verifies the feature's stated acceptance criterion: justify TXT body text so
both margins are flush (the convention CJK readers expect), fixing the user's
reported "padding on left and right is not same wide" ragged-right-edge
complaint. WI-1 (the single `paragraphStyle.alignment = .justified` seam in
`TXTAttributedStringBuilder.build`) is the delivered, verified change; it is
inherited by all three TXT render paths (non-chunked, chunked, paged body
text) through the shared builder seam.

## Observations

- The rendered text is unambiguously justified: in the captured reader screen,
  every non-final line of each multi-line paragraph reaches the right inset,
  while each paragraph's terminal line is left-ragged — the exact, correct
  justified-paragraph behavior. The drop-cap chapter-start decoration
  (feature #68) coexists with the justified body, confirming the
  `TXTChapterStartDecorator` `.center` override on the title does not regress
  the body's `.justified`.
- The TXT seed cleanly resolved to a non-chaptered fixture (non-chunked
  render path). The chunked path (the user's literal large-CJK case) is not
  separately screenshotted here, but it consumes the **same** `build()` seam
  (`TXTChunkedReaderBridge.swift:1042` per the plan) that criterion 3's unit
  suite proves, so its body text justifies identically by construction.

## Known limitation (descoped residual — WI-2)

- **Paged-mode page-bottom mid-paragraph line.** When a paragraph crosses a
  PAGE boundary in paged layout, the line at the bottom of the page is a
  mid-paragraph line that TextKit lays out as the page's terminal line, so it
  stays natural-aligned (ragged) rather than justified. This affects only
  paged mode (opt-in; the default + the user's reported case is scroll), only
  the single page-bottom line, and only when a paragraph spans the boundary.
  The WI-2 render-extension attempt leaked off-page text into the selectable
  surface (Codex `019e9434`); the clean fixes (selection-clamp /
  shared-layout custom render) need interactive on-device selection
  verification (long-press-drag / Select-All / copy at the page boundary) that
  is not expressible CU-free. It is descoped from #92 as a documented known
  limitation and tracked in the #92 row history; revisit with device access.
  This residual is beyond the feature's stated acceptance criterion ("justify
  so both margins are flush"), which WI-1 satisfies for the default reading
  experience.
