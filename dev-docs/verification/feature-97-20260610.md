# Feature #97 — `list_library` agentic tool — Gate-5b acceptance (verification-exception)

## Why a verification-exception

Acceptance criterion 1 ("a real tool-use LLM, given the `list_library` definition +
the updated agentic system prompt, chooses to call it for 'what's in my library?'
and enumerates the library") **cannot be reproduced on a device with the
deterministic harness**: `MockAIProvider` does not implement `supportsToolUse`
(`AIProvider`'s default is `false`), so the agentic gate in
`AIChatViewModel+Streaming.swift` falls back to the streaming path and never calls
a tool. Only real Anthropic / OpenAI-compatible providers do tool-use, and none is
configured for an autonomous run (no API key). The model-routing step is genuine
LLM behavior, not a deterministic path the mock can drive.

Per the close gate's verification-exception clause, the feature is verified by
deterministic high-fidelity tests that drive the SAME code paths the production
agentic flow hits, through the real subsystem boundaries (not casual stubs).

## Observations

- The integration test drives `registry.run(ToolCall(name: "list_library"))` — the
  exact dispatch `AgenticChatDriver` performs for each tool-use turn — so the only
  unexercised hop is the LLM's decision to emit that tool_use, which is provider
  behavior. With a real tool-use provider + agentic enabled, the user's original
  repro ("library 里有哪些书?") now has a tool that answers from the real library.
- "Tool only (require agentic)" per the user's decision: the `agenticTools` default
  stays OFF, so the default chat is unchanged (criterion 6).
