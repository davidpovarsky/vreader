# Bug Tracker

Track bugs here. Tell the agent "fix bug #N" to start a fix.

## Rules

- **Bugs vs features**: If something was implemented but doesn't work correctly, it is a **bug** ‚Äî track it here. If something was never implemented, it is a **feature** ‚Äî track it in `docs/features.md`. Never mix them.
- **Partial implementations**: If something is partially implemented, the broken part is a bug here; the missing capability is a feature in `docs/features.md`. Link them.
- **Source of truth**: The **Summary table** at the bottom is the single source of truth for bug status. The Description/Cause/Solution/Lesson sections are historical reference.

## How to use

1. Add bugs as you find them (fill in Summary and File/Area at minimum)
2. Tell the agent: "fix bug #N" ‚Äî it will follow the workflow below
3. Agent updates Status when done

- **Bug fix workflow** (follow this order for every bug):
  1. **Understand**: Read the file/area, reproduce the symptom, identify root cause (not just location). Run `/codex-toolkit:bug-analyze`. If it is not a bug, move it to `docs/features.md`.
  2. **RED**: Write a failing test that proves the bug exists.
  3. **GREEN**: Minimal fix to make the test pass.
  4. **REFACTOR**: Clean up without changing behavior.
  5. **Verify**: Run tests, confirm the fix, check for regressions. Run `/codex-toolkit:audit-fix` on changed files.
  6. **Track**: Update status in the Summary table to FIXED.
  7. Do NOT commit unless explicitly requested.
  8. Record the cause of the bugs, the solutions, and the lessons learned.

## Statuses

- `TODO` ‚Äî not started
- `IN PROGRESS` ‚Äî being worked on
- `FIXED` ‚Äî fix verified in working tree (not necessarily committed)
- `REOPENED` ‚Äî previously fixed but regressed; link to original fix
- `DUPLICATE` ‚Äî duplicate of another bug; note `DUPLICATE OF #N`
- `WONT FIX` ‚Äî intentional behavior or out of scope

### Cause

- **#13**: UITextView with TextKit 1 allocates glyph storage for the entire NSAttributedString upfront (\~65MB for a 9.5MB CJK file). Even with `allowsNonContiguousLayout`, the initial allocation and layout computation causes extreme scroll jank because UITextView must manage millions of glyphs in a single text container.
- **#14**: `LibraryView` checked `viewModel.isEmpty` immediately on appear ‚Äî before `loadBooks()` had a chance to run. Since `books` starts empty, the empty state ("Your Library is Empty") flashed briefly before the actual book list loaded, making the app feel slow.
- **#15, #17**: `TXTReaderContainerView` (and `MDReaderContainerView`) passed `viewModel.currentOffsetUTF16` ‚Äî an `@Observable` property that changes on every scroll ‚Äî directly to the bridge as `restoreOffset`. This created an observation feedback loop: scroll ‚Üí viewModel update ‚Üí SwiftUI re-render ‚Üí `restoreScrollPosition` ‚Üí scroll again, infinitely. The `lastRestoredOffset` guard failed because each scroll produced a slightly different offset value.
- **#16**: The chunked reader's `restoreChunkIndex()` also read `viewModel.currentOffsetUTF16` in the body (same observation cycle). Additionally, position restore only worked in `makeUIView`, not on subsequent renders.
- **#19**: The one-shot scroll position restore in `TXTTextViewBridge.makeUIView` used `DispatchQueue.main.async`, but the UITextView hasn't been sized by SwiftUI at that point. TextKit's `lineFragmentRect` returns `CGRect.zero` when the text container has zero width, so `charOffsetToScrollOffset` returned 0 for all offsets. Before the #15/#17 fix, the restore was re-applied in `updateUIView` on every render (eventually working once the view had a valid frame), but the one-shot approach failed silently on the first attempt.
- **#20**: WKWebView in `EPUBWebViewBridge` intercepts all tap events through its internal gesture recognizers. SwiftUI's `onTapGesture` modifier on the parent `ReaderContainerView` never fires because UIKit gesture recognizers have priority.
- **#21**: `UITextView` with `isSelectable = true` has internal gesture recognizers that consume single-tap events (for cursor placement). The first tap is consumed by UITextView; the second tap reaches SwiftUI's `onTapGesture` only because the text view's gesture state has settled.
- **#22**: When `.toolbar(.hidden, for: .navigationBar)` and `.statusBarHidden(true)` are applied, the navigation bar and status bar disappear but the safe area inset remains. The content doesn't extend into the vacated area, leaving a gap with the window's black background visible.
- **#23**: Three compounding issues: (1) `TXTReaderContainerView` and `MDReaderContainerView` never called `viewModel.onBackground()` / `viewModel.onForeground()`, so backgrounding the app skipped the immediate position save ‚Äî progress since the last debounce was lost if the app was killed. (2) `close()` / `open()` race condition: if `close()` interleaved with an in-progress `open()` at the `positionStore.loadPosition()` suspension point, `close()` saved `currentOffsetUTF16 = 0` (the initial value before position restore), overwriting the correct saved position. (3) The `asyncAfter(0.15s)` one-shot scroll restore had no fallback if the view still had no valid frame ‚Äî TextKit returned `scrollY = 0`, and subsequent scroll events overwrote the saved position with a near-zero offset.
- **#24**: `onBackground()` used a fire-and-forget `Task { try? await positionStore.savePosition(...) }` that required an async actor hop to `PersistenceActor`. iOS could suspend the process before the hop completed, leaving the position unsaved. `.onDisappear` used the same fire-and-forget pattern for `close()`. The 2-second debounce was the only reliable save path ‚Äî short reading sessions (<2s) had no saved position at all. "Disappears after a while" because longer sessions have debounced saves that persist a recent position. EPUB and PDF container views lacked `scenePhase` wiring entirely (dead `onBackground()` code).
- **#25**: TextKit 1 compatibility mode switch (`UITextView is switching to TextKit 1 compatibility mode because its layoutManager was accessed`) triggers a full relayout that resets `contentOffset` to near zero. After position restore, `scrollViewDidScroll` fires twice: first with a close offset (layout snap, ~19 chars drift), then with offset near 0 (TextKit relayout). The second callback was treated as a genuine user scroll, cleared the restore guard, and the debounce saved offset ~0 ‚Äî overwriting the correct saved position. Each open/close cycle drifted the position backward; severe cases jumped to the file beginning.
- **#26**: The #25 fix suppressed ghost scroll callbacks but did NOT fix the visual position. The Phase 1 restore at t+0.15s set the correct `contentOffset`, but the TextKit 1 relayout happened AFTER that and physically reset `contentOffset` to 0. The user saw content at the top of the file. When the suppress window expired, the first real scroll callback reported offset \~591 (wherever the user had scrolled to from the beginning), and subsequent normal scrolling saved a position far from the original. The suppress guards only prevented saving the wrong offset ‚Äî they did not re-apply the correct visual scroll position after the TextKit relayout destroyed it.
- **#27**: UX issue: the Phase 2 restore (t+0.8s) works correctly, but the UITextView is visible from the moment it's created ‚Äî showing content at offset 0 for \~0.8s before jumping to the correct restored position. The user sees a jarring flash of wrong content followed by a jump.
- **#28**: FTS5 `snippet()` returns one snippet per database row (one per source unit), but `findAllMatchOffsets` expands each row to multiple hit occurrences. All expanded hits from the same row shared the same snippet text, making results appear identical. The `source_texts` table was missing, so original text wasn't available for per-occurrence snippet generation.
- **#29**: `TXTReaderContainerView`'s `attrStringKey` composite key (used to trigger `.task(id:)` rebuilds of NSAttributedString) only included `fontSize`, `fontName`, and `lineSpacing`. It excluded `textColor` and `backgroundColor`, so theme changes (which modify colors) didn't trigger a rebuild. The old attributed string with stale colors remained displayed.
- **#30**: The settings sheet used `.presentationDetents([.medium, .large])`, allowing the sheet to expand to full height and cover the reader content entirely, preventing users from previewing changes while adjusting settings.
- **#31**: No bookmark creation mechanism existed. Each reader format has a different ViewModel type, and `ReaderContainerView` doesn't have direct access to them. There was no way to trigger bookmark creation from the toolbar or pass the model container to format-specific container views.
- **#32**: Same root cause as #20/#21 ‚Äî PDFView's internal gesture recognizers consume tap events before they reach SwiftUI's `onTapGesture` modifier. The toolbar toggle notification never fires in PDF mode.
- **#33**: TXT reader's `ReaderContainerView` had no bottom overlay showing reading progress or session time. The ViewModel already computed `totalProgression` and `sessionTimeDisplay`, but no view displayed them.
- **#34**: `ReadingStats.recompute(from:)` existed in the data model but was never called. No code path triggered stats recomputation after reading sessions ended, so `totalReadingTime` and `lastReadDate` in `ReadingStats` remained at their initial values. Library sorting by these fields showed all books as equal.
- **#35**: Bottom overlays used `.ultraThinMaterial` which follows system light/dark mode but NOT the reader's custom theme (sepia, dark). The navigation bar also used system defaults ‚Äî no `.toolbarColorScheme()` was applied. In sepia theme, the content is warm-toned but bars remain white.
- **#36**: The `onNavigate` callback in `ReaderContainerView.searchSheet` was a no-op stub ‚Äî it just closed the sheet without navigating. The comment said "format-specific readers will wire this when they support search navigation" but it was never wired.
- **#37**: When theme or font settings changed, `attrStringKey` triggered `.task(id:)` which set `isBuildingAttrString = true`, showing a loading spinner and hiding the existing content. The attributed string rebuild on a background thread took noticeable time, causing a visible flash.
- **#38**: Position restore used a fixed 0.8s Phase 2 delay (`asyncAfter(deadline: .now() + 0.8)`) to wait for TextKit 1 relayout to settle. This delay was conservative ‚Äî TextKit layout often completes much faster. The UITextView was hidden (`alpha = 0`) for the entire duration, making reopen feel slow.
- **#39**: `isChromeVisible` was a `@State` in `ReaderContainerView` that controlled only the navigation bar and status bar. The bottom overlays lived in format-specific container views (TXT, EPUB, PDF, MD) which had no access to `isChromeVisible` ‚Äî they were always visible regardless of tap state.
- **#40**: Two independent issues: (1) `allowsNonContiguousLayout` causes TextKit to use estimated line heights for unvisited regions ‚Äî `charOffsetToScrollOffset` returns inaccurate Y positions. (2) `SearchHitToLocatorResolver` didn't populate `charRangeStartUTF16`/`charRangeEndUTF16` on Locators, so match ranges couldn't be carried to the bridge.
- **#41**: Position restore used a 0.15s Phase 1 + 0.15s Phase 2 delay totaling ~0.3s, with a 0.15s fade-in animation on top. The `ensureLayout` call from bug #38 fix made layout complete faster, but Phase 2 delay was still conservatively 0.15s and the animation added unnecessary perceived wait.
- **#42**: `AnnotationsPanelSheet` had no-op `onNavigate` closures ‚Äî all four tabs (bookmarks, TOC, highlights, annotations) ignored tap. Also, `BookmarkPersisting` protocol lacked `updateBookmarkTitle`, so bookmarks couldn't be renamed.
- **#43**: When navigating to a search result, the bridge scrolled to the offset but applied no visual indicator on the matched text. Without a highlight, users couldn't identify which text was the search match.
- **#44**: Selection tracking worked in `TXTReaderViewModel` (capturing start/end UTF-16 offsets), but no UI was presented to create highlights or notes from the selection. UITextView's default edit menu only offered Copy/Select All ‚Äî no "Highlight" or "Add Note" options.

When navigating back from the reader (where `recomputeStats()` updates `lastReadAt`), the library showed stale cached data. Sorting by "Last Read" used correct logic but operated on outdated values.

### Solution

- **#13**: Added `TXTTextChunker` (splits text into \~16K UTF-16 chunks at newline boundaries) and `TXTChunkedReaderBridge` (UITableView where each cell renders one chunk as a UITextView). Files over 500K UTF-16 code units are automatically routed to the chunked renderer. Only visible cells build NSAttributedStrings, with LRU caching of up to 20 chunks.
- **#14**: Added `isInitialLoad` property to `LibraryViewModel` (true until first `loadBooks()` completes). `LibraryView` shows a `ProgressView` spinner during initial load, preventing the false empty state flash.
- **#15, #16, #17**: Replaced live `viewModel.currentOffsetUTF16` binding with `@State var initialRestoreOffset` captured once after `viewModel.open()` completes. Replaced `lastRestoredOffset` comparison in bridge with a `hasRestoredPosition` one-shot flag. Removed the scroll-position-restore block from `updateUIView` entirely ‚Äî restore is now one-shot in `makeUIView` only. Applied the same fix to `MDReaderContainerView`.
- **#19**: Changed `DispatchQueue.main.async` to `DispatchQueue.main.asyncAfter(deadline: .now() + 0.15)` in both `TXTTextViewBridge` and `TXTChunkedReaderBridge`. The 150ms delay allows SwiftUI to complete the layout pass, giving the text view a valid frame before TextKit computes glyph positions.
- **#20, #21**: Replaced SwiftUI `onTapGesture` with bridge-level tap handlers. Added `UITapGestureRecognizer` (with `shouldRecognizeSimultaneously = true`) to `TXTTextViewBridge` and `TXTChunkedReaderBridge`. Added JavaScript `click` event listener (excluding `<a>` elements) to `EPUBWebViewBridge`. All bridges post `Notification.Name.readerContentTapped`; `ReaderContainerView` listens via `.onReceive` and toggles `isChromeVisible`.
- **#22**: Added `.ignoresSafeArea(edges: isChromeVisible ? [] : [.top])` to `ReaderContainerView`. When chrome is hidden, content extends into the top safe area, eliminating the black gap.
- **#23**: Three fixes: (1) Added `@Environment(\.scenePhase)` with `.onChange(of: scenePhase)` to both `TXTReaderContainerView` and `MDReaderContainerView`, calling `viewModel.onBackground()` on `.background`/`.inactive` and `viewModel.onForeground()` on `.active`. (2) Added `isOpenComplete` flag to both ViewModels ‚Äî set to `true` only after `open()` finishes position restore and session start; `close()` only saves position when `isOpenComplete` is true. (3) Moved scroll restore logic from the bridge struct into the Coordinator with frame-validity check and retry (up to 5 retries at 100ms intervals) ‚Äî if `textView.bounds.width == 0`, the restore is deferred until the view has a valid frame.
- **#24**: Two fixes: (1) Made `onBackground()` async in all four ViewModels (TXT, MD, EPUB, PDF) ‚Äî the save is now `await`-ed directly instead of fire-and-forget, guaranteeing completion before the method returns. (2) Added `UIApplication.shared.beginBackgroundTask` in all four container views around both `onBackground()` and `close()` calls ‚Äî this requests iOS background execution time (\~30s), ensuring the save completes before the process is suspended. Also added missing `scenePhase` wiring to EPUB and PDF container views.
- **#25**: Two-layer fix: (1) **Bridge level** (`TXTTextViewBridge`): Added `suppressScrollCallbacks` flag to the Coordinator. Set to `true` when restore begins (before any TextKit layout), cleared 0.5s after `setContentOffset` completes. All `scrollViewDidScroll` and `sendScrollPosition` calls return immediately when suppressed ‚Äî no ghost callbacks reach the ViewModel at all. (2) **ViewModel level** (`TXTReaderViewModel`): Added time-based `restoreSuppressUntil` guard (1s after restore). Any scroll callbacks that escape the bridge-level guard are still blocked. `currentOffsetUTF16` stays at the restored value until the user genuinely scrolls.
- **#26**: Two-phase scroll restore in `TXTTextViewBridge.makeUIView`: Phase 1 at t+0.15s applies the initial `setContentOffset` (for the case where TextKit doesn't relayout). Phase 2 at t+0.8s re-applies `attemptScrollRestore` to override the TextKit relayout that reset `contentOffset` to 0, then clears `suppressScrollCallbacks`. Extended ViewModel `restoreSuppressDuration` to 1.5s (was 1.0s) to cover the full Phase 2 window. The last `contentOffset` set before scroll callbacks resume is now guaranteed to be the correct restored position.
- **#27**: Hide the UITextView (`alpha = 0`) when a non-zero `restoreOffset` is provided. Phase 2 at t+0.8s restores the correct position and then reveals the text view with a 0.15s fade-in animation. The user sees the loading spinner until the content appears at the correct position ‚Äî no flash of wrong content. Offset 0 (start of file) skips the hide/reveal cycle entirely since no position jump occurs.
- **#28**: Added `source_texts` table to `SearchIndexStore` to persist original text per source unit during indexing. Added `extractSnippet(from:matchStart:matchEnd:contextChars:)` method that generates per-occurrence context windows with `<b>` tags around the match. Each hit now gets its own unique snippet derived from the stored original text.
- **#29**: Added `textColor` and `backgroundColor` hash values to the `attrStringKey` composite key in `TXTReaderContainerView`. Theme changes now produce a different key, triggering `.task(id:)` to rebuild the NSAttributedString with updated colors.
- **#30**: Changed `.presentationDetents([.medium, .large])` to `.presentationDetents([.medium])` in `ReaderContainerView`, constraining the settings sheet to half-screen height so readers can preview changes.

Passed `modelContainer` to all four format-specific container views (TXT, MD, EPUB, PDF). ### Lesson

- TextKit 1 has a fundamental scaling limit: a single UITextView cannot smoothly handle multi-megabyte attributed strings regardless of layout optimization flags. For large documents, virtualized rendering (UITableView/UICollectionView with cell reuse) is the correct pattern ‚Äî the same principle as "windowing" in web development.
- Always distinguish "not loaded yet" from "loaded and empty" in ViewModel state. A simple boolean flag prevents confusing UI states during async data fetching.
- **Never read a rapidly-mutating ************************************`@Observable`************************************ property in a SwiftUI body that gets passed back to the same UIViewRepresentable that updates it.** This creates a feedback loop: UIKit event ‚Üí Observable mutation ‚Üí SwiftUI re-render ‚Üí UIKit state change ‚Üí UIKit event. Use `@State` to capture one-shot values instead.
- **`DispatchQueue.main.async`**\*\* in ****`makeUIView`**** is NOT reliable for layout-dependent operations.\*\* The UIView hasn't been added to the view hierarchy or sized by SwiftUI when `makeUIView` returns. Use `asyncAfter` with a delay (‚â•100ms) to allow the layout pass to complete before computing positions that depend on the view's frame.
- **UIKit gesture recognizers in ********************************`UIViewRepresentable`******************************** views intercept touches before SwiftUI gesture modifiers.** `WKWebView` and `UITextView` with `isSelectable = true` both consume single-tap events. To handle taps inside these views, install your own `UITapGestureRecognizer` (with `shouldRecognizeSimultaneously = true`) or use JavaScript event handlers (for WKWebView). Using `NotificationCenter` to communicate back to the SwiftUI parent avoids callback threading through the view hierarchy.
- \*\*Conditional ****`.ignoresSafeArea(edges:)`**** with ****`Edge.Set`**** as an \*\***`OptionSet`** supports passing `[]` for no-op. This enables safe area behavior that changes with state (e.g., extend content when toolbar is hidden).
- **If a ViewModel has ****************************`onBackground()`****************************/********`onForeground()`********\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\* methods, the View MUST wire them to ****************************`@Environment(\.scenePhase)`****************************.** Otherwise the immediate-save and session-pause logic is dead code. Debounced saves alone are insufficient ‚Äî the app can be killed before the debounce fires.
- **`@MainActor`**\*\* async methods can interleave at ****`await`**** suspension points.\*\* Two `@MainActor` methods (like `open()` and `close()`) can race when both are called from different Tasks. Use generation counters and completion flags (e.g., `isOpenComplete`) to guard against saving stale intermediate state.
- **`asyncAfter`**\*\* with a fixed delay is fragile for layout-dependent operations.\*\* Instead of trusting that 150ms is enough, check `bounds.width > 0` and retry with a bounded retry count. This handles slow devices and complex view hierarchies where SwiftUI's layout pass takes longer than expected.
- **Never use fire-and-forget ************************`Task { }`************************ for critical saves in ************************`onBackground()`************************ or ************************`.onDisappear`************************.** The Task requires an async actor hop that may not complete before iOS suspends the process. Make `onBackground()` async and `await` the save. Wrap the caller's Task with `UIApplication.shared.beginBackgroundTask` to request execution time from iOS before suspension.
- **Every reader container view must wire ************************`@Environment(\.scenePhase)`************************.** Without it, `onBackground()`/`onForeground()` are dead code. The debounced save alone is insufficient ‚Äî the app can be killed at any time between debounce intervals.
- **TextKit 1 compatibility mode switch destroys scroll position.** Accessing `UITextView.layoutManager` triggers a full relayout that resets `contentOffset`. Suppress all `scrollViewDidScroll` callbacks from the moment text is loaded until after the scroll restore settles. Use a flag on the Coordinator (not just ViewModel-level guards) because the ghost callbacks originate in UIKit before reaching SwiftUI.
- **Distance-based scroll guards are insufficient for layout storms.** TextKit relayout can jump `contentOffset` to zero, which exceeds any reasonable tolerance. Use time-based suppression (suppress for N seconds after restore) or flag-based suppression (suppress until explicitly cleared) instead.
- **Suppressing callbacks is not enough ‚Äî the visual position must also be re-applied.** When TextKit relayout destroys a restored `contentOffset`, the user sees the wrong position even if no bad data is saved. The fix requires re-applying `setContentOffset` *after* the relayout settles (Phase 2 restore), not just suppressing the resulting scroll callbacks.
- **Hide content during async position restore.** When scroll position restore requires delayed phases (for TextKit layout settling), set `alpha = 0` on the view until the final restore completes. Showing content at the wrong position and then jumping is worse UX than showing a loading state for 0.8s longer.
- **FTS5 ********`snippet()`******** is per-row, not per-occurrence.** When expanding FTS5 results to multiple match offsets within one row, each occurrence needs its own context window. Store the original source text during indexing so you can generate per-occurrence snippets at query time.
- **Composite keys for ********`@Observable`********-driven rebuilds must include ALL mutable properties.** If a `.task(id:)` key omits a property, changes to that property won't trigger a rebuild. Audit all properties that affect the output and include them in the key.
- **Sheet ********`.presentationDetents`******** should match the use case.** Settings/preview sheets should constrain to `.medium` so the underlying content remains visible for real-time preview. Only use `.large` when the sheet needs full screen space.
- **`NotificationCenter`**\*\* is the right cross-view communication pattern in SwiftUI when views don't share a direct data path.\*\* Format-specific container views can't easily share a common ViewModel protocol for diverse operations like bookmark creation. A notification + `modelContainer` passthrough is simpler than threading callbacks or generic protocols through the view hierarchy.
- **UIKit views with internal gesture recognizers (PDFView, UITextView, WKWebView) need ********`shouldRecognizeSimultaneouslyWith`******** for tap coexistence.** This is a recurring pattern ‚Äî any UIViewRepresentable wrapping an interactive UIKit view will consume taps before SwiftUI sees them.
- **Wire computed stats to lifecycle events, not just data mutations.** Having a `recompute(from:)` method is useless if nothing calls it. Stats aggregation should be triggered at reader close ‚Äî the natural lifecycle event where session data is finalized.
- **Reader chrome (bars, overlays) must use the reader's theme, not system theme.** `.ultraThinMaterial` follows system light/dark mode but ignores custom reader themes like sepia. Use the theme's actual colors with slight transparency for overlays, and `.toolbarColorScheme()` for the navigation bar.
- **Never leave stub callbacks in shipping code.** A no-op `onNavigate: { _ in }` is indistinguishable from a bug. If a feature isn't ready, show a "coming soon" toast or disable the trigger ‚Äî don't silently swallow the user's action.
- **Show old content during settings-driven rebuilds.** When a settings change triggers an expensive rebuild (e.g., NSAttributedString), keep the previous content visible and swap when the new content is ready. Only show a loading spinner for the initial load when no content exists.
- **Use ****`layoutManager.ensureLayout(forCharacterRange:)`**** instead of fixed delays for TextKit layout.** Forcing synchronous layout is faster and more reliable than waiting a fixed time. Fall back to a short safety delay as a belt-and-suspenders approach.
- **Decouple chrome visibility from the view hierarchy.** When parent and child views need to share toggle state but don't have a direct binding path, `NotificationCenter` is the right pattern. Each view listens for the same notification and maintains its own `@State` ‚Äî simple and reliable.
- **`ensureLayout(forCharacterRange:)` is essential before any programmatic scroll in non-contiguous layout mode.** Without it, TextKit uses estimated line heights for unvisited regions, giving incorrect Y positions. The range should cover at least the target offset for accurate results.
- **Search result match ranges must travel end-to-end through the pipeline.** From tokenizer offset ‚Üí FTS5 hit ‚Üí Locator ‚Üí bridge. If any stage drops the range, highlights and precise navigation break. Always propagate `charRangeStartUTF16/charRangeEndUTF16` alongside `charOffsetUTF16`.
- **Temporary visual highlights should auto-clear.** A search highlight that persists indefinitely confuses users. Use a Timer to remove background attributes after a few seconds. Always clamp ranges to text storage length to avoid crashes.
- **`textView(_:editMenuForTextIn:suggestedActions:)` (iOS 16+) is the cleanest way to add custom actions to UITextView's selection menu.** It replaces the deprecated `UIMenuController.shared.menuItems` approach and integrates naturally with the system edit menu.
- **`.task` in SwiftUI only runs once per view lifetime.** If data changes while the view is off-screen (e.g., stats updated in reader), the library won't refresh automatically. Use `.onAppear` with a throttled `refresh()` to pick up changes when the user navigates back.
- **`NotificationCenter` scales well for cross-view communication in UIViewRepresentable.** Bridge-level actions (highlight, note from edit menu) can't easily call SwiftUI callbacks. Posting notifications with lightweight info structs and observing via `.onReceive` decouples the bridge from SwiftUI state management.
